## Why This Exists

First impressions are everything. When Yahya or Yara logs into Hayah for the first time, the app doesn't know who they are yet — their profile exists in the database but is hollow. No name, no face. Before they can use anything meaningful, they need to feel welcomed and invited to make the app theirs. This screen is that invitation.

This is not an onboarding wizard. It's not a settings form. It's a single, warm, focused moment: "Welcome to Hayah — tell us who you are." It should feel like the app is excited to meet them, not like filling out a form at the DMV.

The screen appears as a full-screen overlay the first time and only the first time. After the user taps "Get Started", it vanishes and the home dashboard emerges beneath it. It should never appear again unless the display_name is wiped back to a default state.

---

## User Story

As Yahya or Yara, when I log in for the first time, I see a warm welcome screen that asks me to confirm my name and optionally add a photo. I can type my preferred name, tap my avatar circle to upload a photo from my phone, and tap "Get Started". The app saves my choices and the overlay disappears, revealing my home dashboard. From that moment on, the app knows who I am and always greets me by name.

---

## Full UX Flow

1. **First login completes** → AuthProvider checks the profiles table for the current user's `display_name`
2. **Profile is incomplete** → `display_name` is null, empty, or matches a seeded default value (e.g. "User" or the raw email prefix)
3. **ProfileSetupOverlay renders** over the main layout — the rest of the app is behind it but not interactive
4. **User sees the welcome screen:**
   - "Welcome to Hayah" in Playfair Display
   - "Let's set up your profile" subtitle
   - An 80px avatar circle (tappable) pre-showing a default silhouette icon
   - A display name text input, pre-filled with whatever partial data exists (email prefix, etc.)
   - A "Get Started" button in copper, full width
5. **User taps the avatar circle:**
   - Triggers a native `<input type="file" accept="image/*">` click (hidden input)
   - On iOS this presents the photo library / camera sheet
   - User selects a photo → it is cropped/previewed inside the avatar circle immediately (client-side `URL.createObjectURL`)
   - No upload happens yet — the file is held in form state
6. **User edits the display name** in the text input (min 1 char, max 50 chars)
7. **User taps "Get Started":**
   - Button shows a loading spinner, becomes disabled
   - If an avatar file was selected: upload to Supabase Storage at path `avatars/{userId}/{timestamp}.{ext}` — get back the public URL
   - PATCH the `profiles` table: set `display_name` and `avatar_url` (if uploaded)
   - On success: dismiss the overlay with a fade-out animation
   - Home dashboard becomes fully interactive beneath it
8. **On error:** Show an inline error message below the button, re-enable the form
9. **On subsequent logins** — `display_name` is now set, AuthProvider skips the overlay entirely

## Implementation Notes

- `"use client"` directive required — uses React Hook Form, file input, Framer Motion
- Use React Hook Form with Zod resolver for the form:
  ```ts
  const schema = z.object({
    display_name: z.string().min(1, 'Name is required').max(50, 'Name is too long'),
    avatar: z.instanceof(File).optional(),
  })
  ```
- The hidden file `<input>` must be referenced via `useRef` and triggered by clicking the avatar circle div — do NOT use a visible file input element
- Avatar preview uses `URL.createObjectURL(file)` stored in local state — revoke the URL on component unmount with `URL.revokeObjectURL`
- Upload path convention: `avatars/${userId}/${Date.now()}.${ext}` inside the `avatars` Supabase Storage bucket (assume bucket is public)
- Use the Supabase client from T202 (`@/lib/supabase/client`)
- After successful upload, call `supabase.storage.from('avatars').getPublicUrl(path)` to retrieve the URL
- PATCH to profiles: `supabase.from('profiles').update({ display_name, avatar_url }).eq('id', userId)`
- The AuthProvider (T204) should expose a `profileNeedsSetup: boolean` value derived from checking `profile.display_name` — this component is shown when that flag is true
- The overlay sits in the root layout file at `src/app/layout.tsx` or inside `AppShell` — rendered after auth is confirmed, wraps children
- Use `AnimatePresence` from Framer Motion around the overlay so the exit animation plays when `onComplete` is called and the component unmounts
- Use `cn()` from `@/lib/utils` for all conditional class logic
- Avatar upload is optional — if no file selected, `avatar_url` is omitted from the PATCH

## Edge Cases

- **User submits without changing the name** — form is valid as long as the pre-filled name passes Zod (min 1 char) — allow it
- **User clears the name field** — Zod validation fires, error message appears below the input, button stays disabled
- **File selected is not an image** — `accept="image/*"` on the input prevents this at the browser level, but validate file type on change as a second guard
- **File is too large** — enforce a 5MB max on the client before attempting upload; show a friendly error if exceeded
- **Upload succeeds but PATCH fails** — show generic error, re-enable form; orphaned file in Storage is acceptable at this stage
- **User force-closes the app mid-setup** — on next open, `display_name` is still unset, overlay appears again; form state is reset (no draft persistence needed)
- **Network is offline when tapping Get Started** — Supabase call will fail; show error message, re-enable form; no retry logic needed
- **Both users log in on the same device** — each user's profile is independent; overlay checks the currently authenticated user's profile only
- **`initialName` is an email address** (e.g. "yahya@gmail.com") — truncate at the `@` sign before pre-filling the input so it doesn't look odd

## What This Connects To

- **T201 (profiles table)** — this component reads and writes to the `profiles` table; `display_name` and `avatar_url` are the fields modified
- **T202 (Supabase client)** — imports `createClient` from `@/lib/supabase/client` for DB and Storage operations

- **T207** — depends on profile setup being complete; T207 assumes `display_name` and optionally `avatar_url` are set and reads them for display throughout the app

## Do NOT

- Don't make this a separate route — it is a conditional overlay inside the layout, not `/profile/setup`
- Don't use `router.push()` to navigate away on completion — call `onComplete` and let the AuthProvider update state
- Don't show this overlay for users who already have a valid `display_name`
- Don't build a multi-step wizard — this is a single screen, single action
- Don't add social auth avatar auto-import — if Google avatar is available that's a future enhancement
- Don't use `<img>` for the avatar preview — use Next.js `Image` or a plain `img` tag with `object-cover`; do NOT use CSS background-image for the preview
- Don't hardcode any color hex values — use Tailwind theme tokens from the design system
- Don't persist form draft state across sessions — if the user abandons, start fresh next time
- Don't allow the overlay background to be tapped to dismiss — the user must complete or the app is unusable
