## Why This Exists

Sending a push notification to your partner is one of the most intimate gestures in Hayah. It's not a message — it's a tap on the shoulder across the city. When Yahya is at work and thinks of Yara, he doesn't want to write a paragraph. He wants to pick a heart emoji, type three words, and send a warm copper button press into her phone. That's what this component delivers.

The NotificationBuilder is the full composition UI for that moment. It lives inside the `/us` page's "Send" tab. It needs to feel effortless — form fields that don't feel like a form, a real-time preview that builds anticipation, and a send button that feels like dropping a love note through a letterbox.

---

## User Story

As Yahya or Yara, when I open the "Send" tab on the /us page, I see a simple composition form. I pick an emoji from a curated row of romantic options, type a short title and message, and watch a live iOS-style push notification preview update in real time. I tap Send, and it's delivered. I see how many free sends I have left for the day. If I've used them up, I know how to get more.

---

## Full UX Flow

1. **User arrives at the Send tab** → NotificationBuilder is mounted, form is empty
2. **Emoji row** — a horizontal scrollable row of ~12 curated emojis is shown at the top. Default: none selected (or first emoji pre-selected). Tapping one selects it with an accent-soft background ring.
3. **Title input** — single-line text input, placeholder "e.g. Hey babe". Required, max 50 chars.
4. **Message textarea** — 3-row textarea, placeholder "Write something...". Required, max 200 chars.
5. **Live preview** — as user types, the preview card below updates in real time to show what the push notification will look like on iOS. Shows: emoji + title on line 1, body text on line 2.
6. **Send button** — copper button labelled "Send 💌". Tapping it:
   - If form is invalid → react-hook-form shows inline field errors, no request made
   - If !canSend → button is disabled or shows "No sends remaining" state + link to buy more (T407)
   - If canSend and form valid → button enters loading state (spinner), calls sendNotification() from T402
   - On success → form clears, button resets, success message "Sent! 💌" appears inline (no confetti, just warm text)
   - On error → inline error message below button
7. **Limit indicator** — below the button, the LimitIndicator component (T406) shows how many free sends remain today (e.g. "2 sends remaining today")

## Zod Schema

```ts
import { z } from "zod"

export const notificationSchema = z.object({
  emoji: z.string().optional(),
  title: z.string().min(1, "Add a title").max(50, "Title is too long"),
  body: z.string().min(1, "Write a message").max(200, "Message is too long"),
})

export type NotificationFormValues = z.infer<typeof notificationSchema>
```

## Implementation Notes

- `"use client"` directive required (hooks, form, motion)
- Use `useForm<NotificationFormValues>` from `react-hook-form` with `zodResolver(notificationSchema)`
- Emoji row: define a `PRESET_EMOJIS` constant array of 12 strings at the top of the file — do NOT install any emoji picker library
- Emoji selection is local state (`useState<string | undefined>`), not a form field tracked by RHF — it is read at submit time and merged into the payload
- Watch title and body with `watch()` to power the live preview
- Partner name comes from `useAuth()` (T204) — read `session.partner.name` or similar; fall back to "your partner" if not available
- `sendNotification`, `canSend`, and `remainingSends` come from `useNotifications()` (T402)
- On submit: call `sendNotification({ emoji, title, body })` — handle promise with try/catch
- Success state: local boolean `sent`, display "Sent! 💌" as a `motion.p` that fades in (`initial={{ opacity: 0 }} animate={{ opacity: 1 }}`)
- Loading state: use `isSubmitting` from `formState` to disable button and show a Loader2 spinner (Lucide) inside the button
- Error state: display the error message as a `<p>` with `text-destructive` below the button
- Use `cn()` from `@/lib/utils` for all conditional class merging
- The preview card should always render, even when fields are empty — show placeholder text in muted color if title/body are empty
- `LimitIndicator` (T406) receives `used` and `limit` as props — derive from `remainingSends` from the hook

## Edge Cases

- **Partner name unavailable** → fall back to "your partner" gracefully
- **Both title and body empty** → Send button should be disabled before user touches fields (use `formState.isValid` with `mode: "onChange"` or keep button enabled and let Zod surface errors on submit)
- **canSend is false** → disable the Send button, replace its label with "No sends remaining", show a link beneath it ("Get more sends →") pointing to the T407 CoYYns purchase flow
- **sendNotification throws** → catch error, store message in local state, display inline beneath button — do not crash
- **Rapid double-tap on Send** → `isSubmitting` guard prevents duplicate sends
- **Very long message approaching 200 chars** → consider showing a character counter (`body.length / 200`) beneath the textarea in text-muted
- **No emoji selected** → notification is sent without an emoji prefix — both title and body remain valid; preview simply shows title without emoji

## Do NOT

- Don't install an emoji picker library — use the hardcoded `PRESET_EMOJIS` array
- Don't hardcode color hex values — use Tailwind theme tokens (`text-primary`, `bg-accent-primary`, etc.)
- Don't skip the live preview — it is a core part of the experience, not optional
- Don't use `router.push()` for the "Get more sends" link — use `next/link`
- Don't show confetti or full-screen success modals — the celebration is just the inline "Sent! 💌" message
- Don't store form state in Zustand — local component state and react-hook-form only
- Don't call `sendNotification` without first checking form validity via RHF's handleSubmit
