## Why This Exists

Yahya needs to know where Yara is in her cycle at a glance — not buried in a sub-menu, not requiring a tap to reveal, just visible the moment he opens the app or the health page. This widget surfaces that single most important number: the day. Not a chart, not a history, not a log — just "Day 14, Active Phase, 7 days until break." That is enough for Yahya to adjust his day, be more thoughtful, stock up on comfort snacks, or simply know that tonight is a good night to plan something special.

This is YAHYA-ONLY data. Yara does not see this widget. It is Yahya's awareness tool, not Yara's health tracker. Yara's cycle information is private to Yahya's view so that he can support her without making her feel monitored. The widget renders absolutely nothing when the authenticated user is Yara.

---

## User Story

As Yahya, when I open the health page or the home dashboard, I see a compact card that tells me the current cycle day, the active phase, and how many days remain until the next phase. The card changes color when Yara is in her PMS window or period phase so I know to be extra present. If the cycle has not been configured yet, I see a prompt to set it up. If I am Yara, I see nothing — this component does not exist for me.

---

## Full UX Flow

1. **Yahya opens the app** → CycleDayWidget is visible on the home dashboard (compact mode) and on the /health page (full mode)
2. **Widget reads from useCycle hook (T502)** → hook returns current day, phase, days remaining, and isYahya flag
3. **If isYahya is false** → component returns null immediately, renders nothing, zero DOM output
4. **If cycleConfig is null** (no cycle set up yet) → render a "Set up cycle tracking" CTA card with a link to /health/setup
5. **Normal state (compact=false):**
   - Top line: "Day 14" in JetBrains Mono, 24px, accent-primary color
   - Second line: phase label in DM Sans, 13px, text-secondary
   - Progress dots row: 28 dots total — filled for completed days, ring/glow on current day, hollow for future days
   - Bottom line: "7 days until break" in 12px, text-muted
6. **Compact mode (compact=true):** single line — "Day 14 · Active" — no dots, no subtitle, no bottom line. Used on the home dashboard where space is precious.
7. **Phase colors shift automatically** based on the phase from useCycle — no manual prop needed

## Implementation Notes

- `"use client"` directive required (uses hooks: useCycle, useAuth)
- **First line of the component body must be the Yahya guard:**
  ```tsx
  const { isYahya, cycleDay, phase, daysUntilNextPhase, cycleConfig } = useCycle()
  if (!isYahya) return null
  ```
  This is non-negotiable. No other logic, no conditional rendering, no wrapper — return null immediately.
- Import `useCycle` from `@/hooks/use-cycle` (T502)
- Import `useAuth` from `@/hooks/use-auth` (T204) — useCycle uses it internally, but be aware of the dependency
- Use `cn()` from `@/lib/utils` for conditional class merging
- Use Framer Motion `motion.div` for the card entrance: `initial={{ opacity: 0, y: 8 }}`, `animate={{ opacity: 1, y: 0 }}`, `transition={{ duration: 0.3, ease: 'easeOut' }}`
- Progress dots are rendered via `Array.from({ length: 28 }, (_, i) => i + 1).map(day => <DotComponent>)`
- Extract a local `<CycleDot>` sub-component inside the file (not exported) to keep the dot logic readable
- Phase determination: use a `getPhaseState(day: number)` pure function inside the file — returns `'active' | 'pms' | 'break'`
- For the compact mode, use a single `<span>` with JetBrains Mono for the number and DM Sans for the label — inline, separated by `·`
- The "Set up cycle tracking" CTA card uses `next/link` pointing to `/health/setup`
- Never hardcode the cycle length (28) as a magic number — define `const CYCLE_LENGTH = 28` and `const BREAK_START_DAY = 22` at the top of the file

## Edge Cases

- **isYahya is false** → return null, zero DOM, no wrapper div, nothing. Do not render a placeholder, do not render empty space.
- **cycleConfig is null** → render the setup CTA card, not the main card. The CTA card is also Yahya-only (guarded by the same check above it).
- **cycleDay is 0 or undefined** → treat as setup-needed state, render CTA card
- **cycleDay > 28** → cycle has overrun. Cap display at 28, show "Day 28" — the parent hook (T502) should handle reset but the widget must not crash
- **compact=true on a large screen** → compact mode stays compact regardless of screen size, the parent decides which mode to use
- **Phase transition exactly on day 19 or 22** → phase state must update immediately, no stale display
- **Animation on re-render** → use `key={cycleDay}` on the animated container so Framer re-mounts the entrance animation when the day changes (e.g. at midnight)

## Do NOT

- Don't render anything at all when isYahya is false — not a blank div, not a loading state, not an empty fragment
- Don't hardcode color hex values — use Tailwind theme tokens tied to CSS variables
- Don't import Yara's user data or render anything conditional on Yara's state; this component only cares about isYahya being true
- Don't add a loading spinner while useCycle resolves — render nothing until data is available
- Don't build any cycle configuration UI inside this widget — the CTA links to /health/setup, that page is a separate task
- Don't animate the progress dots individually — only the card container entrance animates
- Don't use `left`/`right` CSS properties — use logical properties (`ps`, `pe`, `ms`, `me`) for RTL compatibility
- Don't export CycleDot or getPhaseState — they are internal to this file only
