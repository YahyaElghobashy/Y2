## Why This Exists

Yahya tracks Yara's hormonal pill cycle not to monitor her — but to understand her. To know when she might be more sensitive, to have chocolates ready before she even asks, to give her extra space on the hard days without needing an explanation. This is one of the most intimate forms of care: paying attention before she has to say anything.

This database migration creates the data layer that makes that possible. Two tables: one for cycle configuration (when the pack started, how many active days, when to expect the next period) and one for daily logs (mood observations, notes to self). Together they let Yahya compute where Yara is in her cycle on any given day — active pills, break week, PMS window — and surface gentle reminders in the app.

This is NOT a fertility tracker. There are no ovulation windows, no conception calculations, no graphs for Yara. This data exists solely in Yahya's mental model of care, expressed through a private digital tool that she never sees.

**Privacy is the entire point.** If Yara ever opened the app and saw this data, the intimacy of the gesture would collapse. Every byte of this feature must be invisible to her.

---

## User Story

As Yahya (and only Yahya), I want to configure Yara's pill cycle once and then have the app silently track where she is in the cycle every day. I want to log mood observations or notes privately. I never want Yara to be able to see, query, or even detect that this data exists in the system — not through the UI, not through a direct Supabase query, not through any API surface.

---

## Full DB Flow

1. **First setup (one-time):** Yahya opens the cycle tracker section, enters the date the current pill pack started. He confirms active days (21) and break days (7). The app writes one row to `cycle_config` with his `owner_id`.
2. **Daily use (optional):** Yahya optionally logs a mood observation or note for a given date. The app writes or upserts one row to `cycle_logs` per date.
3. **Read path:** On any given day, Yahya's client queries `cycle_config` to compute the current cycle phase (active, break, PMS window). It can also fetch recent `cycle_logs` entries to show historical context.
4. **RLS enforcement:** Every SELECT, INSERT, UPDATE, DELETE against both tables is filtered by `auth.uid() = owner_id`. Yara's authenticated session returns zero rows from both tables — not an error, not a permission denied message, just empty results. She cannot distinguish this table from a table that does not exist.
5. **No partner read:** Unlike every other table in the app, there is deliberately NO policy granting Yara access to these tables. The partner-read pattern from T201 does NOT apply here.

## SQL Schema

**File to create:** `supabase/migrations/004_cycle_tracker.sql`

```sql
-- ============================================================
-- T501: Cycle Tracker Migration
-- Creates cycle_config and cycle_logs tables for hormonal
-- pill cycle tracking.
--
-- PRIVACY CRITICAL: These tables are OWNER-ONLY.
-- No partner access. No anon access. No exceptions.
-- Yara must never see this data under any circumstance.
--
-- This is migration 004. Requires 001 (profiles) to exist.
-- ============================================================


-- ── 1. CYCLE_CONFIG TABLE ─────────────────────────────────────
-- One row per owner. Stores the pill cycle configuration.
-- Yahya creates this once on first use and updates it when
-- a new pack starts (pill_start_date resets each cycle).

create table public.cycle_config (
  id                uuid        primary key default gen_random_uuid(),
  owner_id          uuid        not null unique references public.profiles (id) on delete cascade,
  pill_start_date   date        not null,
  active_days       integer     not null default 21,
  break_days        integer     not null default 7,
  pms_warning_days  integer     not null default 3,
  notes             text,
  created_at        timestamptz not null default now(),
  updated_at        timestamptz not null default now()
);

-- Index: owner_id is UNIQUE so Postgres creates an index automatically.
-- No additional index needed — this table will always have at most one row.

comment on table public.cycle_config is
  'PRIVATE — Yahya only. Stores the hormonal pill cycle configuration for tracking purposes. '
  'Yara must never have access to this table. RLS enforces owner_id = auth.uid() with no exceptions.';
comment on column public.cycle_config.owner_id is
  'Always Yahya''s UUID. UNIQUE constraint guarantees one config per owner.';
comment on column public.cycle_config.pill_start_date is
  'The date the current pill pack was started. Reset each time a new pack begins.';
comment on column public.cycle_config.active_days is
  'Number of days Yara takes active pills. Default 21 for a standard combined pill pack.';
comment on column public.cycle_config.break_days is
  'Number of pill-free (break) days. Period typically begins during this window. Default 7.';
comment on column public.cycle_config.pms_warning_days is
  'Days before the break starts to surface a PMS awareness reminder. Default 3.';


-- ── 2. CYCLE_LOGS TABLE ───────────────────────────────────────
-- One row per (owner_id, date). Yahya logs mood observations
-- and notes. This is a private journal, not a medical record.
-- Entries are optional — most days will have no log entry.

create table public.cycle_logs (
  id          uuid        primary key default gen_random_uuid(),
  owner_id    uuid        not null references public.profiles (id) on delete cascade,
  date        date        not null,
  mood        text        check (mood in ('good', 'neutral', 'sensitive', 'difficult')),
  symptoms    text[],
  notes       text,
  created_at  timestamptz not null default now(),

  unique (owner_id, date)
);

-- Index for efficient range queries (e.g., last 30 days of logs)
create index cycle_logs_owner_date_idx on public.cycle_logs (owner_id, date desc);

comment on table public.cycle_logs is
  'PRIVATE — Yahya only. Daily observation log for cycle awareness. '
  'One row per (owner_id, date). Yara must never have access to this table.';
comment on column public.cycle_logs.mood is
  'Optional mood tag observed for the day. '
  'Allowed values: good, neutral, sensitive, difficult. NULL if not logged.';
comment on column public.cycle_logs.symptoms is
  'Optional array of symptom tags (e.g., {''cramps'', ''fatigue'', ''headache''}). '
  'Free-form text array — no enum constraint, as observations vary.';


-- ── 3. UPDATED_AT TRIGGER FOR CYCLE_CONFIG ───────────────────
-- Re-uses the set_updated_at() function created in T201.
-- cycle_logs does not have updated_at — entries are insert-only
-- or replaced via upsert on (owner_id, date).

create trigger cycle_config_set_updated_at
  before update on public.cycle_config
  for each row
  execute function public.set_updated_at();


-- ── 4. ROW LEVEL SECURITY — CYCLE_CONFIG ─────────────────────
-- STRICT OWNER-ONLY. No partner access. No anon access.
-- auth.uid() = owner_id is the ONLY condition that grants access.
-- Yara's auth.uid() will never equal owner_id (which is Yahya's UUID),
-- so she receives empty results from every query — silently.

alter table public.cycle_config enable row level security;

-- Policy: owner can read their own config
create policy "cycle_config: owner select"
  on public.cycle_config
  for select
  using (auth.uid() = owner_id);

-- Policy: owner can insert their own config (first-time setup)
create policy "cycle_config: owner insert"
  on public.cycle_config
  for insert
  with check (auth.uid() = owner_id);

-- Policy: owner can update their own config (e.g., new pack start date)
create policy "cycle_config: owner update"
  on public.cycle_config
  for update
  using (auth.uid() = owner_id)
  with check (auth.uid() = owner_id);

-- No DELETE policy on cycle_config — config rows should persist.
-- No SELECT/INSERT/UPDATE policy for any other user — partner access
-- is explicitly and intentionally absent. This is the only table
-- in the entire app that blocks partner reads.


-- ── 5. ROW LEVEL SECURITY — CYCLE_LOGS ───────────────────────
-- Same strict owner-only enforcement. All four operations (SELECT,
-- INSERT, UPDATE, DELETE) are permitted for the owner.
-- DELETE is included so Yahya can remove an incorrect log entry.
-- No operation is permitted for any other user.

alter table public.cycle_logs enable row level security;

-- Policy: owner can read their own logs
create policy "cycle_logs: owner select"
  on public.cycle_logs
  for select
  using (auth.uid() = owner_id);

-- Policy: owner can insert new log entries
create policy "cycle_logs: owner insert"
  on public.cycle_logs
  for insert
  with check (auth.uid() = owner_id);

-- Policy: owner can update existing log entries
create policy "cycle_logs: owner update"
  on public.cycle_logs
  for update
  using (auth.uid() = owner_id)
  with check (auth.uid() = owner_id);

-- Policy: owner can delete a log entry (e.g., entered wrong date)
create policy "cycle_logs: owner delete"
  on public.cycle_logs
  for delete
  using (auth.uid() = owner_id);

-- No policies for any other user. Partner access does not exist.
-- Yara querying either table receives 0 rows and no error.


-- ── 6. NO SEED DATA ───────────────────────────────────────────
-- cycle_config is populated by Yahya on first use via the app UI.
-- No default rows are inserted here. The app must handle the
-- "not configured yet" state gracefully (empty state screen in T502).
```

## Implementation Notes

- **File location:** `supabase/migrations/004_cycle_tracker.sql`. This is migration 004. Migrations 001 (profiles), 002, and 003 must exist and have been applied before this one runs. Do not renumber existing migrations.
- **`gen_random_uuid()`** is used for PK defaults. This is available natively in Postgres 13+ (which Supabase uses). Do not use `uuid_generate_v4()` — it requires the `uuid-ossp` extension.
- **`on delete cascade`** on both `owner_id` columns means if Yahya's profile is ever deleted, all his cycle data is purged automatically. This is intentional and correct — orphan cycle data must never exist.
- **`UNIQUE (owner_id)` on `cycle_config`** enforces that there is exactly one configuration row per owner. The client should always UPSERT on this table, not INSERT, to avoid constraint violations on reconfiguration.
- **`UNIQUE (owner_id, date)` on `cycle_logs`** enforces one log entry per day per owner. The client should UPSERT using `on conflict (owner_id, date) do update` to handle editing an existing day's entry.
- **The mood `CHECK` constraint** limits values to `'good'`, `'neutral'`, `'sensitive'`, `'difficult'`. If future variants are needed, alter the constraint — do not bypass it with free text.
- **`set_updated_at()` function** was created in T201 migration. This migration reuses it. Do not redefine it here. If running this migration in isolation (e.g., during testing), ensure T201 has run first.
- **cycle_logs has no `updated_at`** because entries are effectively replaced via upsert. The `created_at` of the upserted row will update on conflict — this is acceptable for a personal log.
- **Privacy at the query level:** Because RLS is enforced server-side, even a developer using the Supabase SQL editor while authenticated as Yara will see zero rows from these tables. The protection is not client-side — it is enforced by Postgres on every query.
- **No Edge Functions or Storage** are created by this migration. Those come in later tasks if needed.
- Run via Supabase CLI: `supabase db push` or `supabase migration up`.

## Edge Cases

- **Yahya runs first-time setup twice:** The `UNIQUE (owner_id)` constraint on `cycle_config` will reject a second INSERT. The client must use UPSERT (`on conflict (owner_id) do update set ...`) to safely overwrite the existing config. Never raw INSERT from the client.
- **New pill pack starts mid-cycle:** Yahya updates `pill_start_date` to the new pack's start date. Existing `cycle_logs` entries from the previous cycle remain intact — they are historical data and should be preserved.
- **Log entry for a future date:** The schema does not prevent this. If Yahya accidentally logs tomorrow's date, it will be stored. The client should validate `date <= today` before submitting, but this is a client concern, not a DB constraint.
- **`pms_warning_days` set to 0:** Valid. It simply disables PMS warnings. No DB-level guard needed — the application layer computes the window.
- **`active_days + break_days` arithmetic:** The DB stores the raw values. The cycle phase computation (which day of which phase is today?) happens entirely in the `useCycle` hook (T502). The DB does no computed columns or generated fields for cycle state.
- **Yara queries these tables directly via Supabase client:** RLS returns an empty result set, not an error. She cannot distinguish this from a table that does not contain data, or a table that does not exist. This is the correct and intended behavior.
- **Service role bypasses RLS:** Any code running with the Supabase service role key (e.g., Edge Functions, admin scripts) bypasses RLS entirely. Never use the service role key in client-side code. Never write an Edge Function that exposes cycle data to any user other than the owner.
- **Migration applied on non-empty DB:** `create table` will error if the table already exists. For development resets, run `supabase db reset` on a local instance only. Never reset production.

## What This Connects To

- **T201 (Auth Profiles)** — `owner_id` in both tables references `public.profiles.id`. This migration must run after T201.
- **T502 (useCycle hook)** — the React hook that reads `cycle_config`, computes the current cycle phase (active, PMS window, break), and exposes it to UI components. It also handles UPSERT for `cycle_logs` entries. T502 is the primary consumer of this schema.
- **T503 (Cycle Tracker UI)** — the screen Yahya sees. It reads from T502's hook. Only rendered for Yahya's session — the route is hidden from Yara entirely at the navigation level, in addition to the RLS protection at the data level. Defense in depth.
- **AppShell / routing** — the cycle tracker route should not appear in Yara's BottomNav or page structure. RLS protects the data; routing protects the surface. Both layers are required.
- **No other feature** reads from these tables. The cycle tracker is a self-contained privacy island.

```sql
-- Run after migration to verify correctness
-- IMPORTANT: Run each verification block while authenticated as the
-- appropriate user (Yahya or Yara) to confirm RLS behavior.

-- 1. Both tables exist with correct columns
select column_name, data_type, is_nullable
from information_schema.columns
where table_schema = 'public'
  and table_name in ('cycle_config', 'cycle_logs')

  and constraint_type = 'UNIQUE';
-- Expected: 1 row for (owner_id, date) composite unique constraint

-- 7. Updated_at trigger exists on cycle_config
select trigger_name, event_manipulation, event_object_table
from information_schema.triggers
where trigger_name = 'cycle_config_set_updated_at';
-- Expected: 1 row, UPDATE, cycle_config

-- 8. CHECK constraint on cycle_logs.mood
```

Additionally, the auditor must manually verify RLS behavior in the Supabase SQL editor:

- **Authenticated as Yahya** → INSERT a row into `cycle_config`. SELECT from `cycle_config` returns exactly 1 row. INSERT a row into `cycle_logs`. SELECT from `cycle_logs` returns that row.
- **Authenticated as Yara** → SELECT from `cycle_config` returns 0 rows with no error. SELECT from `cycle_logs` returns 0 rows with no error. INSERT into either table fails (RLS `with check` rejects it because `auth.uid()` will not equal Yahya's `owner_id`).
- **Authenticated as Yara** → attempting to INSERT with `owner_id` set to Yahya's UUID must fail. The `with check (auth.uid() = owner_id)` policy prevents spoofing the owner.

## Do NOT

- Do not add a partner-read policy to either table — this is the one exception to the partner-access pattern used everywhere else in the app
- Do not use `auth.uid() = owner_id OR owner_id = partner_id` — this exact pattern from other tables is FORBIDDEN here
- Do not add a DELETE policy to `cycle_config` — the config row must persist; Yahya updates it, never deletes it
- Do not expose these tables through any Edge Function that does not enforce `auth.uid() = owner_id`
- Do not display cycle tracker UI elements in Yara's session — RLS is the data layer protection; routing is the UI layer protection — both are required
- Do not add `updated_at` to `cycle_logs` — entries are upserted, not tracked for modification history
- Do not use the service role key in any client code that touches these tables
- Do not add ovulation, fertility window, or conception-related columns — this is not a fertility tracker
- Do not seed any rows — Yahya configures on first use
- Do not redefine `set_updated_at()` — it was created in T201 and is reused here
- Do not skip the `comment on table` statements — the privacy intent must be documented in the schema itself
