# T107: AppShell — The Container of Everything

## Why This Exists

The AppShell is the invisible frame that holds everything together. It's the warm cream background that every page lives on. It's the reason the BottomNav is always there. It's the container that manages scroll behavior, safe areas, and the boundary between "app chrome" (nav) and "app content" (pages). The user never sees the AppShell directly — they feel it. It's the reason the app feels like one cohesive space rather than disconnected pages.

---

## User Story

As a user, every page I visit feels like it belongs to the same app. The bottom navigation is always present. Content scrolls smoothly without hiding behind the nav. On my iPhone, nothing gets clipped by the notch or home bar. The background is consistently warm cream. I never think about the container — I just think about what's inside it.

---

## Full UX Flow

1. I open the app → the entire viewport is filled with warm cream (`bg-primary` #FBF8F4)
2. At the bottom: BottomNav (always visible, never scrolls away)
3. Above the nav: page content scrolls freely
4. Content has enough bottom padding that the last item is never hidden behind the nav
5. On iPhone: the notch area at top and home bar area at bottom are respected — no interactive elements in those zones
6. Navigating between pages: the AppShell stays static, only the content area changes (BottomNav doesn't re-render or flicker)

```
│                              │
│     {children}               │ ← scrollable content area
│     (page components)        │
│                              │
│                              │
│     padding-bottom: ~96px    │ ← ensures content doesn't hide behind nav
│                              │
├──────────────────────────────┤
│     BottomNav                │ ← fixed to bottom, z-50
│     64px + safe area bottom  │
└──────────────────────────────┘ ← viewport bottom
```

## Implementation Details

```tsx
"use client"

import { BottomNav } from "@/components/shared/BottomNav"

export function AppShell({ children }: AppShellProps) {
  return (
    <div className="min-h-[100dvh] bg-bg-primary">
      <main className="pb-24">
        {children}
      </main>
      <BottomNav />
    </div>
  )
}
```

Key decisions:
- `min-h-[100dvh]` — uses dynamic viewport height (dvh) which accounts for mobile browser chrome. Better than `100vh` which causes content to extend behind browser UI on iOS Safari.
- `pb-24` (96px) on the content area — this is enough to clear the BottomNav (64px) plus breathing room. Content never gets hidden.
- `bg-bg-primary` on the outer div — the warm cream background extends to fill the viewport
- No horizontal padding on the shell — individual pages add their own `px-6` where needed, allowing full-bleed elements (like edge-to-edge cards or images)
- BottomNav handles its own fixed positioning — the shell just renders it

## Integration with Layout

After building this, modify `src/app/layout.tsx` to use AppShell:

```tsx
// In the body of layout.tsx:
<body className="...">
  <AppShell>
    {children}
  </AppShell>
</body>
```

**Important timing note:** In Phase 2 (Auth), the AppShell will need to be conditional — only wrapping authenticated routes, not the login page. For now (Phase 1), it wraps everything. The auth middleware (T205) will restructure this.

## Edge Cases

- Content shorter than viewport → cream background still fills the full screen (min-h-[100dvh] handles this)
- Very long scrollable content → content scrolls, nav stays fixed, bottom padding prevents overlap
- Keyboard open on mobile (input focused) → the nav might shift on some browsers. We accept this for V1 — it's standard PWA behavior.
- Landscape orientation → still works, nav at bottom, content fills available width

## Do NOT

- Don't add a top bar or status bar — that's the pages' job (via PageHeader)
- Don't add scroll-to-top buttons
- Don't add a backdrop blur on the nav area
- Don't add horizontal padding to the shell — pages control their own horizontal spacing
- Don't make the nav conditionally render based on route (that's for Phase 2 auth)
