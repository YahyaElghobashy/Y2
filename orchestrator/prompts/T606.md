## Why This Exists

A coupon is a promise. When Yahya gives Yara a "One Free Massage" coupon, it isn't just data — it's an intention, a memory, a contract between two people. CouponDetail is the moment where that promise becomes real. It's the full-screen view that says: this is what you're giving, or this is what you've been given. Every word, every image, every action button should feel intentional and warm.

This is the most action-dense view in the coupons module. The same component renders for both the creator and the recipient, but what they see — and what they can do — is completely different depending on their role and the coupon's current status. Getting this logic right is the core engineering challenge of this task.

---

## User Story

As Yara (recipient), when I tap a coupon in my wallet, I want to see the full detail: the emoji, the title, the description, any photo, who gave it to me, when it expires, and a clear action button so I know exactly what to do next.

As Yahya (creator), when I tap a coupon I created, I want to see the same detail but with my creator actions: I can approve or reject a pending redemption request, or reveal a surprise coupon I haven't surfaced yet.

In both cases, the view feels warm, focused, and tactile. The action button is never ambiguous.

## Full UX Flow

1. **Opening the detail view:**
   - CouponDetail receives a `coupon` prop and an `open` boolean
   - It renders as a bottom-sheet modal or full-screen overlay (use a `motion.div` that slides up from the bottom)
   - Background dims with a semi-transparent overlay (`bg-black/40`)
   - Tapping the overlay or the close button calls `onClose()`

2. **Reading the coupon content:**
   - Top: a back/close button (`←` or `×`) in the header area
   - Large emoji centered (64px, not an icon — a real rendered emoji character)
   - Title in Playfair Display, 24px, centered
   - Description in DM Sans, 14px, `text-secondary`, centered, wraps naturally
   - If a photo exists on the coupon (`coupon.imageUrl`), render it below the description as a rounded image (full width, max-h-48, object-cover, rounded-xl)
   - Category badge (pill shape, `bg-accent-soft`, `text-accent-primary`, 12px DM Sans) + creation date text side by side
   - Metadata rows: "From: [name]", "Status: [label with dot]", "Expires: [date or Never]"

3. **Action button area (most important):**
   - A single section at the bottom of the modal, above the safe area
   - The button rendered depends on the viewer's role and the coupon's status — see the full matrix below
   - For the reject flow: tapping "Reject" reveals a `textarea` (animated in with Framer Motion height expand) for an optional reason, then a "Confirm Reject" button replaces the original button row
   - After any action completes successfully, call `onStatusChange?.()` then `onClose()`

4. **Loading and error states:**
   - While an action is in flight (redeemCoupon, approveCoupon, rejectCoupon, revealSurprise), the action button shows a spinner and is disabled
   - On error, show a small red error message below the button — do not close the modal

## Action Button Logic — Full Matrix

This is the most important part of this component. Every combination of (viewer role) × (coupon status) must be handled explicitly.

### Recipient viewing their coupon

| Coupon Status | Button | Style | Behavior |
|---|---|---|---|
| `active` | "Redeem This Coupon" | `bg-accent-primary` copper, white text | Calls `redeemCoupon(coupon.id)`, shows spinner while loading |
| `pending_approval` | "Waiting for approval…" | `bg-warning/20` amber bg, `text-warning` text, disabled | Non-interactive. No onClick. Visually dimmed. |
| `redeemed` | "Redeemed ✓" | `bg-success/20` green bg, `text-success` text, disabled | Non-interactive. Celebrates the completed state. |
| `rejected` | "Rejected" + reason below | `bg-error/20` red bg, `text-error` text, disabled | Show `coupon.rejectionReason` in small text below the button if it exists |

### Creator viewing their own coupon

| Coupon Status | Button | Style | Behavior |
|---|---|---|---|
| `active` and `is_surprise === true` | "Reveal to [partnerName]" | `bg-accent-primary` copper, white text | Calls `revealSurprise(coupon.id)`. After success, the coupon becomes visible in the partner's wallet. |
| `active` and `is_surprise === false` | No action button rendered | — | Render a subtle text note: "This coupon is in [partnerName]'s wallet" in `text-secondary` 13px centered |
| `pending_approval` | "Approve" (left, 48%) + "Reject" (right, 48%) | Approve: `bg-success` green, white text. Reject: border `border-error`, `text-error`, transparent bg | Approve calls `approveCoupon(coupon.id)`. Reject initiates the reject flow (see below). |
| `redeemed` | "Approved ✓" | `bg-success/20`, `text-success`, disabled | Non-interactive. Marks end state. |
| `rejected` | No action button | — | Render subtle text: "You rejected this redemption" in `text-secondary` 13px |

### Reject Flow (creator taps "Reject")

1. The "Approve / Reject" button row animates out (Framer Motion `AnimatePresence`, height collapse)
2. A `textarea` animates in below: placeholder "Add a reason (optional)", `border-subtle`, rounded-xl, 3 rows, DM Sans 14px
3. Below the textarea: a "Confirm Reject" button full width, `border-error text-error` outline style
4. Next to or above: a "Cancel" text link that collapses the textarea and restores the original button row
5. On "Confirm Reject": calls `rejectCoupon(coupon.id, reason)` where `reason` is the textarea value (may be empty string)
6. Shows spinner on "Confirm Reject" while in flight

## Implementation Notes

- `"use client"` directive required (Framer Motion, hooks, event handlers)
- Use `useAuth()` from T204 to get `currentUser.id` — compare with `coupon.creatorId` to determine if viewer is creator or recipient
- Use `useCoupons()` from T602 to access `redeemCoupon`, `approveCoupon`, `rejectCoupon`, `revealSurprise`
- Use `useState` for: `isLoading`, `errorMessage`, `showRejectInput`, `rejectReason`
- Use `useRef<HTMLTextAreaElement>` for the reject reason textarea — auto-focus it when it appears
- The modal overlay uses `AnimatePresence` so the open/close transition is smooth
- Modal slides up: `initial={{ y: "100%" }}`, `animate={{ y: 0 }}`, `exit={{ y: "100%" }}`, spring transition `{ type: "spring", damping: 30, stiffness: 300 }`
- Backdrop fades: `initial={{ opacity: 0 }}`, `animate={{ opacity: 1 }}`, `exit={{ opacity: 0 }}`, duration 0.2
- The reject textarea expansion uses `motion.div` with `initial={{ height: 0, opacity: 0 }}`, `animate={{ height: "auto", opacity: 1 }}`
- Format the creation date with `format(new Date(coupon.createdAt), "MMM d")` — date-fns is already in the project
- Format expiry: if `coupon.expiresAt` is null, display "Never"; otherwise format as `"MMM d, yyyy"`
- Use `cn()` from `@/lib/utils` for all conditional class logic
- Use `Loader2` from Lucide with `animate-spin` for the loading spinner inside buttons
- Use `ArrowLeft` from Lucide for the back/close button in the header
- Never derive the partner's name from hardcoded strings — get it from the auth context or coupon metadata

## Edge Cases

- **Coupon with no description** — skip the description element entirely, do not render an empty paragraph
- **Coupon with no image** — skip the image block entirely, no placeholder shown
- **Very long title** — allow wrapping, do not truncate; Playfair Display at 24px wraps gracefully
- **Very long description** — cap at 4 lines with `line-clamp-4`, show full text on expand (future feature, for now just clamp)
- **Action in flight, user taps backdrop** — do not close the modal while `isLoading` is true; ignore the backdrop tap
- **rejectReason is whitespace only** — trim before passing to `rejectCoupon`; if empty after trim, pass `undefined` not an empty string
- **Both users open the same coupon simultaneously** — Supabase realtime (T602) will push status updates; the hook should reflect the new status automatically
- **Unknown status value** — render nothing in the action area and log a warning; do not crash

## Do NOT

- Don't hardcode `"Yahya"` or `"Yara"` — derive names from auth context or coupon data
- Don't use `left`/`right` CSS properties — use logical properties (`start`, `end`) for RTL readiness
- Don't close the modal on backdrop tap while an action is loading
- Don't show the action button area at all for unrecognized status values — fail silently
- Don't use `alert()` or `window.confirm()` for the reject confirmation — use the inline textarea flow
- Don't hardcode hex colors — use Tailwind design tokens only
- Don't use `router.push()` — this is a modal, not a page navigation
- Don't forget `AnimatePresence` around the reject textarea — the height animation requires it
