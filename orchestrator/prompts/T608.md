## Why This Exists

Every coupon in Hayah has a life story. It was created by one person, maybe revealed, maybe redeemed, maybe approved or rejected. This history is a quiet witness to the care exchanged between Yahya and Yara. The CouponHistory component makes that story visible тАФ a timeline that lets both users see exactly what happened to a coupon and when.

This isn't a debug log. It's a memory. "You redeemed that massage coupon on March 5th" тАФ that's a moment in the relationship. The visual should feel like a soft timeline entry in a journal, not a system audit table. Warm, readable, unobtrusive.

---

## User Story

As Yahya or Yara, when I open a coupon detail or browse the Coupons tab, I can see a chronological list of everything that happened to that coupon тАФ who created it, when it was redeemed, whether it was approved or rejected, and any notes left along the way. The actor is shown as "you" when it was me, or my partner's name when it was them. Each event type has a distinct visual cue so I can scan the timeline at a glance.

---

## Full UX Flow

1. **CouponDetail opens** (T606) тЖТ CouponHistory renders below the coupon card, scoped to that `couponId`
2. **Coupons tab global feed** тЖТ CouponHistory renders without a `couponId`, showing a combined feed of all recent coupon events for coupons the current user is involved with
3. **Each history entry appears as a timeline row:**
   - A colored dot on a vertical line indicates the event type
   - The action label names what happened and who did it
   - If the event had a coupon title context (global feed), the coupon name is shown below the action
   - A timestamp is shown below in muted text (e.g., "Mar 1 ┬╖ 2:30 PM")
   - If a note was left, it appears below the timestamp in italic muted text
4. **Actor resolution:** if `actor_id` matches the current user's ID тЖТ display "you", otherwise display the partner's first name
5. **StaggerList (T103) animates the entries in** тАФ each row fades and slides up with a staggered delay as the list mounts
6. **If no history exists** тЖТ render an EmptyState (T105): "No activity yet" with a Clock icon
7. **Limit prop** defaults to 20 entries; no pagination in v1

## Implementation Notes

- `"use client"` directive required тАФ uses hooks (`useAuth`, `useState`, `useEffect`)
- Import Supabase client from `@/lib/supabase/client` (T202)
- Import `useAuth` from `@/hooks/use-auth` (T204) to get `currentUser.id` and partner name
- **When `couponId` is provided:** query `coupon_history` where `coupon_id = couponId`, ordered by `created_at ASC`, limit to `limit` rows
- **When no `couponId`:** query `coupon_history` joined with `coupons` where `coupons.creator_id = currentUser.id OR coupons.recipient_id = currentUser.id`, order by `coupon_history.created_at DESC`, limit to `limit` rows; include `coupons.title` in the select
- **Actor resolution:** compare `actor_id` to `currentUser.id` тАФ if match, use "you"; otherwise use partner's first name from auth context
- **Date formatting:** use `format(new Date(created_at), 'MMM d ┬╖ h:mm a')` тАФ `date-fns` is already in package.json
- Use `StaggerList` (T103) to wrap the list of `CouponHistoryEntry` sub-components for staggered mount animation
- Render `EmptyState` (T105) when fetched data is an empty array тАФ icon: `Clock`, title: "No activity yet", subtitle: "Events will appear here as they happen"
- Show a loading skeleton while fetching: 3 placeholder rows using `LoadingSkeleton` (T104) variant `list`
- The timeline line is a CSS pseudo-element or a positioned `<div>` spanning from the first to last dot тАФ use a relative container with a 2px left border on the inner wrapper

## Edge Cases

- **`expired` action has no actor:** skip actor resolution entirely; label is always "Expired"
- **`note` is null:** omit the note element entirely тАФ do not render an empty `<p>` tag
- **Global feed with many coupons:** `limit` prop caps the query; no pagination in v1 тАФ a "View more" link is a future task
- **Both users have activity interleaved:** global feed orders by `created_at DESC` so newest events appear first regardless of which coupon they belong to
- **Timestamps are stored in UTC:** ensure `date-fns` formats in local time тАФ `new Date(created_at)` handles this automatically in the browser
- **Actor is current user on both "you" ends:** if Yahya views the timeline and Yahya both created and approved, both show "you" тАФ this is correct and intentional
