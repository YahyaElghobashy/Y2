## Why This Exists

When Yara taps a marketplace item — say, "Extra Notification Sends" — she should feel the weight of the decision before it's made. CoYYns are earned slowly and spent intentionally. This modal is a moment of pause: here is what you're buying, here is what it costs, here is what you'll have left. The balance breakdown isn't bureaucracy — it's care. It says: we see your balance, we see your intent, we confirm together.

The modal must feel decisive and warm, never alarming. A disabled confirm button with a "Not enough CoYYns" message is not a failure state — it's the system being honest. No shame, just clarity.

---

## User Story

As Yahya or Yara, when I tap a marketplace item I want to buy, a modal appears confirming the details before any CoYYns are spent. I can see the item, its cost, my current balance, and exactly what I'll have left. If I can afford it, I tap Confirm and it happens. If I can't, I'm told clearly without drama. I can always cancel and go back.

---

## Full UX Flow

1. **User taps a marketplace item** on the marketplace page (T701)
2. **Modal opens** with `scaleIn` animation — scales from 0.95 to 1.0, 200ms ease-out
3. **Modal shows:**
   - Item icon + title at the top
   - A balance breakdown table: Cost, Current Balance, Balance After
   - If affordable: "Balance After" in `accent-primary` (#C4956A)
   - If not affordable: "Balance After" shows negative number in `error` (#C27070), a "Not enough CoYYns" message appears below the breakdown, Confirm button is disabled
4. **User taps "Confirm Purchase":**
   - Button enters loading state (spinner, text changes to "Processing…", disabled)
   - Calls `spendCoyyns(price, title, category)` from `useCoyyns` hook
   - On success: modal closes, toast fires ("Purchased! 🎉"), `onSuccess` callback is called
   - On error: loading clears, inline error message appears below breakdown ("Something went wrong — please try again")
5. **User taps "Cancel"** → modal closes, no action taken, no state mutation

## Implementation Notes

- `"use client"` directive required (Framer Motion, useCoyyns hook, React state)
- Use `AnimatePresence` + `motion.div` for the overlay and modal panel
- Modal entry animation: `initial={{ scale: 0.95, opacity: 0 }}` → `animate={{ scale: 1, opacity: 1 }}` → `exit={{ scale: 0.95, opacity: 0 }}`, duration 0.2s, ease `[0.25, 0.46, 0.45, 0.94]`
- Overlay animation: `initial={{ opacity: 0 }}` → `animate={{ opacity: 1 }}` → `exit={{ opacity: 0 }}`, duration 0.15s
- Use `useCoyyns()` from T302 to read `balance` and call `spendCoyyns(amount, description, category)`
- Balance calculation: `const remaining = balance - item.price` — compute at render time, not in effect
- `canAfford = remaining >= 0` — gates the confirm button and controls "After" value color
- Loading state: local `useState<boolean>` — set true on confirm tap, false on resolve/reject
- Error state: local `useState<string | null>` — cleared on modal open (`useEffect([open])`)
- On confirm: wrap `spendCoyyns` call in try/catch inside async handler
- On success: call `onClose()` first, then `onSuccess?.()`, then fire toast
- Toast: use the project's existing toast utility — do not install a new library
- Format CoYYns values with `toLocaleString()` for thousands separator (e.g. 1,250)
- Use `cn()` from `@/lib/utils` for all conditional class merging
- Use design tokens only — no hardcoded hex colors anywhere
- Close modal on overlay click (tap outside) — same as pressing Cancel
- Do NOT close modal on content area click (stop propagation)
- Use `role="dialog"` and `aria-modal="true"` on the modal panel for accessibility

## Edge Cases

- **Balance is exactly equal to price** → `remaining === 0`, still affordable, "After" shows `0 CoYYns` in `accent-primary`
- **Balance is 0** → modal opens, confirm disabled immediately, "Not enough CoYYns" visible on open
- **Balance changes between modal open and confirm tap** → `spendCoyyns` in T302 must validate server-side; if it rejects, surface the error inline without closing modal
- **Price is 0** → not expected in current marketplace spec, but if it occurs: treat as affordable, confirm succeeds trivially
- **item.icon is a complex React node** → render it as-is in a 24px container with `shrink-0`
- **Very long item title** → truncate with `line-clamp-2` to prevent layout break
- **Rapid double-tap on Confirm** → loading state disables the button after first tap, so second tap is a no-op
- **Modal opened on a slow connection** → balance from `useCoyyns` may be stale — this is acceptable; server-side check in T302 is the source of truth

## Do NOT

- Don't call Supabase directly — all spending goes through `spendCoyyns` from `useCoyyns` (T302)
- Don't hardcode color hex values — use Tailwind tokens from the design system
- Don't use `alert()` or `window.confirm()` anywhere
- Don't close the modal automatically on error — keep it open so user can retry or cancel
- Don't add sound effects or haptic feedback
- Don't show the full CoYYns transaction history in this modal — that belongs in T302/T306 views
- Don't install new packages — use AnimatePresence and motion from the existing Framer Motion dependency
