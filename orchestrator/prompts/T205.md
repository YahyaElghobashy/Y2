## Why This Exists

Hayah is a private app for two people. It holds health data, relationship rituals, spiritual practices, and daily routines. None of that should be visible to anyone who is not Yahya or Yara. Without middleware, a direct URL visit to `/health` or `/us` would render the full page before any client-side auth check could redirect — potentially flashing private content.

The middleware runs at the edge, before any page renders. It catches unauthenticated requests to protected routes and redirects immediately to `/login`, with zero content flash. It also refreshes the Supabase session on every request so the JWT never silently expires mid-session. This is the security perimeter of the entire app.

---

## User Story

As Yahya or Yara, I expect that the app never shows me anyone else's data, and that I never see a private page if I'm not logged in. When I open a deep link to `/health` and my session has expired, I should be taken to the login page immediately — not shown a broken or empty page first. When I log in, I should land back on the protected page I was trying to reach.

---

## Full UX Flow

1. **User visits any protected route unauthenticated** (e.g., direct link to `/health`):
   - Middleware intercepts the request
   - No session found
   - Redirects to `/login?redirectTo=/health`
   - Login page renders — BottomNav is NOT visible (auth layout has no AppShell)

2. **User completes login successfully**:
   - Auth callback reads `redirectTo` param from URL
   - Redirects user to `/health` (or `/` if no redirectTo)
   - Middleware runs again, finds valid session, allows request through
   - AppShell with BottomNav renders around the page

3. **User visits `/login` while already authenticated**:
   - Middleware detects valid session
   - Redirects immediately to `/` — no login page rendered

4. **User is mid-session (page refresh, tab switch, background/foreground)**:
   - Middleware calls `updateSession()` on every request
   - Supabase refreshes the JWT if it is close to expiry
   - User experiences no interruption

5. **User visits a static asset or API route** (`/_next/`, `/api/`, `/favicon.ico`, images):
   - Middleware matcher config excludes these paths entirely
   - Zero overhead on asset requests

6. **Layout selection based on route group**:
   - Routes under `app/(auth)/` — e.g., `app/(auth)/login/page.tsx` — use `app/(auth)/layout.tsx`, which renders NO AppShell and NO BottomNav
   - All other routes use `app/layout.tsx` which wraps content in AppShell (BottomNav included)

## Implementation Notes

- The middleware file lives at `src/middleware.ts` — Next.js 15 with the `src/` directory convention picks this up automatically
- Call `createMiddlewareClient(request)` from T202 to get `{ supabase, response }`
- Call `supabase.auth.getUser()` — NOT `getSession()`. Supabase SSR docs are explicit: `getSession()` reads from the cookie without server-side validation; `getUser()` validates the JWT with Supabase servers and is the safe call for middleware
- After the auth check, always return the `response` object from `createMiddlewareClient`, not a plain `NextResponse.next()` — the middleware client mutates `response` to write refreshed session cookies back
- Protected routes list: `['/','/ us','/health','/spirit','/ops','/settings']` — use `startsWith` matching so nested routes like `/health/log` are also protected
- Redirect to login preserves the intended destination: `?redirectTo=${request.nextUrl.pathname}`
- The `(auth)` and `(app)` route groups are Next.js parallel layout segments — they do NOT appear in the URL. `/login` stays `/login`, `/health` stays `/health`
- The root `app/layout.tsx` should only contain `<html>`, `<body>`, global providers (AuthProvider from T204), and `{children}` — no shell logic here
- `app/(auth)/layout.tsx` renders a centered, full-screen container appropriate for a login form — no nav, no padding assumptions from AppShell
- `app/(app)/layout.tsx` renders `<AppShell>{children}</AppShell>` — this is where the BottomNav lives, via AppShell from T107

## Edge Cases

- **JWT expired but cookie present** — `getUser()` will return an error; treat this the same as no session and redirect to `/login`. Do not assume a cookie means a valid session.
- **Supabase unreachable during middleware** — if `getUser()` throws a network error, fail open (allow the request through) rather than redirect-looping users. Log the error. Do not hard crash the middleware.
- **Redirect loop guard** — if the request is already going to `/login` and there is no session, do not redirect again. Check `request.nextUrl.pathname === '/login'` before any redirect logic.
- **`/settings` is protected** — it is not in the BottomNav tabs but it is still a private route. The middleware must protect it.
- **Trailing slash variants** — `/health/` and `/health` should both be protected. Normalize by checking `startsWith` after removing trailing slash, or use the pathname directly since Next.js normalizes trailing slashes by default.
- **`/api/` routes** — these are excluded by the matcher. If API routes need their own auth, they use the Supabase server client directly (T203), not this middleware.
- **Static file false positives** — the matcher regex must be tight enough to exclude all `_next/` assets, image optimization routes, and common static extensions to avoid running auth logic on every font or image request.

- **T204 (AuthProvider)** — the root layout's AuthProvider gives client components access to auth state; middleware is the server-side complement to this

- **T101 (BottomNav)** — lives inside AppShell; because the auth layout never mounts AppShell, the BottomNav is automatically absent on `/login`
- **T206 (Login Page)** — depends on this middleware being in place to handle the redirect-after-login flow and to redirect already-authenticated users away from `/login`
- **T207 (Auth Callback)** — the OAuth/magic-link callback route reads `redirectTo` from the URL after session exchange; this middleware writes the `redirectTo` param when bouncing users to login

## Do NOT

- Do not use `getSession()` — use `getUser()` for all server-side auth validation
- Do not return `NextResponse.next()` directly — always return the `response` from `createMiddlewareClient` so cookies are written
- Do not put AppShell in `app/layout.tsx` — it belongs in `app/(app)/layout.tsx` only
- Do not protect `/api/` routes here — API routes handle their own auth via the server client
- Do not hardcode the list of public routes as a single string match — use `startsWith` or pathname prefix checks so nested routes are covered correctly
- Do not add business logic to the middleware — it does exactly two things: refresh the session and enforce route access
- Do not import React or any client-side code into `middleware.ts` — it runs in the Next.js Edge Runtime
