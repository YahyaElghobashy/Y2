## Why This Exists

CoYYns are the currency of Hayah. The two YYs in the name are Yahya and Yara — it is their coin, their game, their private economy. When Yahya completes a challenge, he earns CoYYns. When Yara wants to redeem a love coupon, she spends them. The system is deliberately playful and non-transactional — no real money, no real stakes, just a shared incentive layer that makes the app feel alive.

But the playfulness is built on a serious foundation. A currency system demands an immutable ledger. Balances cannot be set arbitrarily from the client — they must be computed from a history of earn and spend events. If a bug in the UI could subtract CoYYns without a corresponding transaction record, the system becomes untrustworthy and the game breaks. This migration establishes the two tables that make CoYYns safe: `coyyns_wallets` (the current state) and `coyyns_transactions` (the source of truth).

The wallet balance is never written directly from the client. It is always a derived value, maintained by a database trigger that fires on every transaction insert. The client can only create transactions — the balance takes care of itself.

---

## User Story

As Yahya or Yara, my CoYYns balance is always accurate, always visible, and always trustworthy. When I earn CoYYns (by completing a challenge, logging a check-in, or receiving a gift from my partner), the balance goes up immediately. When I spend them (on marketplace items, love coupons, or app customizations), the balance goes down. I can always trust the number I see because it reflects every earn and spend event in my history — no manual edits, no surprises. My partner can see my balance too, so the game is transparent and fair.

## Full DB Flow

### Earning CoYYns
1. An event occurs — a challenge is completed, a check-in is logged, or a feature grants CoYYns
2. The app calls the relevant Supabase function or client mutation
3. A row is inserted into `coyyns_transactions` with `type = 'earn'` and a positive `amount`
4. The `handle_coyyn_transaction` trigger fires on INSERT
5. The trigger UPDATE on `coyyns_wallets`: increments `balance` by `amount`, increments `lifetime_earned` by `amount`
6. The wallet row reflects the new balance immediately (within the same transaction)

### Spending CoYYns
1. The user taps a marketplace item, redeems a coupon, or triggers a spend event
2. The app inserts a row into `coyyns_transactions` with `type = 'spend'` and a negative `amount`
3. The `handle_coyyn_transaction` trigger fires on INSERT
4. The trigger attempts to UPDATE `coyyns_wallets`: decrements `balance` by `abs(amount)`, increments `lifetime_spent` by `abs(amount)`
5. If `balance - abs(amount) < 0`, the trigger raises an exception — the transaction is rolled back
6. The wallet row reflects the new balance only if the balance constraint is satisfied

### Viewing a Partner's Balance
1. User navigates to `/us` page
2. The `useCoyyns` hook (T302) fetches both wallet rows using the authenticated user's `partner_id`
3. RLS allows reading the partner's wallet row because the partner read policy is satisfied
4. Both balances are displayed side by side

## SQL Schema

**File to create:** `supabase/migrations/002_coyyns.sql`

```sql
-- ============================================================
-- T301: CoYYns Database Migration
-- Creates wallets, transactions, RLS, balance trigger, and seed
-- Depends on: 001_auth_profiles.sql (profiles table must exist)
-- ============================================================


-- ── 1. COYYNS WALLETS TABLE ───────────────────────────────────

create table public.coyyns_wallets (
  id               uuid        primary key default gen_random_uuid(),
  user_id          uuid        not null unique references public.profiles (id) on delete cascade,
  balance          integer     not null default 0 check (balance >= 0),
  lifetime_earned  integer     not null default 0 check (lifetime_earned >= 0),
  lifetime_spent   integer     not null default 0 check (lifetime_spent >= 0),
  created_at       timestamptz not null default now(),
  updated_at       timestamptz not null default now()
);

-- Index on user_id for fast wallet lookups by authenticated user
create index coyyns_wallets_user_id_idx on public.coyyns_wallets (user_id);

comment on table public.coyyns_wallets is
  'One row per user. Stores current CoYYns balance and lifetime earn/spend totals. Balance is maintained exclusively by trigger — never write directly from client.';
comment on column public.coyyns_wallets.balance is
  'Current spendable balance. Always >= 0. Written only by handle_coyyn_transaction trigger.';
comment on column public.coyyns_wallets.lifetime_earned is
  'Cumulative total CoYYns earned across all time. Never decreases.';
comment on column public.coyyns_wallets.lifetime_spent is
  'Cumulative total CoYYns spent across all time. Never decreases.';


-- ── 2. COYYNS TRANSACTIONS TABLE ─────────────────────────────

create table public.coyyns_transactions (
  id           uuid        primary key default gen_random_uuid(),
  user_id      uuid        not null references public.profiles (id) on delete cascade,
  amount       integer     not null,
  type         text        not null check (type in ('earn', 'spend')),
  category     text        not null,
  description  text,
  metadata     jsonb       not null default '{}',
  created_at   timestamptz not null default now()
);

-- Index on user_id for fetching a user's transaction history in order
create index coyyns_transactions_user_id_created_at_idx
  on public.coyyns_transactions (user_id, created_at desc);

-- Index on category for analytics queries (most common earn/spend sources)
create index coyyns_transactions_category_idx on public.coyyns_transactions (category);

comment on table public.coyyns_transactions is
  'Immutable ledger of all CoYYns earn and spend events. Positive amount for earn, negative for spend. Source of truth for balance.';
comment on column public.coyyns_transactions.amount is
  'Positive integer for earn events, negative integer for spend events. The trigger uses sign to determine direction.';
comment on column public.coyyns_transactions.type is
  'earn or spend. Must match the sign of amount — earn must be positive, spend must be negative.';
comment on column public.coyyns_transactions.category is
  'Machine-readable source of the transaction. E.g. manual, challenge, marketplace_purchase, notification_purchase, check_in, coupon_redeem.';
comment on column public.coyyns_transactions.metadata is
  'Flexible JSON bag for extra context. E.g. { "challenge_id": "...", "item_name": "..." }. Never required, always safe to be empty.';


-- ── 3. UPDATED_AT TRIGGER FOR WALLETS ────────────────────────

create trigger coyyns_wallets_set_updated_at
  before update on public.coyyns_wallets
  for each row
  execute function public.set_updated_at();
-- Note: set_updated_at() was defined in 001_auth_profiles.sql and is reused here.


-- ── 4. BALANCE TRIGGER ───────────────────────────────────────
-- Fires on every INSERT to coyyns_transactions.
-- Updates the wallet row for the transacting user.
-- This is the ONLY place balance is ever modified.

create or replace function public.handle_coyyn_transaction()
returns trigger
language plpgsql
security definer set search_path = public
as $$
begin
  -- Validate: earn transactions must have positive amount
  if new.type = 'earn' and new.amount <= 0 then
    raise exception 'Earn transactions must have a positive amount. Got: %', new.amount;
  end if;

  -- Validate: spend transactions must have negative amount
  if new.type = 'spend' and new.amount >= 0 then
    raise exception 'Spend transactions must have a negative amount. Got: %', new.amount;
  end if;

  -- Upsert the wallet row (handles the case where a wallet doesn't exist yet,
  -- though wallets are seeded at user creation time)
  insert into public.coyyns_wallets (user_id, balance, lifetime_earned, lifetime_spent)
  values (new.user_id, 0, 0, 0)
  on conflict (user_id) do nothing;

  if new.type = 'earn' then
    update public.coyyns_wallets
    set
      balance         = balance + new.amount,
      lifetime_earned = lifetime_earned + new.amount
    where user_id = new.user_id;
  else
    -- spend: amount is negative, so we subtract abs(amount) from balance
    -- The CHECK (balance >= 0) constraint on the table will reject this
    -- UPDATE if it would drive balance below zero, rolling back the transaction.
    update public.coyyns_wallets
    set
      balance        = balance + new.amount,   -- new.amount is negative
      lifetime_spent = lifetime_spent + abs(new.amount)
    where user_id = new.user_id;
  end if;

  return new;
end;
$$;

create trigger on_coyyn_transaction_insert
  after insert on public.coyyns_transactions
  for each row
  execute function public.handle_coyyn_transaction();


-- ── 5. ROW LEVEL SECURITY — WALLETS ──────────────────────────

alter table public.coyyns_wallets enable row level security;

-- Policy: users can read their own wallet
create policy "coyyns_wallets: own read"
  on public.coyyns_wallets
  for select
  using (auth.uid() = user_id);

-- Policy: users can read their partner's wallet (for /us page balance display)
create policy "coyyns_wallets: partner read"
  on public.coyyns_wallets
  for select
  using (
    user_id = (
      select partner_id
      from public.profiles
      where id = auth.uid()
    )
  );

-- No INSERT policy from client — wallets are seeded at migration time and
-- auto-created by the trigger if missing. Clients never insert wallet rows.
-- No UPDATE policy from client — balance is maintained exclusively by trigger.
-- No DELETE policy — wallets are permanent.


-- ── 6. ROW LEVEL SECURITY — TRANSACTIONS ─────────────────────

alter table public.coyyns_transactions enable row level security;

-- Policy: users can read their own transactions
create policy "coyyns_transactions: own read"
  on public.coyyns_transactions
  for select
  using (auth.uid() = user_id);

-- Policy: users can insert transactions for themselves only
create policy "coyyns_transactions: own insert"
  on public.coyyns_transactions
  for insert
  with check (auth.uid() = user_id);

-- No UPDATE policy — the ledger is immutable. Transactions are never edited.
-- No DELETE policy — the ledger is append-only.
-- Note: Partners cannot read each other's transactions (only balances via wallets).
-- If a "shared history" feed is needed in the future, add a partner read policy here.


-- ── 7. SEED DATA ─────────────────────────────────────────────
-- Creates a wallet row for each of the two pre-seeded users.
-- Uses the same placeholder UUIDs as 001_auth_profiles.sql.
-- Replace with the same real UUIDs you used in that migration.
--
-- This seed is safe to run after 001_auth_profiles.sql seed has been run
-- (i.e., after both profile rows exist in public.profiles).

do $$
declare
  yahya_id uuid := '00000000-0000-0000-0000-000000000001'; -- REPLACE with real UUID
  yara_id  uuid := '00000000-0000-0000-0000-000000000002'; -- REPLACE with real UUID
begin
  -- Insert wallets for both users (balance starts at 0)
  insert into public.coyyns_wallets (user_id, balance, lifetime_earned, lifetime_spent)
  values
    (yahya_id, 0, 0, 0),
    (yara_id,  0, 0, 0)
  on conflict (user_id) do nothing;
end;
$$;
```

## Implementation Notes

- **File location:** `supabase/migrations/002_coyyns.sql` — Supabase CLI processes migrations in numeric order. This must run after `001_auth_profiles.sql` because it references `public.profiles`.
- **`set_updated_at()` reuse:** The function was defined in migration 001. Migration 002 creates the trigger that calls it but does not redefine the function. If 002 is ever run standalone (e.g., in a test environment), 001 must have been run first.
- **Trigger as single writer for balance:** The `handle_coyyn_transaction` trigger is the only place `coyyns_wallets.balance` is ever modified. No RLS UPDATE policy exists for wallets. No application code should ever write to `balance`, `lifetime_earned`, or `lifetime_spent` directly. Enforcing this at the DB layer (no UPDATE policy) means a buggy client cannot corrupt the ledger.
- **`CHECK (balance >= 0)` as the guard:** When a spend transaction would drive the balance below zero, the trigger's UPDATE will violate this check constraint. Postgres rolls back the entire INSERT transaction — the transaction row is never committed, and the balance is unchanged. The client receives a `23514` constraint violation error, which the `useCoyyns` hook (T302) must catch and surface as an "insufficient CoYYns" message.
- **Amount sign convention:** Earn rows have a positive `amount`. Spend rows have a negative `amount`. This makes it trivial to sum all transactions and arrive at the correct balance: `SUM(amount) = balance`. The `type` column is a human-readable redundancy and an index target, not the source of sign truth.
- **`security definer` on `handle_coyyn_transaction`:** Required so the trigger can UPDATE `coyyns_wallets` as the function owner (service role), bypassing the no-UPDATE-policy restriction that protects against direct client writes. The trigger itself is the controlled write path.
- **`on conflict (user_id) do nothing` in trigger:** Defensive upsert in the trigger body. If a wallet row somehow doesn't exist (missed seed, testing environment), the trigger creates it rather than crashing. This makes the trigger self-healing without exposing a client insert path.
- **Compound index on `(user_id, created_at desc)`:** Transaction history queries will always filter by user and sort newest-first. This index makes those queries index-only scans for reasonable history lengths (< 10,000 rows per user, which is years of usage).
- **`metadata jsonb`:** Kept flexible intentionally. Challenge completions may store `challenge_id`. Marketplace purchases may store `item_id` and `item_name`. Notification purchases may store `notification_type`. The schema does not enforce metadata shape — that is the responsibility of the feature writing the transaction.
- **No `updated_at` on transactions:** Transactions are immutable. An `updated_at` column would be misleading — it implies the row can be updated, which it cannot by policy.

## Edge Cases

- **User attempts to spend more than their balance:** The trigger's `balance + new.amount` resolves to a negative number. The `CHECK (balance >= 0)` constraint fires. Postgres rolls back the INSERT. The client receives error code `23514`. The `useCoyyns` hook must handle this gracefully and show a user-friendly message, never a raw Postgres error string.
- **Two concurrent spend transactions from the same user:** Because the trigger issues an UPDATE with a WHERE clause, Postgres serializes concurrent updates to the same wallet row via row-level locking. The second concurrent transaction will wait until the first completes, then proceed (or fail if balance is now insufficient). There is no race condition.
- **Wallet row missing at transaction time:** The `on conflict (user_id) do nothing` INSERT in the trigger handles this by creating the wallet with zero balance before the UPDATE. However, the wallet seed in section 7 should have created it already. This path is purely defensive.
- **Earn transaction with `amount = 0`:** The `amount > 0` check in the trigger body raises an exception. Zero-value transactions are meaningless and should never be created.
- **`type` and `amount` sign mismatch:** e.g., `type = 'earn'` but `amount = -5`. The trigger's explicit validation catches this and raises an exception before any balance change occurs. The client-side `useCoyyns` hook (T302) should also validate this before inserting — defense in depth.
- **Category values not in the expected set:** The `category` column is unconstrained text. If a new feature (e.g., challenge rewards in T401) introduces a new category string, no migration is required. The tradeoff is that typos (`'chalenge'` vs `'challenge'`) will not be caught by the database. The client code must use constants or an enum type from the TypeScript types file.
- **Seed run before profile rows exist:** If the seed block in this migration runs before the profile rows from 001 exist, the INSERT will fail on the FK constraint. The correct order is: run 001, create auth users, run 001 seed block, run 002 (which is this file including the seed block).
- **Partner cannot see transaction history, only balance:** The current RLS allows partners to read wallet rows (balance + lifetime totals) but not transaction rows. This is intentional for privacy — individual earn/spend events are personal. If the product requires a shared transaction feed, add a partner read policy on `coyyns_transactions` in a future migration, not here.

## What This Connects To

- **T201 (Auth DB Migration)** — this migration depends on `public.profiles` existing. The `user_id` FK in both tables references `profiles.id`.
- **T302 (useCoyyns hook)** — the React hook that the entire client uses to read balances and insert transactions. It calls Supabase directly against these two tables and handles the spend-rejection error code.
- **T303 (CoYYns UI on /us page)** — renders both users' wallet balances side by side. Uses the partner-read wallet policy established here.
- **T401 (Challenges feature)** — when a challenge is completed, it inserts a row into `coyyns_transactions` with `category = 'challenge'`. The `metadata` field will carry `challenge_id`.
- **T501 (Marketplace feature)** — spend transactions for marketplace items will insert with `category = 'marketplace_purchase'` and `metadata = { item_id, item_name }`.
- **T601 (Notification Store)** — spend transactions for premium notification types will insert with `category = 'notification_purchase'`.
- **All future CoYYns-earning features** — any feature that rewards CoYYns inserts a row here. The wallet updates automatically via the trigger. No feature needs to know about wallet mechanics; they only need to know the correct `category` string and `amount`.

## Do NOT

- Do not add a client-side `UPDATE` policy to `coyyns_wallets` — the balance must only ever be modified by the trigger. A client that can write `balance` directly can cheat.
- Do not use `float` or `numeric` types for `amount` or `balance` — CoYYns are whole numbers. Integer arithmetic has no rounding surprises.
- Do not allow negative values in `balance` — the `CHECK (balance >= 0)` constraint is the last line of defense. Never remove it.
- Do not store `amount` as always-positive and rely solely on `type` to determine direction — the sign convention on `amount` makes aggregation trivial (`SUM(amount) = balance`) and must be preserved.
- Do not add a partner read policy to `coyyns_transactions` in this migration — that is a product decision for a future task, not an assumption to make here.
- Do not grant the anon role any access to either table — CoYYns data is authenticated-users-only.
- Do not create a `DELETE` policy on either table — the ledger is immutable and wallets are permanent.
- Do not add an `UPDATE` trigger to `coyyns_transactions` — the table has no `updated_at` and its rows should never change after insert.
- Do not hardcode category strings anywhere other than a constants file in TypeScript — the database does not constrain categories, so the application layer is the only enforcement point.
- Do not run this migration before `001_auth_profiles.sql` — the FK constraint on `user_id → profiles.id` will fail if the `profiles` table does not exist.
- Do not skip the `set search_path = public` hardening on the trigger function.
