## Why This Exists

Every notification Yahya sends to Yara (or vice versa) carries weight. The send limit exists to keep things intentional — you don't spam someone you love. But sometimes life calls for more. An extra nudge on a hard day. A second reminder about a date. A message that needed to be sent twice.

BuyExtraSendModal is the moment where care becomes tangible. You're spending CoYYns — the shared currency of this relationship — to say "this message matters enough to send again." The modal needs to feel deliberate and warm, not transactional. It confirms your balance, shows you the cost, and lets you choose how much to spend. No friction beyond what's necessary.

This modal is the connection between the CoYYns economy (T302) and the notification system (T402). It's where both systems meet.

---

## User Story

As Yahya or Yara, when I've used all my free sends for the day and I really need to send another notification, I can open this modal to buy extra sends using CoYYns. I see exactly how many CoYYns I have, what each extra send costs, and I can choose to buy more than one if I want. After confirming, my balance updates and I can immediately send my notification.

---

## Full UX Flow

1. **Modal opens** — triggered from "Buy more" link in SendLimitIndicator (T406) or NotificationBuilder
2. **Balance check on open** — if balance < 25, the quantity selector is locked at 0 and a "Not enough CoYYns" message appears
3. **Quantity selector** — shows a centered counter with — and + buttons
   - Min value: 1 (or 0 if balance < 25)
   - Max value: `Math.floor(balance / 25)` — you can only buy as many as you can afford
   - Default: 1 (if affordable)
4. **Cost display** — updates live as quantity changes: `quantity × 25 CoYYns`
5. **Tap "Buy N Extra Sends"** — button label reflects current quantity
6. **On confirm:**
   - Call `spendCoyyns(cost, "Extra notification sends", "notification_purchase")` from `useCoyyns`
   - On success: call the notifications hook to increment `bonus_sends_available` by quantity in `daily_send_limits`
   - Close modal
   - Show toast: "X extra send(s) purchased!"
7. **On error** — display inline error below the buy button; modal stays open
8. **Cancel** — close modal, no changes made

## Implementation Notes

- `"use client"` directive required (Framer Motion, hooks, state)
- Use `AnimatePresence` + `motion.div` for the modal backdrop and panel — use `scaleIn` variant from DESIGN_SYSTEM.md for the panel entrance
- Backdrop: `initial={{ opacity: 0 }}` → `animate={{ opacity: 1 }}` → `exit={{ opacity: 0 }}`, duration 200ms
- Panel: `initial={{ opacity: 0, scale: 0.95 }}` → `animate={{ opacity: 1, scale: 1 }}` → `exit={{ opacity: 0, scale: 0.95 }}`, duration 250ms, ease-out cubic
- The modal renders as a fixed overlay (`fixed inset-0 z-50 flex items-end justify-center` for bottom-sheet feel, or `items-center` for centered — follow DESIGN_SYSTEM.md modal pattern)
- Use `useCoyyns()` hook from T302 — call `spendCoyyns(cost, label, category)` on confirm
- Use `useNotifications()` hook from T402 — call the appropriate method to increment `bonus_sends_available`
- Define `EXTRA_SEND_COST = 25` as a module-level constant (configurable, not hardcoded inline)
- Quantity state: `const [quantity, setQuantity] = useState(1)`
- Max quantity derived: `const maxQuantity = Math.floor(balance / EXTRA_SEND_COST)`
- Affordable check: `const canAfford = balance >= EXTRA_SEND_COST`
- Cost derived: `const totalCost = quantity * EXTRA_SEND_COST`
- Button label: `Buy ${quantity} Extra Send${quantity === 1 ? '' : 's'}`
- Loading state during async operation — disable both quantity buttons and the buy button while pending
- Reset quantity to 1 (or 0 if unaffordable) each time the modal opens (`useEffect` on `open`)
- Format the balance with `toLocaleString()` for the comma-separated display
- Use `cn()` from `@/lib/utils` for all conditional class merging
- Toast: use the project's toast primitive (check DESIGN_SYSTEM.md for toast usage pattern)
- Error state: `const [error, setError] = useState<string | null>(null)` — clear on each new attempt

## Edge Cases

- **Balance exactly 25** → max quantity is 1, + button is disabled, flow works normally
- **Balance is 0 or < 25** → quantity defaults to 0, selector is read-only (both buttons disabled), buy button disabled, "Not enough CoYYns" message shown
- **User changes quantity to max, balance drops** — balance is read fresh from hook on each render, max recalculates
- **Double-tap on Buy** — disable button immediately on first tap (loading state) to prevent double spend
- **spendCoyyns fails** — show inline error, do NOT increment bonus_sends_available, do NOT close modal
- **bonus_sends update fails after spend** — this is a critical inconsistency; show error and log. Do not close modal. Consider rollback if useCoyyns supports it.
- **Modal closes mid-loading** — `onClose` should be blocked while `isLoading` is true (ignore backdrop tap and Cancel during pending state)
- **quantity × cost overflows display** — unlikely at typical balances, but `toLocaleString()` handles large numbers correctly

## Do NOT

- Don't hardcode `25` inline — always reference the `EXTRA_SEND_COST` constant
- Don't hardcode color hex values — use Tailwind design tokens from the theme
- Don't allow the quantity to go below 1 (when affordable) or above `maxQuantity`
- Don't close the modal on backdrop tap while a purchase is in progress
- Don't increment `bonus_sends_available` before `spendCoyyns` resolves successfully
- Don't use `alert()` or `confirm()` — use the project's toast and inline error patterns
- Don't add upsell copy or pressure language — the tone is warm and neutral
- Don't forget to reset quantity state when the modal opens with a fresh session
