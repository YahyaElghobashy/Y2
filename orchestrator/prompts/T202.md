## Why This Exists

Every piece of data in Hayah — love coupons, prayer streaks, health logs, shared grocery lists — lives in Supabase. Before any feature can read or write anything, the app needs a properly configured Supabase client. This is not a feature; it's the plumbing that everything else is built on top of.

The reason this task exists as its own unit is that Supabase has three distinct client contexts in a Next.js 15 App Router project: the browser (client components), the server (server components and route handlers), and middleware (session refresh on every request). Each one is initialized differently and must be set up correctly or auth will silently break in ways that are very hard to debug.

This task also establishes the database types file — the single source of truth for TypeScript types generated from the live Supabase schema. Every query in the codebase flows through these types.

---

## User Story

As a developer building features for Yahya and Yara, I want a single, consistent way to access Supabase from anywhere in the codebase — whether I'm in a server component, a client component, or middleware — so that I never have to think about client initialization again. I import the right helper, I get a fully-typed client, and I build the feature.

---

## Full UX Flow

This task has no direct UX — it is infrastructure. Its correctness is verified by downstream tasks:

1. **T203 (Login page)** calls the browser client to sign in → session is created and stored in cookies
2. **T204 (AuthProvider)** uses the browser client to subscribe to auth state changes → UI reacts to login/logout
3. **T205 (Middleware)** uses the middleware client on every request → expired tokens are silently refreshed before the page renders
4. **Any server component** uses the server client to fetch user-specific data → fully typed, secure, never exposes anon key usage to the client unnecessarily

If this task is implemented correctly, none of the downstream tasks will need to touch client initialization. They simply import and use.

## Implementation Notes

### 1. `src/lib/supabase/client.ts` — Browser Client

Use a singleton pattern so one client instance is shared across all client components. Use `createBrowserClient` from `@supabase/ssr`.

```ts
import { createBrowserClient } from '@supabase/ssr'
import type { Database } from '@/lib/types/database.types'

let client: ReturnType<typeof createBrowserClient<Database>> | undefined

export function getSupabaseBrowserClient() {
  if (client) return client
  client = createBrowserClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
  return client
}
```

- Lazy initialization — client is not created until first call
- Singleton — subsequent calls return the same instance
- Typed with `Database` generic throughout
- Export as a named function, not a default export

### 2. `src/lib/supabase/server.ts` — Server Client

Creates a fresh client per request. Must use `cookies()` from `next/headers` to read and write session cookies. Use `createServerClient` from `@supabase/ssr`.

```ts
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'
import type { Database } from '@/lib/types/database.types'

export async function getSupabaseServerClient() {
  const cookieStore = await cookies()
  return createServerClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll()
        },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            )
          } catch {
            // setAll called from a Server Component — cookies are read-only.
            // The middleware handles refreshing the session in this case.
          }
        },
      },
    }
  )
}
```

- `async` function because `cookies()` in Next.js 15 is async
- The `try/catch` in `setAll` is intentional — server components cannot write cookies, only middleware and route handlers can. The catch is silent by design.
- Never call this in a client component

### 3. `src/lib/supabase/middleware.ts` — Middleware Client

Used inside `src/middleware.ts`. Takes the incoming `request` and a mutable `response`, refreshes the session, and returns the updated response with the new cookies written.

```ts
import { createServerClient } from '@supabase/ssr'
import { NextResponse, type NextRequest } from 'next/server'
import type { Database } from '@/lib/types/database.types'

export async function updateSession(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request })

  const supabase = createServerClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll()
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) =>
            request.cookies.set(name, value)
          )
          supabaseResponse = NextResponse.next({ request })
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          )
        },
      },
    }
  )

  // IMPORTANT: Do not write any logic between createServerClient and
  // supabase.auth.getUser(). A simple mistake could make it very hard to
  // debug issues with users being randomly logged out.
  await supabase.auth.getUser()

  return supabaseResponse
}
```

- Exports a single `updateSession` function — T205 (middleware) imports and calls this
- The `getUser()` call is what triggers the token refresh — it must not be skipped
- Returns the full `NextResponse` with refreshed cookies set

## Edge Cases

- **Missing env vars at runtime** — The `!` non-null assertion will throw at runtime if the vars are undefined. This is intentional: a missing env var is a deployment error and should fail loudly, not silently produce broken behavior.
- **Server client called from a client component** — The `cookies()` import from `next/headers` will throw. The file separation enforces this, but add a comment in `server.ts` noting it is server-only.
- **Multiple calls to `getSupabaseBrowserClient()`** — The singleton guard ensures only one WebSocket connection is opened for realtime subscriptions.
- **Middleware running on static asset routes** — The `src/middleware.ts` (T205) must add a matcher to exclude `/_next/`, `/favicon.ico`, and other static paths. This file doesn't own that logic but must be compatible with it.
- **`cookies()` is async in Next.js 15** — `getSupabaseServerClient()` must be `async` and `await`ed at every call site. Do not forget the `await`.

## Do NOT

- Do not use `@supabase/auth-helpers-nextjs` — that package is deprecated. Use `@supabase/ssr` only.
- Do not create a default Supabase client at module scope in `server.ts` — it must be created fresh per request
- Do not put real Supabase credentials in `.env.local.example` or any committed file
- Do not import `server.ts` inside any `"use client"` component — it will throw at runtime
- Do not skip the `await supabase.auth.getUser()` call in the middleware client — without it, sessions will not be refreshed and users will be randomly logged out
- Do not add service-role key handling here — that belongs in a separate admin client if ever needed
- Do not use `process.env` assertions at module scope in a way that crashes during `next build` type-checking — the `!` assertion is fine
