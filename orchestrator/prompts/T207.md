## Why This Exists

The settings page (T113) was built with hardcoded placeholder data: "Yahya", "yahya@email.com", a static avatar. It worked as a visual shell. Now that AuthProvider (T204) is live and profile setup (T206) is complete, there is no reason for settings to be disconnected from reality. When Yara opens settings, she should see her name. When Yahya logs out, it should actually sign him out.

This task is the final wiring step of Phase 2 auth. It closes the loop: user data flows from Supabase through AuthProvider into the UI, and user actions (edit profile, sign out) flow back out to Supabase. After T207, the settings page is fully alive вҖ” no fake data, no dead buttons.

---

## User Story

As Yahya or Yara, when I open Settings I see my actual name, email, and avatar вҖ” not a placeholder. I can tap on my profile to edit my display name or change my avatar photo. When I tap Log Out, it asks me to confirm, then signs me out and sends me to the login screen. Any name change I save appears everywhere in the app immediately.

---

## Full UX Flow

1. **Navigate to /settings** вҶ’ profile card renders with real data from `useAuth()`:
   - Avatar circle shows uploaded image, or initials fallback if `avatar_url` is null
   - Display name shows `profile.display_name`
   - Email shows `profile.email` (read-only, always)

2. **Tap the "Profile" row** вҶ’ `ProfileEditForm` expands inline below the row (no modal):
   - Display name input pre-filled with current `profile.display_name`
   - Avatar upload circle shows current avatar with a camera icon overlay on tap
   - Save button and Cancel button appear at the bottom of the form

3. **Edit display name** вҶ’ user types new name вҶ’ tap Save:
   - Validates with Zod (non-empty, max 40 chars)
   - On success: `UPDATE profiles SET display_name = ? WHERE id = ?`
   - AuthProvider re-fetches profile so the new name propagates app-wide
   - Form collapses, profile card updates instantly

4. **Tap avatar upload circle** вҶ’ native file picker opens (input type="file" accept="image/*"):
   - Selected image uploads to Supabase Storage bucket `avatars/{user.id}`
   - On success: `UPDATE profiles SET avatar_url = ?` with the public URL
   - Avatar in the form and in the profile card update immediately

5. **Tap Cancel** вҶ’ form collapses with no changes, no API calls

6. **Tap "Log Out"** вҶ’ confirmation alert dialog appears:
   - Title: "Log Out?"
   - Body: "You'll need to sign in again to access Hayah."
   - Buttons: "Cancel" (secondary) | "Log Out" (destructive red)
   - Tap "Log Out" вҶ’ calls `signOut()` from `useAuth()` вҶ’ redirect to `/login`

## Implementation Notes

- `"use client"` on `src/app/settings/page.tsx` (already required for interactivity)
- `"use client"` on `ProfileEditForm.tsx`
- Call `useAuth()` at the top of the settings page to get `{ user, profile, signOut }`
- Guard: if `profile` is null (loading), show `LoadingSkeleton` (T104) for the profile card area
- Avatar initials fallback: take first letter of `display_name`, uppercase вҖ” no library needed
- Avatar upload: use Supabase Storage client directly in `ProfileEditForm` вҖ” bucket name `avatars`, path `{userId}/{timestamp}.jpg`
- After upload, update `avatar_url` in the `profiles` table, then call `onSave()`
- AuthProvider must re-fetch the profile after save вҖ” if `onSave` triggers a parent re-render that calls `refreshProfile()` or re-runs the auth query, the new name propagates automatically
- For the sign-out confirmation, use shadcn/ui `AlertDialog` вҖ” do not build a custom modal
- After `signOut()` resolves, use `router.push("/login")` from `next/navigation`
- React Hook Form schema for ProfileEditForm:
  ```ts
  const schema = z.object({
    display_name: z.string().min(1, "Name is required").max(40, "Max 40 characters"),
  })
  ```
- File input for avatar should be a hidden `<input type="file" accept="image/*">` triggered by clicking the avatar circle вҖ” do not use a third-party file upload library
- Image upload should show a local loading spinner inside the avatar circle while uploading
- Use Framer Motion `AnimatePresence` and `motion.div` with `initial={{ height: 0, opacity: 0 }}` and `animate={{ height: "auto", opacity: 1 }}` for the form expand/collapse

## Edge Cases

- **`profile` is null on mount** (auth still loading) вҶ’ show `LoadingSkeleton` for the profile card, do not attempt to render form
- **`avatar_url` is null** вҶ’ show initials fallback; never render a broken `<img>` tag
- **User uploads a non-image file** вҶ’ the `accept="image/*"` attribute prevents this in most browsers; add Zod/manual check as a second layer and show an inline error if it slips through
- **Upload to Supabase Storage fails** вҶ’ show an inline error message inside the form; do not close the form or update the profile row
- **Display name save fails** (network error or RLS rejection) вҶ’ show inline error below the input; keep the form open
- **User taps Log Out then immediately taps Cancel** вҶ’ `AlertDialog` closes, nothing happens, user stays in settings
- **`signOut()` itself throws** вҶ’ catch the error, show a toast or inline error, do not redirect вҖ” the user is still signed in
- **Display name is only whitespace** вҶ’ Zod `.trim().min(1)` should catch this; show "Name cannot be blank"

## Do NOT

- Do not replace the existing settings page layout from T113 вҖ” only update the data and add the edit form
- Do not build a custom modal for the log out confirmation вҖ” use shadcn/ui `AlertDialog`
- Do not store the avatar file in component state beyond the upload moment вҖ” upload it, get the URL, update the DB, done
- Do not allow the email field to be edited вҖ” email changes require a separate auth flow (not in scope)
- Do not call `signOut()` without confirmation вҖ” always show the `AlertDialog` first
- Do not hardcode any user names, emails, or avatar URLs вҖ” all data comes from `useAuth()`
- Do not install new packages for file upload or image cropping вҖ” keep it native `<input type="file">`
- Do not add partner profile editing here вҖ” settings only edits the current user's own profile
