## Why This Exists

Love coupons are one of the most emotionally loaded features in Hayah. A coupon is a promise — Yahya writing "One free breakfast in bed" and handing it to Yara is an act of vulnerability and care. The database must honor the weight of that. It must track not just the promise itself, but its entire journey: from the moment it was created, to the moment it was redeemed, to the moment it was honoured (or gently declined).

The surprise mechanic makes this schema non-trivial. A coupon marked as a surprise must be invisible to the recipient at the data layer — not just hidden in the UI, but genuinely unreachable via any authenticated query — until the creator decides to reveal it. This means the RLS policy for recipients must check both `is_surprise` and `surprise_revealed` together. Getting this wrong would spoil surprises at the database level.

The audit log (`coupon_history`) is equally important. When Yara opens a coupon and taps "Redeem," she deserves to know it was received and is being considered. When Yahya approves it three days later, that moment is recorded forever. These two tables together — `coupons` and `coupon_history` — are the complete record of every promise made and kept between two people.

---

## User Story

As Yahya or Yara, I can create a love coupon for my partner — a redeemable promise with a title, optional description, emoji, and category. My partner can see the coupon in their collection, tap to redeem it, and I receive a notification to approve or decline. The whole lifecycle is tracked. If I mark a coupon as a surprise, my partner cannot see it at all until I choose to reveal it. Every action — creation, redemption, approval, rejection, reveal — is recorded in an immutable history log.

---

## Full DB Flow

1. **Coupon created** — Yahya (creator) inserts a row into `coupons` with `status = 'active'`, `recipient_id = yara_id`
2. **Trigger fires** on `coupons INSERT` → auto-inserts a row into `coupon_history` with `action = 'created'`, `actor_id = creator_id`
3. **Yara sees the coupon** in her collection — IF `is_surprise = false` OR `surprise_revealed = true` (RLS enforces this)
4. **Surprise reveal** (optional path) — Yahya updates `surprise_revealed = true` → `coupon_history` row inserted with `action = 'revealed'` → Yara can now see it
5. **Yara redeems** — she updates `status = 'pending_approval'`, `redeemed_at = now()` → `coupon_history` row inserted with `action = 'redeemed'`
6. **Yahya approves** — he updates `status = 'redeemed'`, `approved_at = now()` → `coupon_history` row inserted with `action = 'approved'`
   - OR Yahya rejects — he updates `status = 'rejected'`, `rejected_at = now()`, optional `rejection_reason` → history row with `action = 'rejected'`
7. **Expiry** (background) — a Supabase cron job (future task) can mark expired coupons with `status = 'expired'`; `coupon_history` row inserted with `action = 'expired'`

## SQL Schema

**File to create:** `supabase/migrations/005_coupons.sql`

```sql
-- ============================================================
-- T601: Love Coupons Migration
-- Creates coupons + coupon_history tables, RLS policies,
-- triggers, and indexes.
-- Depends on: 001_auth_profiles.sql (profiles table)
-- ============================================================


-- ── 1. COUPONS TABLE ─────────────────────────────────────────

create table public.coupons (
  id                uuid        primary key default gen_random_uuid(),
  creator_id        uuid        not null references public.profiles (id) on delete restrict,
  recipient_id      uuid        not null references public.profiles (id) on delete restrict,
  title             text        not null,
  description       text,
  emoji             text,
  category          text        not null default 'general',
  image_url         text,
  status            text        not null default 'active',
  is_surprise       boolean     not null default false,
  surprise_revealed boolean     not null default false,
  redeemed_at       timestamptz,
  approved_at       timestamptz,
  rejected_at       timestamptz,
  rejection_reason  text,
  expires_at        timestamptz,
  created_at        timestamptz not null default now(),
  updated_at        timestamptz not null default now(),

  -- status must be one of the defined lifecycle values
  constraint coupons_status_check check (
    status in ('active', 'pending_approval', 'redeemed', 'rejected', 'expired')
  ),

  -- category must be one of the defined values
  constraint coupons_category_check check (
    category in ('romantic', 'practical', 'fun', 'food', 'general')
  ),

  -- creator and recipient must be different people
  constraint coupons_no_self_gift check (creator_id <> recipient_id),

  -- surprise_revealed can only be true if is_surprise is true
  constraint coupons_reveal_requires_surprise check (
    surprise_revealed = false or is_surprise = true
  )
);

-- Indexes for common query patterns
create index coupons_creator_id_idx    on public.coupons (creator_id);
create index coupons_recipient_id_idx  on public.coupons (recipient_id);
create index coupons_status_idx        on public.coupons (status);
create index coupons_expires_at_idx    on public.coupons (expires_at)
  where expires_at is not null;

comment on table public.coupons is
  'One row per love coupon. Tracks the full lifecycle from creation to redemption.';
comment on column public.coupons.is_surprise is
  'When true, the recipient cannot see this coupon until surprise_revealed is set to true.';
comment on column public.coupons.surprise_revealed is
  'Creator flips this to true to make a surprise coupon visible to the recipient.';
comment on column public.coupons.status is
  'Lifecycle state: active → pending_approval → redeemed | rejected. Also: expired.';


-- ── 2. COUPON_HISTORY TABLE ───────────────────────────────────

create table public.coupon_history (
  id          uuid        primary key default gen_random_uuid(),
  coupon_id   uuid        not null references public.coupons (id) on delete cascade,
  action      text        not null,
  actor_id    uuid        not null references public.profiles (id) on delete restrict,
  note        text,
  created_at  timestamptz not null default now(),

  constraint coupon_history_action_check check (
    action in ('created', 'redeemed', 'approved', 'rejected', 'revealed', 'expired')
  )
);

-- Indexes for fetching history by coupon
create index coupon_history_coupon_id_idx on public.coupon_history (coupon_id);
create index coupon_history_actor_id_idx  on public.coupon_history (actor_id);

comment on table public.coupon_history is
  'Immutable audit log of every state change on a coupon. Rows are never updated or deleted.';
comment on column public.coupon_history.action is
  'What happened: created | redeemed | approved | rejected | revealed | expired';
comment on column public.coupon_history.actor_id is
  'The profile ID of the user who performed this action.';


-- ── 3. UPDATED_AT TRIGGER FOR COUPONS ────────────────────────
-- Reuses set_updated_at() created in 001_auth_profiles.sql

create trigger coupons_set_updated_at
  before update on public.coupons
  for each row
  execute function public.set_updated_at();


-- ── 4. AUTO-LOG 'created' TO coupon_history ON INSERT ────────

create or replace function public.handle_coupon_created()
returns trigger
language plpgsql
security definer set search_path = public
as $$
begin
  insert into public.coupon_history (coupon_id, action, actor_id)
  values (new.id, 'created', new.creator_id);
  return new;
end;
$$;

create trigger on_coupon_created
  after insert on public.coupons
  for each row
  execute function public.handle_coupon_created();


-- ── 5. ROW LEVEL SECURITY: COUPONS ───────────────────────────
--
-- SURPRISE COUPON LOGIC (read carefully):
--
--   A coupon is readable by the recipient ONLY IF:
--     (a) is_surprise = false         (not a surprise — always visible)
--   OR
--     (b) is_surprise = true AND surprise_revealed = true
--         (was a surprise, but has now been revealed by the creator)
--
--   In other words: if is_surprise = true AND surprise_revealed = false,
--   the recipient CANNOT see the coupon. The NOT EXISTS check below
--   prevents the row from appearing in any recipient SELECT query.
--
--   The creator can always see all their own coupons regardless of
--   surprise status — they know what they created.

alter table public.coupons enable row level security;

-- Creator: full read access to all coupons they created
create policy "coupons: creator read"
  on public.coupons
  for select
  using (auth.uid() = creator_id);

-- Recipient: can read coupons where they are the recipient,
-- EXCEPT surprise coupons that have not yet been revealed.
create policy "coupons: recipient read (non-surprise)"
  on public.coupons
  for select
  using (
    auth.uid() = recipient_id
    and (
      is_surprise = false
      or (is_surprise = true and surprise_revealed = true)
    )
  );

-- Creator: can INSERT a new coupon (must set themselves as creator)
create policy "coupons: creator insert"
  on public.coupons
  for insert
  with check (auth.uid() = creator_id);

-- Creator: can UPDATE a coupon they own.
-- This covers: approve (status → 'redeemed'), reject (status → 'rejected'),
-- reveal (surprise_revealed → true), set rejection_reason, set approved_at, etc.
create policy "coupons: creator update"
  on public.coupons
  for update
  using (auth.uid() = creator_id)
  with check (auth.uid() = creator_id);

-- Recipient: can UPDATE a coupon to redeem it.
-- Specifically: set status = 'pending_approval' and redeemed_at = now().
-- The check constrains that recipient cannot change creator_id, recipient_id,
-- title, or any creator-only fields — enforced at app layer.
create policy "coupons: recipient update (redeem)"
  on public.coupons
  for update
  using (
    auth.uid() = recipient_id
    and (
      is_surprise = false
      or (is_surprise = true and surprise_revealed = true)
    )
  )
  with check (auth.uid() = recipient_id);

-- No DELETE policy — coupons are permanent records


-- ── 6. ROW LEVEL SECURITY: COUPON_HISTORY ────────────────────

alter table public.coupon_history enable row level security;

-- Both creator and recipient of a coupon can read its history.
-- We join back to coupons to determine membership.
create policy "coupon_history: creator or recipient read"
  on public.coupon_history
  for select
  using (
    exists (
      select 1
      from public.coupons c
      where c.id = coupon_history.coupon_id
        and (c.creator_id = auth.uid() or c.recipient_id = auth.uid())
    )
  );

-- Users can only insert history rows where they are the actor.
-- The auto-trigger for 'created' runs as security definer, bypassing this.
-- Client-side history inserts (e.g., app logging 'redeemed') must have actor_id = auth.uid().
create policy "coupon_history: actor insert"
  on public.coupon_history
  for insert
  with check (
    auth.uid() = actor_id
    and exists (
      select 1
      from public.coupons c
      where c.id = coupon_history.coupon_id
        and (c.creator_id = auth.uid() or c.recipient_id = auth.uid())
    )
  );

-- No UPDATE or DELETE on coupon_history — the log is append-only
```

## Implementation Notes

- **File location:** `supabase/migrations/005_coupons.sql` — must run after `001_auth_profiles.sql`. Supabase CLI runs migrations in filename order.
- **`on delete restrict` on creator_id and recipient_id** — prevents a profile from being deleted while they have coupons. This is intentional: profiles in this app are permanent (see T201), so this FK constraint is a safety net, not a practical concern.
- **`on delete cascade` on coupon_history.coupon_id** — if a coupon is ever deleted (e.g., admin cleanup), its history goes with it. Orphaned history rows must never exist.
- **`security definer` on `handle_coupon_created()`** — the trigger must run as the function owner (service role) to bypass RLS when inserting into `coupon_history`. Without this, the trigger would need the inserting user to have INSERT permission on `coupon_history`, which conflicts with the intent to make the 'created' entry automatic and tamper-proof.
- **`set_updated_at()` is reused** from migration 001. It must already exist before migration 005 runs. Do not redefine it here — use `create or replace function` only for `handle_coupon_created()`.
- **The surprise RLS uses `or (is_surprise = true and surprise_revealed = true)`** — do NOT simplify this to `surprise_revealed = true`. A non-surprise coupon where `surprise_revealed` is still false (its default) would become invisible if you relied only on `surprise_revealed = true`.
- **`image_url` stores a Supabase Storage URL** — the Storage bucket (`coupon-images`) and upload flow are handled in a future task. This column simply stores the final URL string; no FK constraint.
- **App-layer responsibility for history inserts on state changes** — the trigger handles only 'created'. For 'redeemed', 'approved', 'rejected', 'revealed', the app (T602 hook) must explicitly insert into `coupon_history` as part of the same logical operation. This keeps the trigger simple and avoids complex trigger chains.
- **Expiry enforcement** — the `expires_at` column exists but no trigger marks coupons as expired automatically. A Supabase cron job (future task) will run periodically, update status and insert 'expired' history rows. Do not add this logic here.
- **The `coupons_no_self_gift` constraint** enforces that `creator_id <> recipient_id`. With exactly two users this should never trigger, but it is correct defensive design.

## Edge Cases

- **Recipient queries coupons before surprise is revealed:** The `recipient read (non-surprise)` policy filters them out entirely — the row does not appear in any `select * from coupons` query by the recipient. The count of visible coupons changes the moment `surprise_revealed` is flipped to `true`. This is correct behavior.
- **Creator tries to redeem their own coupon:** The `recipient update (redeem)` policy uses `auth.uid() = recipient_id`. If `creator_id = auth.uid()`, this policy does not apply. Since `coupons_no_self_gift` prevents self-gifting, a creator can never be the recipient — this case cannot arise.
- **Recipient tries to approve/reject:** The `creator update` policy uses `auth.uid() = creator_id`. A recipient is never the creator (per constraint). Only the creator can approve or reject.
- **Double redemption attempt:** If a coupon is already in `'pending_approval'` and the recipient triggers redeem again, the app layer should guard against this. The DB does not have a constraint preventing a second `status` update to `'pending_approval'`, so the hook (T602) must check current status before updating.
- **Coupon history visibility for surprise coupons:** The `coupon_history: creator or recipient read` policy does not filter by `is_surprise`. This means once a surprise coupon is revealed, the recipient can see the full history including the original 'created' entry — and learn when it was made. This is intentional: once revealed, the full story is shared.
- **Expired coupon redemption attempt:** The `status` field will be `'expired'`. The app layer (T602) must reject redemption attempts on expired coupons. No DB-level block exists beyond the `coupons_status_check` constraint (which does allow updating to any valid status from any other valid status).
- **`rejection_reason` on approval:** If a creator approves after setting `rejection_reason` (e.g., changed their mind), the app should clear `rejection_reason` on approval. This is an app-layer concern — the DB allows any text in that column at any status.
- **Migration 005 run before 001:** Will fail with `relation "public.profiles" does not exist`. Always run migrations in order via `supabase db push` which handles ordering automatically.

- **T201 (Auth DB Migration)** — `coupons.creator_id`, `coupons.recipient_id`, and `coupon_history.actor_id` all reference `public.profiles.id`. This migration cannot run without T201.

- **T603 (CouponCard component)** — renders a single coupon. Reads `title`, `emoji`, `category`, `status`, `is_surprise`, `surprise_revealed` from the `coupons` shape returned by T602.
- **T604 (CouponsPage)** — the full coupons screen. Uses T602 hook and T603 card. Displays active coupons for the current user as recipient, and outgoing coupons as creator.
- **Supabase Storage (future task)** — a `coupon-images` bucket will be created for `image_url` uploads. The URL stored in `coupons.image_url` will be a public or signed Storage URL.
- **Push notifications (future task)** — redemption and approval events from `coupon_history` will trigger Edge Function notifications to the other user.

## Do NOT

- Do not add a DELETE policy for coupons — coupons are permanent records of promises made
- Do not add a DELETE policy for coupon_history — the audit log is append-only and must never be erased
- Do not simplify the surprise RLS check to `surprise_revealed = true` alone — a non-surprise coupon defaults to `surprise_revealed = false` and would become invisible to the recipient
- Do not add an UPDATE policy for coupon_history — once logged, history entries are immutable
- Do not store binary image data in the `image_url` column — only Supabase Storage URLs
- Do not redefine `set_updated_at()` in this migration — it already exists from 001_auth_profiles.sql
- Do not add a trigger for every status transition — only the 'created' action is auto-logged; all other history entries are written explicitly by the app hook (T602)
- Do not allow the anon role any access to either table — these are authenticated-only tables
- Do not run `supabase db reset` in production — it wipes all coupon data permanently
- Do not add a `deleted_at` soft-delete column — use `status = 'expired'` or leave as-is; soft delete adds complexity with no benefit for a 2-user app
