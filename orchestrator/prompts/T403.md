## Why This Exists

Hayah is a private, intimate app for two people. When Yahya sends Yara a "thinking of you" nudge, or when the system fires a daily check-in reminder, the notification has to actually arrive — on her lock screen, on her wrist, wherever she is. A notification saved to a database row that never reaches her phone is not a notification. It's a whisper that no one heard.

This Edge Function is the delivery engine. It sits between the database (where notifications are stored) and the Web Push API (which talks to APNS / FCM on the device). It runs in Supabase's Deno runtime, meaning it lives close to the database, authenticates via JWT, and executes instantly when called. It is the last hop before a notification lights up Yara's screen.

---

## User Story

As the system (or Yahya), when a notification is created and this function is called, every device that Yara has registered for push should receive the notification within seconds. If a device subscription is stale or expired, it is silently pruned so it doesn't waste future delivery attempts. If delivery fails on every device, the notification row is marked as failed so the calling system knows and can decide how to handle it.

## Full Request / Response Flow

1. **Caller sends POST** to `/functions/v1/send-notification` with:
   ```json
   {
     "notification_id": "uuid",
     "recipient_id": "uuid"
   }
   ```
2. **JWT validation** — Supabase automatically verifies the Authorization header. If invalid or missing → 401 immediately.
3. **Sender verification** — Query `notifications` table: confirm the row exists AND `sender_id` matches the calling user's sub from the JWT. If not found → 404. If sender mismatch → 403.
4. **Fetch notification payload** — Get `title`, `body`, `emoji`, `type`, `status` from the `notifications` row. If `status` is already `delivered` → return 200 early (idempotent, no double-send).
5. **Fetch push subscriptions** — Select all rows from `push_subscriptions` where `user_id = recipient_id`. If none → update notification status to `'failed'`, return 200 with `{ warning: "no_subscriptions" }`.
6. **Send Web Push to each subscription** — For every subscription row, call `webpush.sendNotification()` with the VAPID-signed payload.
7. **Handle per-subscription results:**
   - Success → mark that subscription as delivered
   - HTTP 410 (Gone) → the device has unsubscribed; delete the `push_subscriptions` row silently
   - Other error → log, continue to next subscription (don't abort the whole batch)
8. **Update notification status** — If at least one delivery succeeded → `'delivered'`. If all failed → `'failed'`. Write back to `notifications` row.
9. **Return response** — `{ success: true, delivered: N, failed: M }` or `{ success: false, error: "..." }`

## Environment Variables (Supabase Secrets)

Set via `supabase secrets set`:

| Secret name | Description |
|---|---|
| `VAPID_PUBLIC_KEY` | VAPID public key (base64url, from `web-push generate-vapid-keys`) |
| `VAPID_PRIVATE_KEY` | VAPID private key (base64url) |
| `VAPID_SUBJECT` | Contact URI, e.g. `mailto:yahya@example.com` |
| `SUPABASE_URL` | Auto-injected by Supabase runtime |
| `SUPABASE_SERVICE_ROLE_KEY` | Auto-injected by Supabase runtime (use this for DB writes) |

Configure VAPID at the top of the function, before `serve()`:
```typescript
webpush.setVapidDetails(
  Deno.env.get("VAPID_SUBJECT")!,
  Deno.env.get("VAPID_PUBLIC_KEY")!,
  Deno.env.get("VAPID_PRIVATE_KEY")!,
);
```

## Push Payload Shape

The Web Push payload is a JSON string sent in the notification body. The service worker on the client will parse this:

```typescript
const payload = JSON.stringify({
  title: `${notification.emoji ?? ""} ${notification.title}`.trim(),
  body: notification.body,
  icon: "/icons/icon-192x192.png",
  badge: "/icons/badge-72x72.png",
  data: {
    notification_id: notification.id,
    type: notification.type,
    url: "/",
  },
});
```

## Database Tables Used

All tables defined in T401. Use the service role client for writes (status updates, subscription deletion):

```typescript
const supabase = createClient(
  Deno.env.get("SUPABASE_URL")!,
  Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!,
);
```

**`notifications`** — read `title`, `body`, `emoji`, `type`, `status`; write back `status` and `delivered_at`

**`push_subscriptions`** — read all rows for `recipient_id`; delete rows that return 410

**`daily_send_limits`** — this function does NOT check limits itself. Limits are enforced by the caller (T402) before this function is invoked. This function trusts that if it was called, the limit was already validated.

## Error Handling Matrix

| Condition | HTTP Status | Notification status | Notes |
|---|---|---|---|
| Missing/invalid JWT | 401 | unchanged | Supabase auth middleware handles this |
| Missing request body fields | 400 | unchanged | Validate before any DB call |
| Notification row not found | 404 | unchanged | Return clear error message |
| Caller is not the sender | 403 | unchanged | Prevent spoofed sends |
| Notification already delivered | 200 | unchanged | Idempotent — don't resend |
| No push subscriptions for recipient | 200 | `'failed'` | Warn but don't error — user may have push disabled |
| All subscriptions return 410 | 200 | `'failed'` | Subscriptions cleaned up, status written |
| At least one delivery succeeded | 200 | `'delivered'` | Partial success counts as delivered |
| Unhandled Deno / network error | 500 | `'failed'` | Wrap entire handler in try/catch |

## Edge Cases

- **Notification already `delivered`** — check status before sending. Return 200 immediately. Never double-deliver to avoid duplicate lock screen notifications.
- **Subscription endpoint is malformed** — `webpush.sendNotification()` will throw synchronously before the HTTP call. Catch per-subscription, mark as failed, continue.
- **Recipient has 3 devices registered** — send to all 3 in parallel using `Promise.allSettled()`. Never `Promise.all()` — a single 410 should not abort the other two.
- **VAPID keys not set** — `Deno.env.get()` returns undefined, `webpush.setVapidDetails()` will throw. Let it throw at startup so misconfiguration is visible immediately in function logs, not silently swallowed.
- **Very large push payload** — Web Push has a 4KB limit on the payload body. The JSON payload above is well within limits, but if `body` is user-provided text, truncate to 200 chars before serializing.
- **Deno module cold start** — The first invocation after idle will be slightly slower. This is normal and does not affect correctness.

## What This Connects To

- **T401** defines the DB schema this function reads from and writes to. T401 must be deployed first.
- **T402** (Push Subscription Hook) is the client-side hook that registers devices. Its subscriptions live in `push_subscriptions` and are what this function iterates over.
- **T405** (Notification Builder UI) calls this function indirectly — the user composes a notification, the client inserts a row into `notifications`, then calls this function with the resulting `notification_id`.
- **Service Worker** on the client receives the Web Push message and uses `data.url` from the payload to route the user when they tap the notification.

## Test Requirements

File: `supabase/functions/send-notification/index.test.ts`

Use Deno's built-in test runner (`Deno.test`) with a mocked `webpush` module and a mocked Supabase client:

- Returns 401 when Authorization header is absent
- Returns 400 when request body is missing `notification_id` or `recipient_id`
- Returns 404 when notification row does not exist in DB
- Returns 403 when `sender_id` on notification does not match JWT sub
- Returns 200 with `{ warning: "no_subscriptions" }` when recipient has no push_subscription rows
- Calls `webpush.sendNotification()` once per subscription row
- Uses `Promise.allSettled()` — a single 410 response does not prevent other subscriptions from being attempted
- Deletes a push_subscription row when the Web Push response status is 410
- Sets notification `status` to `'delivered'` when at least one subscription succeeds
- Sets notification `status` to `'failed'` when all subscriptions fail
- Returns 200 and skips sending when notification `status` is already `'delivered'`

## Do NOT

- Don't use `fetch()` to call the Web Push endpoints manually — use the `web-push` npm package via `npm:web-push` specifier
- Don't use `Promise.all()` for the subscription batch — use `Promise.allSettled()` so one failure doesn't abort others
- Don't use the `anon` key for DB operations in this function — use `SUPABASE_SERVICE_ROLE_KEY` so RLS doesn't block writes
- Don't check daily send limits inside this function — that is T402's responsibility; this function is a delivery primitive
- Don't hardcode the VAPID keys — they must come from `Deno.env.get()`
- Don't swallow errors silently — always write the final `status` back to the notification row so callers have a reliable signal
- Don't return 5xx for partial failures — if any device was reached, return 200 with delivered/failed counts
