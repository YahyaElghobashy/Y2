## Why This Exists

CoYYns are the emotional currency of this relationship. Earning them is meaningful. Spending them should feel intentional — not frictionless, not mechanical. When Yahya opens his wallet and decides to spend 500 CoYYns on something for Yara, the form should feel like a small ritual. A moment of choice. The balance sits at the top as a gentle constraint: here's what you have, make it count.

This form is also the single entry point for CoYYns spending across the entire app. The marketplace (Phase 7) will prefill it with a product amount and description. Manual redemptions use it too. One form, one pattern, one consistent emotional experience — regardless of what triggered the spend.

---

## User Story

As Yahya or Yara, when I tap "Spend CoYYns" from my wallet, a sheet slides up from the bottom. I see my current balance prominently displayed in warm copper so I know exactly what I have. I type the amount I want to spend, and if it exceeds my balance the form immediately tells me — no surprises after tapping the button. I add a description of what I'm spending on. I tap Spend. The CoYYns leave my wallet and I see a confirmation. The whole interaction feels deliberate, warm, and real.

---

## Full UX Flow

1. **Sheet opens** — slides up from the bottom of the screen with a spring animation
2. **Balance is shown** at the top of the form: `Balance: 1,250` in `accent-primary` copper
3. **Amount input** — large centered number input, numeric keyboard on mobile, autofocused on open
4. **As the user types an amount:**
   - If amount is within balance: input border is normal, button is enabled
   - If amount exceeds balance: red warning text appears below the input ("Insufficient CoYYns"), input border shifts to `error` color (#C27070), Spend button becomes disabled
   - If amount is 0 or empty: button remains disabled
5. **Description textarea** — below the amount, placeholder "What are you spending on?"
6. **Spend button** — copper (`accent-primary`), full-width, disabled state when amount is invalid
7. **Cancel** — either an X icon in the top-right corner, or tapping the backdrop dismisses the sheet
8. **On submit:**
   - Button shows a loading spinner (Framer Motion opacity pulse)
   - Calls `spendCoyyns({ amount, description, category })` from the `useCoyyns` hook
   - On success: sheet closes, a toast confirms the spend, `onSuccess?.()` is called
   - On error: inline error message appears below the Spend button, sheet stays open

## Form Schema (Zod)

```ts
// Schema is defined inside the component so it can close over currentBalance
const spendSchema = (currentBalance: number) =>
  z.object({
    amount: z
      .number({ invalid_type_error: 'Enter a valid amount' })
      .int('Amount must be a whole number')
      .min(1, 'Amount must be at least 1')
      .max(currentBalance, 'Insufficient CoYYns'),
    description: z
      .string()
      .min(1, 'Description is required')
      .max(200, 'Description must be 200 characters or less'),
  })

type SpendFormValues = z.infer<ReturnType<typeof spendSchema>>
```

## Implementation Notes

- `"use client"` directive required (React Hook Form, Framer Motion, useCoyyns hook)
- Use `useCoyyns()` from `@/hooks/use-coyyns` (T302) to get `balance` and `spendCoyyns`
- Wrap the sheet with Framer Motion `AnimatePresence` — animate `y` from `100%` to `0` on open, reverse on close; use `spring` with `stiffness: 300, damping: 30`
- The backdrop fades in/out with opacity 0 → 0.4 using `motion.div`
- Use `useForm` with `resolver: zodResolver(spendSchema(balance))` — recreate schema when balance changes
- Amount input: use `type="number"` with `inputMode="numeric"` for mobile numeric keyboard, suppress browser spin buttons via CSS (`[&::-webkit-inner-spin-button]:appearance-none`)
- Watch the `amount` field with `watch('amount')` to drive the real-time insufficient balance warning — do NOT rely solely on Zod errors for this; derive `isInsufficient = watchedAmount > balance` directly
- `prefilledAmount` and `prefilledDescription` should be set via `reset()` inside a `useEffect` that fires when `open` becomes `true`
- Category defaults to `'manual'` if `prefilledCategory` is not provided
- On successful submit, call `onClose()` followed by `onSuccess?.()` — do not call `onSuccess` before closing
- Use `cn()` from `@/lib/utils` throughout for conditional Tailwind classes
- Use `toast` from shadcn/ui `sonner` for the success notification: "Spent {amount} CoYYns"
- Prevent body scroll while sheet is open: add `overflow-hidden` to `document.body` on mount, restore on unmount

## Edge Cases

- **Balance is 0** — amount input should still be shown, but any value ≥ 1 immediately triggers the insufficient warning. The Spend button is effectively always disabled.
- **prefilledAmount exceeds balance** — the form opens with the warning already showing. The Spend button is disabled immediately. The user can still edit the amount down.
- **User clears the amount field** — treat as empty/invalid, not 0. Button stays disabled, no insufficient warning (the field is simply empty, not wrong).
- **Rapid open/close** — reset form state on every open event so stale values from a previous session do not persist.
- **Network error on spendCoyyns** — catch the error, display it inline below the Spend button, do not close the sheet, re-enable the button.
- **Very long description** — textarea max 200 chars; show character count helper text (`0/200`) that updates live below the textarea.

## What This Connects To

- **T302 (useCoyyns hook)** — provides `balance` and `spendCoyyns({ amount, description, category })`. This form does not manage CoYYns state directly.

## Do NOT

- Do not hardcode `#C4956A` or any hex color — use Tailwind design tokens (`text-accent-primary`, `bg-accent-primary`, etc.)
- Do not call `onSuccess` before `onClose` — close first, then trigger downstream effects
- Do not use `router.push()` or navigate away — this form is purely a modal overlay
- Do not manage CoYYns balance state inside this component — that belongs entirely to `useCoyyns`
- Do not use a native `<dialog>` element — use Framer Motion + a portal for the sheet
- Do not show the sheet as a full-page route — it is always a floating overlay on top of the current screen
- Do not skip the backdrop — the overlay is part of the spatial model and makes the sheet feel grounded
