## Why This Exists

Hayah is a shared life ecosystem for two people. Most of it is reactive — you log something, you see something. But there is one feature that is entirely spontaneous and personal: the push notification. Yahya can send Yara a surprise message — a little tap on the shoulder from across the city. Yara can send Yahya a heart emoji that lights up his lock screen when he is deep in a meeting.

This is not a chat feature. It is not a feed. It is a gesture — deliberate, limited, and meaningful precisely because it is rationed. Each user gets two free sends per day. After that, extra sends cost CoYYns. This scarcity makes each notification feel like a choice, not spam.

But push notifications are stateless by nature. The browser's Web Push API fires and forgets — there is no built-in delivery log, no send history, no rate limiting, no subscription management. This migration builds the database layer that makes all of that possible: subscription storage, a full notification log, and a daily limit tracker that enforces the scarcity model without touching the Edge Function on every send.

---

## User Story

As Yahya, I can send Yara a surprise push notification — a title, a short message, and an optional emoji. I get two free sends every day. If I want to send more, I can spend CoYYns from my wallet to unlock additional sends. I can see how many sends I have left today. When I send a notification, it shows up on Yara's device within seconds, even if the app is closed. My notification history is saved so we can look back at the little moments we sent each other.

As Yara, I receive Yahya's notifications on my device even when the Hayah app is not open. I can manage which of my devices receive notifications. I also have the same two free sends per day to surprise Yahya back.

## Full DB Flow

### Subscribing a Device
1. User opens Hayah in a browser (or installed PWA) for the first time
2. The app requests notification permission via the browser API
3. On grant, the browser generates a `PushSubscription` object (endpoint + keys)
4. The app calls Supabase to upsert a row in `push_subscriptions` with the serialised object
5. The `UNIQUE(user_id, subscription)` constraint prevents duplicate rows if the same device subscribes again
6. The subscription row is now the delivery target for all future notifications to that user

### Sending a Notification
1. Sender taps the notification composer in the app
2. The app calls `get_or_create_daily_limit(sender_id)` to retrieve today's limit row
3. If `free_sends_used < 2`, the send is free — proceed
4. If `free_sends_used >= 2` and `bonus_sends_available > 0`, decrement bonus and proceed
5. If both are exhausted, the app shows a "spend CoYYns to send more" prompt
6. On approval, the app inserts a `coyyns_transactions` row (`type = 'spend'`, `category = 'notification_purchase'`) and increments `bonus_sends_available` in `daily_send_limits`
7. The app inserts a row into `notifications` with `status = 'sent'`
8. The app calls the Edge Function (T403) which reads `push_subscriptions` for the recipient and fires the Web Push API for each active subscription
9. On successful push delivery, the Edge Function updates the `notifications` row to `status = 'delivered'`; on failure it sets `status = 'failed'`
10. `daily_send_limits.free_sends_used` (or `bonus_sends_used`) is incremented atomically in the same operation as step 7

### Receiving a Notification
1. Recipient's device receives the push payload from the browser's push service
2. The service worker (Serwist, T402) intercepts the push event and calls `showNotification()`
3. The notification appears on the lock screen with the `emoji + title` and `body`
4. No database write happens on receipt — the log row was already created by the sender at send time

```sql
-- ============================================================
-- T401: Push Notifications Database Migration
-- Creates push_subscriptions, notifications, daily_send_limits,
-- RLS policies, and the get_or_create_daily_limit function
-- Depends on: 001_auth_profiles.sql (profiles table must exist)
-- ============================================================


-- ── 1. PUSH SUBSCRIPTIONS TABLE ───────────────────────────────

create table public.push_subscriptions (
  id           uuid        primary key default gen_random_uuid(),
  user_id      uuid        not null references public.profiles (id) on delete cascade,
  subscription jsonb       not null,
  device_name  text,
  created_at   timestamptz not null default now(),
  updated_at   timestamptz not null default now(),
  constraint push_subscriptions_user_device_unique unique (user_id, subscription)
);

-- Index on user_id for fast subscription lookups when sending to a recipient
create index push_subscriptions_user_id_idx on public.push_subscriptions (user_id);

comment on table public.push_subscriptions is
  'One row per registered device per user. Stores the Web Push PushSubscription object (endpoint + keys). Used by the Edge Function to deliver notifications to all active devices for a recipient.';
comment on column public.push_subscriptions.subscription is
  'The serialised PushSubscription JSON object from the browser. Contains endpoint, expirationTime, and keys (p256dh, auth). Never expose to the client of other users.';
comment on column public.push_subscriptions.device_name is
  'Optional human-readable label set by the user. E.g. "Yahya iPhone", "Work MacBook". Useful for managing multiple subscriptions.';


-- ── 2. UPDATED_AT TRIGGER FOR PUSH_SUBSCRIPTIONS ─────────────

create trigger push_subscriptions_set_updated_at
  before update on public.push_subscriptions
  for each row
  execute function public.set_updated_at();
-- Note: set_updated_at() was defined in 001_auth_profiles.sql and is reused here.


-- ── 3. NOTIFICATIONS TABLE ────────────────────────────────────

create table public.notifications (
  id           uuid        primary key default gen_random_uuid(),
  sender_id    uuid        not null references public.profiles (id) on delete cascade,
  recipient_id uuid        not null references public.profiles (id) on delete cascade,
  title        text        not null,
  body         text        not null,
  emoji        text,
  type         text        not null default 'custom'
                           check (type in ('custom', 'coupon_approval', 'system')),
  status       text        not null default 'sent'
                           check (status in ('sent', 'delivered', 'failed')),
  metadata     jsonb       not null default '{}',
  created_at   timestamptz not null default now()
);

-- Index for recipient's inbox view (newest first)
create index notifications_recipient_id_created_at_idx
  on public.notifications (recipient_id, created_at desc);

-- Index for sender's outbox view (newest first)
create index notifications_sender_id_created_at_idx
  on public.notifications (sender_id, created_at desc);

comment on table public.notifications is
  'Immutable log of every push notification sent between Yahya and Yara. Status is updated by the Edge Function after delivery attempt. Rows are never edited or deleted by the client.';
comment on column public.notifications.emoji is
  'Optional single emoji prepended to the notification title on the device lock screen. E.g. "🌙", "❤️". Stored separately so the UI can render it distinctly.';
comment on column public.notifications.type is
  'Categorises the source of the notification. custom = user-composed message. coupon_approval = triggered by the coupons feature. system = app-generated alerts.';
comment on column public.notifications.status is
  'Lifecycle state set by the Edge Function. sent = row created, push not yet attempted. delivered = Web Push API accepted the payload. failed = push API returned an error.';
comment on column public.notifications.metadata is
  'Flexible JSON bag for feature-specific context. E.g. { "coupon_id": "..." } for coupon_approval type. Always safe to be empty.';


-- ── 4. DAILY SEND LIMITS TABLE ────────────────────────────────

create table public.daily_send_limits (
  id                     uuid    primary key default gen_random_uuid(),
  user_id                uuid    not null references public.profiles (id) on delete cascade,
  date                   date    not null default current_date,
  free_sends_used        integer not null default 0 check (free_sends_used >= 0),
  bonus_sends_used       integer not null default 0 check (bonus_sends_used >= 0),
  bonus_sends_available  integer not null default 0 check (bonus_sends_available >= 0),
  constraint daily_send_limits_user_date_unique unique (user_id, date)
);

-- Index on user_id for history queries
create index daily_send_limits_user_id_idx on public.daily_send_limits (user_id);

comment on table public.daily_send_limits is
  'One row per user per calendar day. Tracks how many free and bonus notification sends have been used. Rows are lazily created on first send of the day via get_or_create_daily_limit().';
comment on column public.daily_send_limits.free_sends_used is
  'Number of free sends used today. Free allowance is 2 per day. When this reaches 2, only bonus sends are available.';
comment on column public.daily_send_limits.bonus_sends_used is
  'Number of purchased bonus sends used today. Informational — does not gate anything directly.';
comment on column public.daily_send_limits.bonus_sends_available is
  'Bonus sends purchased with CoYYns that have not yet been used today. Decremented by the Edge Function (T403) when a bonus send is consumed. Incremented when CoYYns are spent to purchase more.';


-- ── 5. ROW LEVEL SECURITY — PUSH_SUBSCRIPTIONS ───────────────

alter table public.push_subscriptions enable row level security;

-- Policy: users can read their own subscriptions only
create policy "push_subscriptions: own read"
  on public.push_subscriptions
  for select
  using (auth.uid() = user_id);

-- Policy: users can insert subscriptions for themselves only
create policy "push_subscriptions: own insert"
  on public.push_subscriptions
  for insert
  with check (auth.uid() = user_id);

-- Policy: users can update their own subscriptions (e.g. rename device)
create policy "push_subscriptions: own update"
  on public.push_subscriptions
  for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

-- Policy: users can delete their own subscriptions (unregister device)
create policy "push_subscriptions: own delete"
  on public.push_subscriptions
  for delete
  using (auth.uid() = user_id);

-- Partners CANNOT read each other's subscription objects. The Edge Function
-- reads recipient subscriptions as service role (security definer), not as the
-- authenticated sender. This protects raw endpoint URLs and encryption keys.


-- ── 6. ROW LEVEL SECURITY — NOTIFICATIONS ────────────────────

alter table public.notifications enable row level security;

-- Policy: sender can insert notifications (only for themselves as sender)
create policy "notifications: own insert"
  on public.notifications
  for insert
  with check (auth.uid() = sender_id);

-- Policy: sender can read notifications they sent
create policy "notifications: sender read"
  on public.notifications
  for select
  using (auth.uid() = sender_id);

-- Policy: recipient can read notifications sent to them
create policy "notifications: recipient read"
  on public.notifications
  for select
  using (auth.uid() = recipient_id);

-- No UPDATE policy from client — status updates are performed by the Edge
-- Function (T403) running as service role. Clients never change status.
-- No DELETE policy — the notification log is permanent.


-- ── 7. ROW LEVEL SECURITY — DAILY SEND LIMITS ────────────────

alter table public.daily_send_limits enable row level security;

-- Policy: users can read their own daily limit row
create policy "daily_send_limits: own read"
  on public.daily_send_limits
  for select
  using (auth.uid() = user_id);

-- Policy: users can update their own daily limit row
-- (e.g. incrementing free_sends_used after a successful send)
create policy "daily_send_limits: own update"
  on public.daily_send_limits
  for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

-- No INSERT policy from client — rows are created exclusively via the
-- get_or_create_daily_limit() function which runs as security definer.
-- No DELETE policy — limit rows are historical records.


-- ── 8. FUNCTION: get_or_create_daily_limit ────────────────────
-- Returns the daily_send_limits row for p_user_id and today's date.
-- If no row exists yet (first send of the day), inserts one with default
-- values and returns it. Safe to call multiple times — idempotent.
-- Runs as security definer so it can INSERT despite no client INSERT policy.

create or replace function public.get_or_create_daily_limit(p_user_id uuid)
returns public.daily_send_limits
language plpgsql
security definer set search_path = public
as $$
declare
  v_row public.daily_send_limits;
begin
  -- Attempt to fetch today's row first (the common path)
  select * into v_row
  from public.daily_send_limits
  where user_id = p_user_id
    and date = current_date;

  -- If no row exists, create one with all defaults and return it
  if not found then
    insert into public.daily_send_limits (user_id, date)
    values (p_user_id, current_date)
    on conflict (user_id, date) do nothing;

    -- Re-select after insert (handles the race condition where two concurrent
    -- calls both miss the initial select and both attempt insert — only one
    -- succeeds, but both return the same row via this re-select)
    select * into v_row
    from public.daily_send_limits
    where user_id = p_user_id
      and date = current_date;
  end if;

  return v_row;
end;
$$;

comment on function public.get_or_create_daily_limit(uuid) is
  'Returns today''s daily_send_limits row for the given user, creating it with default values if it does not yet exist. Idempotent and safe under concurrent calls. Called by T403 (Edge Function) before every send attempt.';
```

## Implementation Notes

- **File location:** `supabase/migrations/003_notifications.sql` — must run after `001_auth_profiles.sql`. The `user_id`, `sender_id`, and `recipient_id` columns on all three tables reference `public.profiles.id`. Migration ordering is enforced by the numeric prefix.
- **`set_updated_at()` reuse:** The function was defined in migration 001. Migration 003 adds a trigger on `push_subscriptions` that calls it. Do not redefine the function in this file — just create the trigger.
- **`security definer` on `get_or_create_daily_limit`:** The function must bypass the no-INSERT-policy restriction on `daily_send_limits`. By running as the function owner (service role), it can insert the first row of the day. Without `security definer`, the first send each day would fail with a policy violation error.
- **`on conflict (user_id, date) do nothing` in the function:** Handles the race condition where two near-simultaneous calls both enter the `if not found` branch and both attempt the INSERT. Postgres serialises the two INSERTs via the unique constraint — the second call silently does nothing, and the re-SELECT returns the row created by the first call.
- **Subscription object privacy:** The `push_subscriptions.subscription` column contains cryptographic keys (`p256dh`, `auth`). These must never be readable by the partner user. The RLS policies enforce this at the database level. The Edge Function (T403) reads recipient subscriptions using the Supabase service role client, not the authenticated sender's session.
- **Notifications are immutable from the client:** The `status` column is updated only by the Edge Function after the Web Push API responds. No client-side UPDATE policy exists on `notifications`. If the Edge Function is unreachable, the row stays at `status = 'sent'` until the next delivery attempt.
- **`type` check constraint is additive:** The set `('custom', 'coupon_approval', 'system')` reflects known types at the time of this migration. Adding new types in future migrations requires `ALTER TABLE notifications DROP CONSTRAINT notifications_type_check; ALTER TABLE notifications ADD CONSTRAINT notifications_type_check CHECK (type IN (...new set...));`. Document any additions in the migration that introduces the new type.
- **Daily limits reset automatically:** There is no cron job or trigger to reset limits at midnight. Limits are per `date`, so a new calendar day produces a new row via `get_or_create_daily_limit`. Old rows persist as a historical record of notification activity.
- **CoYYns integration:** The spend path (buying bonus sends) is handled by the application layer — the app inserts a `coyyns_transactions` row with `category = 'notification_purchase'` and `metadata = { "sends_purchased": 1 }`, then calls an Edge Function or RPC to increment `bonus_sends_available`. The database schema does not enforce this link — it is a product-layer concern. See T402 (push service) and T301 (CoYYns) for the full flow.
- **`recipient_id` FK on notifications:** Both sender and recipient FKs cascade-delete if the referenced profile is deleted. In practice, profiles are never deleted in this app — but the constraint is defensive.

## Edge Cases

- **Same device subscribes twice:** The `UNIQUE(user_id, subscription)` constraint prevents duplicate rows. The client should upsert via `ON CONFLICT DO NOTHING` so a repeat subscribe call is silent and safe.
- **Push subscription expires or becomes invalid:** Browsers can invalidate push subscriptions silently. When the Web Push API returns a `410 Gone` or `404 Not Found` response, the Edge Function (T403) must delete the corresponding row from `push_subscriptions`. The schema supports this — no foreign keys block deletion. T403 is responsible for this cleanup.
- **Recipient has no active subscriptions:** If `push_subscriptions` has no rows for the recipient, the Edge Function finds nothing to send to. The notification row should still be inserted into `notifications` with `status = 'failed'` and `metadata = { "reason": "no_subscriptions" }`. This is a valid outcome — the recipient has not registered any device.
- **Sender tries to send to a non-partner user:** The application layer must enforce that `recipient_id = sender.partner_id`. The database schema does not enforce this — `recipient_id` is any valid `profiles.id`. This is an intentional tradeoff: the schema is generic enough to support future notification types (e.g., system notifications with `sender_id = system_user_id`). The Edge Function (T403) must validate the partner relationship before processing any client-initiated sends.
- **Two concurrent sends from the same user at midnight boundary:** If two sends happen within milliseconds of midnight, both calls to `get_or_create_daily_limit` may see `current_date` as either the old day or the new day depending on when the Postgres transaction starts. The UNIQUE constraint and `on conflict do nothing` pattern handle this correctly — each day always has exactly one row per user.
- **`free_sends_used` incremented but push fails:** If the daily limit row is incremented before the Edge Function fires and the Edge Function subsequently fails (network error, etc.), the free send is consumed but no notification was delivered. The notification row will have `status = 'failed'`. This is an acceptable edge case for V1 — do not decrement `free_sends_used` on failure, as this opens a retry loop that could be exploited. Log the failure in `metadata` instead.
- **User has multiple devices:** The Edge Function will find multiple rows in `push_subscriptions` for the recipient. It must fan out and attempt delivery to all of them. The `notifications` row represents one logical notification send, not one device delivery. `status = 'delivered'` means at least one device received it.
- **Notification from system (type = 'system'):** System notifications may have `sender_id` pointing to a special system profile or a service-role-generated UUID. The `notifications: own insert` RLS policy only allows inserts where `auth.uid() = sender_id`. System inserts must come from an Edge Function running as service role (bypassing RLS), not from the client.

## What This Connects To

- **T201 (Auth DB Migration)** — all three tables reference `public.profiles.id`. This migration cannot run without T201.
- **T301 (CoYYns DB Migration)** — the bonus sends purchase flow inserts a `coyyns_transactions` row with `category = 'notification_purchase'`. The CoYYns wallet must exist for the spend to succeed. T401 does not directly reference CoYYns tables — the link is at the application layer.
- **T402 (Push Service — client hook)** — the React hook that calls `get_or_create_daily_limit`, checks the user's remaining sends, manages the subscription registration flow, and exposes a `sendNotification()` function to the UI.
- **T403 (Push Edge Function)** — the Supabase Edge Function that reads `push_subscriptions` for the recipient (as service role), calls the Web Push API, and updates the `notifications` row status. This is the only writer to `notifications.status`.
- **T404 (Notification Composer UI)** — the bottom sheet or modal where Yahya or Yara composes and sends a notification. Uses T402's hook. Reads today's limit from `daily_send_limits` to show the sends remaining indicator.
- **Coupons feature (future)** — when a love coupon is submitted for approval, it will insert a row into `notifications` with `type = 'coupon_approval'` and `metadata = { "coupon_id": "..." }`. The schema supports this without modification.

- Log in as Yahya → attempt `update notifications set status = 'delivered' where sender_id = '<yahya_uuid>'` → receives policy violation error (no UPDATE policy for client)

---

## Do NOT

- Do not allow clients to read another user's `push_subscriptions` rows — the subscription object contains cryptographic keys. Only the Edge Function (T403) reads recipient subscriptions, and it does so as service role.
- Do not add a client-side UPDATE policy to `notifications` — only the Edge Function should update `status`. A client that can set `status = 'delivered'` can fake delivery confirmations.
- Do not add an INSERT policy to `daily_send_limits` — the `get_or_create_daily_limit` function is the only valid creation path. Bypassing it lets a client create a limit row with arbitrary `bonus_sends_available`.
- Do not use a cron job or trigger to reset daily limits at midnight — the per-`date` row model handles resets naturally with zero operational overhead. New day, new row.
- Do not add a `UNIQUE(user_id)` constraint to `push_subscriptions` — users can and should register multiple devices. The unique constraint is on `(user_id, subscription)` to prevent duplicate registrations of the same device endpoint.
- Do not store `free_sends_limit` (the allowance cap of 2) in the database — it is an application constant defined in T402 (`MAX_FREE_SENDS_PER_DAY = 2`). If the allowance changes, update the constant in one place, not a database column across many rows.
- Do not use `text` or `varchar` with a length limit for `subscription` — use `jsonb`. The PushSubscription object is structured data and benefits from Postgres JSON indexing and validation.
- Do not grant the anon role any access to any of these three tables — all notification data is authenticated-users-only.
- Do not add a partner read policy to `daily_send_limits` — each user's send quota is personal information, not shared state.
- Do not run this migration before `001_auth_profiles.sql` — the FK constraints on all three tables reference `public.profiles.id`. Running this migration first will fail immediately.
- Do not skip the `set search_path = public` hardening on `get_or_create_daily_limit`.
- Do not decrement `free_sends_used` when a send fails — this prevents a retry loop exploit. Log the failure in `notifications.metadata` and let the user's free send count stand.
