## Why This Exists

Cycle tracking is deeply personal and entirely YAHYA-ONLY. This form is the configuration gateway — without it, the entire cycle tracking system has no grounding. Yahya opens this the first time she sets up her health module, enters when the current pill pack started, and confirms the cadence that governs every downstream calculation: when PMS warnings trigger, when break days begin, when the next pack starts. In edit mode, she can adjust any parameter as her regimen changes. This form is not shared with Yara. It must never appear in Yara's session under any circumstance.

This is intimate tooling. It should feel calm, unhurried, warm — like writing in a personal health journal. The Playfair Display title, the copper save button, the generous field spacing all reinforce: this is a safe space to input personal health data.

---

## User Story

As Yahya, when I tap "Set up cycle tracking" for the first time (or "Edit settings" later), a form slides up where I can enter when my current pill pack started and configure how many active days, break days, and PMS warning days to track. I fill in the details, tap Save, and the cycle engine immediately knows my current status. I can come back and edit these settings any time my prescription changes.

---

## Full UX Flow

1. **First-time setup** → CycleConfigForm opens with `initialConfig = null`, all fields at their defaults
2. **Edit mode** → CycleConfigForm opens with `initialConfig` pre-filled from the useCycle hook
3. **Form fields visible:**
   - `pill_start_date` — date input at the top, required, labeled "When did the current pack start?"
   - `active_days` — number input, default 21, labeled "Active pill days"
   - `break_days` — number input, default 7, labeled "Break days"
   - `pms_warning_days` — number input, default 3, labeled "PMS warning (days before break)"
   - `notes` — textarea, optional, labeled "Personal notes"
4. **User fills fields and taps Save:**
   - Zod validation runs client-side on submit
   - If valid → calls `updateConfig()` from useCycle hook
   - On success → calls `onSuccess?.()` then `onClose()`
   - Save button shows loading spinner while in flight
5. **User taps Cancel or closes sheet** → `onClose()` fires, no data saved, form resets
6. **Validation errors** → shown inline below each field, red text, never toast-only
7. **YAHYA-ONLY guard** → if the current session user is not Yahya, the component renders `null` and logs a warning. This is enforced at the component level regardless of how it is invoked.

## Zod Schema

```ts
const cycleConfigSchema = z.object({
  pill_start_date: z.string().min(1, "Start date is required"),
  active_days: z.number().int().min(1).max(31),
  break_days: z.number().int().min(1).max(14),
  pms_warning_days: z.number().int().min(0).max(31), // validated against active_days with .refine
  notes: z.string().max(500).optional(),
}).refine(
  (data) => data.pms_warning_days <= data.active_days,
  {
    message: "PMS warning days cannot exceed active days",
    path: ["pms_warning_days"],
  }
)

type CycleConfigFormValues = z.infer<typeof cycleConfigSchema>
```

## Implementation Notes

- `"use client"` directive required — uses React Hook Form, Framer Motion, Sheet
- Import `useCycle` from `@/hooks/use-cycle` (T502) — use `updateConfig` action and `isOwner` flag
- **YAHYA-ONLY guard at the top of the component body:**
  ```tsx
  const { isOwner } = useCycle()
  if (!isOwner) return null
  ```
  `isOwner` is a boolean on the useCycle return that is true only when the authenticated user is Yahya. Do not skip this check. Do not gate it behind a prop. It is always enforced internally.
- Use `useForm<CycleConfigFormValues>` with `zodResolver(cycleConfigSchema)`
- Default values: `{ active_days: 21, break_days: 7, pms_warning_days: 3, notes: "" }`
- When `initialConfig` is provided, use `reset(initialConfig)` inside a `useEffect` when the Sheet opens
- Number inputs: use shadcn Input with `type="number"`, register with `valueAsNumber: true` option on `register()`
- Date input: use `type="date"`, value as ISO string `YYYY-MM-DD`
- Submit handler calls `await updateConfig(values)` — handle the async result to decide success vs error
- Show a loading spinner inside the Save button while submitting (`isSubmitting` from `formState`)
- On successful save, call `onSuccess?.()` followed by `onClose()` and `form.reset()`
- Wrap the Sheet content in `motion.div` with `initial={{ opacity: 0, y: 8 }}` and `animate={{ opacity: 1, y: 0 }}` for a gentle entrance, `transition={{ duration: 0.2, ease: "easeOut" }}`
- Use `cn()` from `@/lib/utils` for all conditional class merging
- All label strings go in `lib/i18n/` — English and Arabic keys required

## Edge Cases

- **First-time setup with no initialConfig** → pill_start_date field is empty, user must fill it. Save button disabled until the field has a value (rely on Zod validation, not manual disable logic).
- **Editing existing config** → `useEffect` must call `form.reset(mappedInitialConfig)` when `open` changes to `true`, not just on mount, so stale values don't persist between opens.
- **pms_warning_days > active_days** → caught by the `.refine` rule on the schema. Error appears under the `pms_warning_days` field.
- **Yara somehow triggers this component** → `isOwner` guard returns `null` immediately. The Sheet never mounts. Nothing leaks.
- **Network failure on save** → `updateConfig` rejects. Catch the error, set a form-level error via `setError("root", { message: "Failed to save. Please try again." })` and display it above the Save button. Do not close the Sheet.
- **User closes Sheet mid-edit without saving** → `onClose()` fires, Sheet unmounts, form state is discarded. On next open, `useEffect` re-initializes from `initialConfig` (clean state).
- **notes field at 500 character limit** → show a character counter below the textarea: `{watchNotes.length}/500` in `text-secondary`.

## Do NOT

- Do not render any part of this component for non-Yahya users — the `isOwner` guard is not optional
- Do not hardcode `#C4956A` or any hex value — use Tailwind theme tokens only
- Do not use `router.push()` or navigate anywhere from this form — it is a modal/sheet
- Do not use `toast` as the sole error reporting mechanism — inline field errors are required
- Do not skip the `valueAsNumber` option on number inputs — string values will break Zod int validation
- Do not use `left`/`right` Tailwind utilities — use logical properties (`ps`, `pe`, `ms`, `me`) for RTL readiness
- Do not define `CycleConfig` type locally — import from `lib/types/health.types.ts`
- Do not add biometric or cycle phase display here — this form is configuration only, not a dashboard
