## Why This Exists

A coupon redemption is a small ceremony between two people. Yara sees a "One Free Breakfast in Bed" coupon, decides today is the day, and taps Redeem. That moment deserves more than a silent API call. It needs a confirmation that makes her feel the intention of the action, a clear waiting state so she knows Yahya is being notified, and a joyful resolution when he says yes — or a soft landing when he says not today.

On Yahya's side, the approval is just as ceremonial. He gets a push notification, opens the app, and sees a clear approval dialog. "Approve" should feel like a gift given. "Reject" should feel handled with care — with an optional reason so Yara isn't left wondering.

`CouponRedeemFlow` is the orchestrator for this entire ceremony. CouponDetail (T606) provides the action buttons. This component provides the confirmation dialogs, the multi-step waiting states, the realtime resolution, and the emotional closure on both ends.

---

## User Story

As Yara (recipient), when I tap "Redeem" on a coupon, I want a confirmation dialog that shows me exactly what I'm about to redeem. After confirming, I want to see a warm waiting state that tells me Yahya has been notified. When he approves, I want a celebration moment. If he rejects, I want to know gently — with any reason he chose to share.

As Yahya (creator), when I get a push notification and tap into the approval flow, I want a clear dialog that tells me what Yara wants to redeem right now. One tap to approve, or a short rejection flow with an optional reason. Both paths feel intentional and unhurried.

## Full UX Flow

### Recipient Side — Redeem Flow

1. Yara taps "Redeem This Coupon" inside CouponDetail (T606)
2. CouponDetail calls `onRedeemIntent?.()` — this opens `RedeemConfirmDialog`
3. **Step 1 — Confirmation:** Dialog slides up. Shows the coupon emoji (large, centered), the title in Playfair Display, a one-line description, and two buttons: "Redeem" (copper, full width) and "Cancel" (ghost, below)
4. Yara taps "Redeem" → button shows spinner, calls `redeemCoupon(coupon.id)` from T602 hook
5. On success → `coupon.status` becomes `'pending_approval'`
6. **Step 2 — Waiting:** Dialog transitions (Framer Motion `AnimatePresence` between steps) to a waiting state: Yahya's name, a gentle pulsing animation (soft copper ring pulsing around the coupon emoji), text: "Waiting for [Yahya]'s approval…"
7. A cancel/dismiss text link is visible: "I'll wait" (closes dialog, coupon remains in pending state)
8. **Realtime resolution:** Supabase realtime (via T602) fires when the coupon status changes
   - If `status → 'redeemed'` → dialog transitions to **Step 3 — Approved**
   - If `status → 'rejected'` → dialog transitions to **Step 3 — Rejected**
9. **Step 3a — Approved:** Celebration state: large emoji bounces in, title, "Approved! Enjoy it." in `text-success`. A "Close" button dismisses. Calls `onRedeemed?.()`.
10. **Step 3b — Rejected:** Soft landing: emoji appears, title, "Not this time" in `text-secondary`. If `coupon.rejection_reason` exists, show it in a quoted block below. A "Close" button dismisses.

### Creator Side — Approval Flow

1. Yahya receives a push notification: "Yara wants to redeem: One Free Massage"
2. He taps the notification → app opens → sees pending approval badge on Coupons tab (badge count from `pendingApprovals.length` in T602)
3. He navigates to CouponDetail for the pending coupon → sees "Approve" and "Reject" buttons (T606 renders these)
4. **Tapping "Approve":** opens `ApprovalDialog`
   - Shows coupon emoji, title, partner's name
   - Text: "Approve this redemption for [Yara]?"
   - Two buttons: "Approve" (success green, full width) and "Cancel" (ghost)
   - On confirm → spinner → calls `approveCoupon(coupon.id)` from T602 → success → calls `onApproved?.()` → dialog closes
5. **Tapping "Reject":** opens `ApprovalDialog` in rejection mode directly
   - Shows coupon emoji, title
   - Text: "Reject this redemption?"
   - `textarea` (3 rows) for optional reason: placeholder "Add a reason for Yara (optional)"
   - Two buttons: "Confirm Reject" (error outline, full width) and "Cancel" (ghost)
   - On confirm → spinner → calls `rejectCoupon(coupon.id, reason)` from T602 → success → calls `onRejected?.()` → dialog closes

## Implementation Notes

- `"use client"` directive required on the file — Framer Motion, hooks, event handlers
- Import `useCoupons` from `@/lib/hooks/use-coupons` (T602) — use `redeemCoupon`, `approveCoupon`, `rejectCoupon`
- Import `useAuth` from `@/lib/providers/AuthProvider` (T204) — used to display partner's name in dialog copy
- `RedeemConfirmDialog` uses `useState<RedeemStep>` to drive the multi-step view — initial value `'confirm'`
- When `redeemCoupon` succeeds, transition to `'waiting'`. Do not close the dialog yet.
- The `'waiting'` step watches `coupon.status` — since T602 maintains realtime sync, `coupon` prop will update when status changes. Use `useEffect` watching `coupon.status`: if `'redeemed'` → set step to `'approved'` and call `onRedeemed?.()`. If `'rejected'` → set step to `'rejected'`.
- Reset step to `'confirm'` when `open` transitions from `false` to `true` (use `useEffect` watching `open`)
- `ApprovalDialog` uses `useState` for `isLoading`, `error`, and `rejectReason` (textarea value)
- When `mode === 'reject'`, the textarea is visible from the start — do not re-animate it in. The dialog already represents the reject path.
- Use `useRef<HTMLTextAreaElement>` for the reject textarea — auto-focus via `ref.current?.focus()` inside `useEffect` when the dialog opens in reject mode
- All buttons show `<Loader2 className="animate-spin" />` from Lucide while `isLoading` is true and are `disabled`
- On successful `approveCoupon`: call `onApproved?.()` then `onClose()`
- On successful `rejectCoupon`: call `onRejected?.()` then `onClose()`
- On error: set `error` state, display below the action button in `text-error` 13px, do not close the dialog
- Backdrop tap while `isLoading` is true → do nothing. Only close on backdrop tap when `isLoading` is false.
- Use `cn()` from `@/lib/utils` for all conditional class logic
- Partner name: read from `useAuth().partner?.full_name ?? 'your partner'`
- Trim the rejection reason before passing: `rejectReason.trim() || undefined`
- Both components export as named exports from the same file

## Edge Cases

- **Coupon status changes to `'redeemed'` before the user opens the waiting state** — if `RedeemConfirmDialog` opens and `coupon.status` is already `'redeemed'`, skip directly to `'approved'` step on mount
- **Realtime update fires while dialog is closed** — the dialog is not mounted when `open` is false, so no state leak; on next open, reset to `'confirm'`
- **Partner name is null** — fall back to `"your partner"` in all dialog copy; never render an empty space where the name should be
- **`approveCoupon` or `rejectCoupon` called on a coupon that is no longer pending** — T602 hook will throw; catch the error, set `error` state, keep dialog open
- **`rejectReason` is whitespace only** — trim before passing; pass `undefined` not empty string so T602 saves `null` in the database
- **User taps "I'll wait" in the waiting state** — dialog closes, `coupon.status` remains `'pending_approval'`. The next time they open CouponDetail, they see the disabled "Waiting for approval…" button (T606 handles that display). `onRedeemed` is NOT called.
- **Very long coupon title** — allow wrapping inside dialogs; max-width `max-w-xs mx-auto` on the title element
- **Both dialogs open simultaneously** — this cannot happen by design (a coupon has one status at a time and each user sees one dialog). No guard needed but document the assumption.

## Do NOT

- Don't call push notification APIs directly from this component — T602 handles notifications inside its action functions
- Don't derive step state from `coupon.status` alone on first render — always reset to `'confirm'` when dialog opens fresh
- Don't hardcode `"Yahya"` or `"Yara"` in any dialog copy — always use the partner name from `useAuth()`
- Don't use `window.confirm()` or `alert()` — all confirmation is handled in-dialog with Framer Motion transitions
- Don't close dialogs while an action is in flight — protect all dismiss paths with the `isLoading` guard
- Don't pass an empty string to `rejectCoupon` — trim and convert to `undefined` if empty
- Don't hardcode hex colors — use Tailwind design tokens only
- Don't skip `AnimatePresence mode="wait"` on step transitions — without `mode="wait"`, exit and enter animations overlap and look broken
- Don't forget `key` props on each step's `motion.div` inside `AnimatePresence` — the key change is what triggers the animation
- Don't use `left`/`right` CSS properties — use logical properties (`ps`, `pe`, `ms`, `me`) for RTL readiness
