## Why This Exists

Hayah is built for two people. There is no public-facing surface, no discovery, no registration flow. The login page is not a marketing moment — it's a private door. When Yahya or Yara picks up their phone and taps the app icon, this is the first thing they see if their session has expired. It should feel calm, warm, and immediate. Not corporate. Not clinical. Just a quiet, beautiful entry into their shared world.

Because there are only two users, we don't need "Forgot password", "Create account", or any recovery flow in the UI. Yahya manages both accounts. The form is minimal by design — email, password, and a copper button that opens the door.

---

## User Story

As Yahya or Yara, when my session has expired or I open the app fresh, I see a clean login screen that asks for my email and password. I type my credentials, tap "Sign In", and I'm taken straight into the app. If I mistype something, I see a clear inline error. The whole experience feels like it takes one second. There's no noise, no distractions, nothing I don't need.

---

## Full UX Flow

1. **User navigates to /login** → sees the warm, minimal login form centered on screen
2. **User types email** → field accepts input normally; validation only triggers on submit or on blur
3. **Email field loses focus** → if the value is not a valid email format, an inline error appears below the field: "Enter a valid email"
4. **User types password** → input is type="password", characters are hidden, no show/hide toggle
5. **User taps "Sign In"** → button immediately changes label to "Signing in..." and shows a subtle spinner; button is disabled to prevent double-submit
6. **Success path** → Supabase `signInWithPassword` returns session → router.push("/") → user lands on home dashboard
7. **Failure path** → Supabase returns an auth error → button resets to "Sign In" → inline error message appears below the form: "Invalid email or password"
8. **Already authenticated** → middleware detects active session and redirects to "/" before this page ever renders — the login page never flashes for authenticated users
9. **Form is empty on submit** → Zod validation fires, inline errors appear on both fields simultaneously

## Implementation Notes

- This page lives in the `(auth)` route group: `src/app/(auth)/login/page.tsx`
- The `(auth)` group uses its own layout (`src/app/(auth)/layout.tsx`) that does NOT include BottomNav or any authenticated shell
- Mark the page as `"use client"` — it uses React Hook Form and Supabase client-side auth
- Import Supabase browser client from `@/lib/supabase/client` (T202 delivers this)
- Use `useRouter` from `next/navigation` for the post-login redirect — call `router.push("/")`
- Use `useForm` from `react-hook-form` with `zodResolver` from `@hookform/resolvers/zod`
- The `handleSubmit` wrapper from RHF handles validation before your submit function fires
- Call `supabase.auth.signInWithPassword({ email, password })` — destructure `{ data, error }`
- On error, use `setError("root", { message: "Invalid email or password" })` from RHF to surface the auth error
- Read the root error via `formState.errors.root?.message` and render it below the button
- Button disabled state: `isSubmitting` from `formState` — RHF sets this automatically during async submit
- Wrap the entire page content in a Framer Motion `motion.div` with `initial={{ opacity: 0, y: 8 }}`, `animate={{ opacity: 1, y: 0 }}`, `transition={{ duration: 0.35, ease: "easeOut" }}` for a gentle entrance
- Use `cn()` from `@/lib/utils` for conditional input border styling (error state gets `border-destructive`)
- Do not use shadcn `<Form>` — use raw RHF with shadcn `<Input>` and `<Button>` primitives for simplicity
- The `(auth)/layout.tsx` should be a simple server component that renders `{children}` with no nav wrappers

## Edge Cases

- **Session already active when navigating to /login** → middleware redirects to `/` before render; the page component itself does not need to handle this
- **Both fields empty on submit** → Zod fires, both fields show inline errors simultaneously; no network call is made
- **Valid email format, wrong credentials** → form passes client validation, Supabase call fires, error returned, root error displayed; field-level errors are cleared
- **Network offline** → Supabase call throws or times out; catch the exception and display "Something went wrong. Check your connection." as the root error
- **Double-tap on Sign In** → button is disabled during `isSubmitting`, second tap is a no-op
- **Password manager autofill** → inputs must have correct `name="email"` and `name="password"` attributes and `autocomplete="email"` / `autocomplete="current-password"` so password managers and iOS Keychain work correctly
- **Very long email address** → input should truncate visually but not clip — `overflow: hidden` with `text-ellipsis` behavior is acceptable
- **Arabic locale** → the "حياة" subtitle must render correctly; ensure the font stack includes a fallback that supports Arabic script even though DM Sans doesn't cover it (system-ui fallback handles this)

## What This Connects To

- **T201 (Profiles)** — after login, the app will fetch the user's profile row from Supabase; login must complete and return a valid session before profile data is accessible
- **T202 (Supabase client)** — this page imports and calls `createBrowserClient` from T202; T202 must be complete before T203 can be implemented
- **T204 (AuthProvider)** — the AuthProvider wraps all authenticated routes and listens for session changes; a successful login here fires the `onAuthStateChange` event that AuthProvider uses to hydrate session state
- **Middleware** — the Next.js middleware reads the Supabase session cookie and redirects unauthenticated users to `/login`, and redirects already-authenticated users away from `/login`; T203 and the middleware are two sides of the same door
- **AppShell (T207)** — the authenticated layout that loads after login; the `(auth)` route group deliberately excludes AppShell so the login page has no nav chrome

## Do NOT

- Don't add a "Forgot password" link — there are only 2 users and Yahya manages accounts
- Don't add a "Register" or "Create account" link — users are pre-created in Supabase
- Don't add a password visibility toggle — unnecessary complexity for a 2-user app
- Don't use `router.replace()` instead of `router.push()` — push is correct here so back-navigation works from the OS level (though middleware will redirect back to / if session exists)
- Don't hardcode color hex values — use Tailwind theme tokens (`bg-accent-primary`, `text-secondary`, etc.)
- Don't add social login buttons (Google, Apple, etc.) — email/password only
- Don't add "Remember me" checkbox — Supabase sessions persist by default
- Don't render BottomNav, PageHeader, or any authenticated shell chrome on this page
- Don't validate the email field on every keystroke — validate on blur and on submit only
- Don't use `alert()` or `window.alert()` for errors — inline RHF errors only
