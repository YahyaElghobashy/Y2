## Why This Exists

Hayah's notifications are not generic app pings — they are intentional acts of love between two people. When Yahya taps "Send Notification", he is reaching through his phone screen and tapping Yara on the shoulder. That action must feel consequential: a clear success state when sent, a visible count of how many sends remain today, and a firm (but kind) wall when the daily limit is reached. The limit itself is not punitive — it exists so that notifications stay meaningful and never become noise.

The push service and notification hook are the two pieces of client-side infrastructure that make all of this possible. The push service manages the browser's PushSubscription lifecycle — requesting permission, storing the subscription endpoint in Supabase so the backend can reach this device, and cleaning it up when the user disables notifications. The `useNotifications` hook is the single interface every notification UI component talks to. It holds the list of notifications, the daily limit state, and the `sendNotification` action that orchestrates the full flow from insert to delivery.

Every notification UI component — the permission prompt (T404), the notification builder (T405), the limit indicator (T406) — imports this hook and nothing else. They never touch Supabase directly.

---

## User Story

As Yahya or Yara, I want to send my partner a push notification from inside the app. I see how many sends I have left today. When I tap send, my partner's phone receives it within seconds. If I've used my daily free sends, I can spend a bonus send instead. If I have no sends at all remaining, the send button is disabled and I understand why. My notification history shows both what I've sent and what I've received, most recent first.

## Full UX Flow

1. **App mounts** → `useNotifications` hook initializes, loads notification history and daily limit from Supabase
2. **User opens notification builder (T405)** → calls `useNotifications` which exposes `canSend`, `remainingSends`, and `sendNotification`
3. **User taps Send:**
   - Hook checks `canSend`. If false, rejects immediately with a clear error message
   - Hook determines which send type to consume: free first, then bonus
   - Inserts a row into `notifications` table with status `pending`
   - Calls the T403 Edge Function to deliver the push notification
   - Edge Function updates the notification row status to `delivered` or `failed`
   - Hook refreshes `daily_send_limits` to reflect the updated counts
   - Hook prepends the new notification to the local `notifications` array (optimistic update)
4. **T406 (Limit Indicator)** reads `remainingSends` and `dailyLimit` from the hook, re-renders reactively
5. **User disables push in browser settings** → `getPushPermission()` returns `denied` → T404 permission prompt reflects this state without breaking
6. **New device / first login** → `subscribeToPush()` is called by T404 after the user grants permission → subscription is upserted to `push_subscriptions` in Supabase

## Implementation Notes

### 1. `src/lib/types/notification.types.ts` — Type Definitions

```ts
export type PushPermissionState = 'granted' | 'denied' | 'default' | 'unsupported'

export type NotificationStatus = 'pending' | 'delivered' | 'failed'

export type Notification = {
  id: string
  sender_id: string
  receiver_id: string
  title: string
  body: string
  emoji: string | null
  status: NotificationStatus
  send_type: 'free' | 'bonus'
  created_at: string
}

export type DailyLimit = {
  id: string
  user_id: string
  date: string                    // ISO date string YYYY-MM-DD
  free_sends_used: number
  bonus_sends_used: number
  bonus_sends_available: number
}

export type UseNotificationsReturn = {
  notifications: Notification[]
  dailyLimit: DailyLimit | null
  canSend: boolean
  remainingSends: number
  isLoading: boolean
  error: string | null
  sendNotification: (title: string, body: string, emoji?: string) => Promise<void>
  refreshLimits: () => Promise<void>
}
```

### 2. `src/lib/services/push-service.ts` — Push Subscription Management

Four exported functions, each with a single responsibility:

**`isPushSupported()`**
```ts
export function isPushSupported(): boolean {
  return (
    typeof window !== 'undefined' &&
    'serviceWorker' in navigator &&
    'PushManager' in window &&
    'Notification' in window
  )
}
```

**`getPushPermission()`**
```ts
export function getPushPermission(): PushPermissionState {
  if (!isPushSupported()) return 'unsupported'
  return Notification.permission as PushPermissionState
}
```

**`subscribeToPush(userId: string)`**
- Calls `Notification.requestPermission()` — returns early with `null` if not `granted`
- Gets the active service worker registration via `navigator.serviceWorker.ready`
- Calls `registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: urlBase64ToUint8Array(process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!) })`
- Converts the `PushSubscription` to JSON: `subscription.toJSON()`
- Upserts to Supabase `push_subscriptions` table: `{ user_id, endpoint, keys, user_agent, created_at }`
- Use `onConflict: 'endpoint'` for the upsert so re-subscribing the same device is idempotent
- Returns the `PushSubscription` object or `null` on failure
- Include a `urlBase64ToUint8Array` helper function (private, not exported) to convert the VAPID key

**`unsubscribeFromPush(userId: string)`**
- Gets the active service worker registration
- Calls `registration.pushManager.getSubscription()`
- If a subscription exists, calls `subscription.unsubscribe()`
- Deletes the row from `push_subscriptions` where `user_id = userId` and `endpoint = subscription.endpoint`
- Returns `true` on success, `false` if no subscription was found

### 3. `src/lib/hooks/use-notifications.ts` — The Notification Hook

`"use client"` is required. Import `useAuth` from `@/lib/hooks/use-auth`.

**State:**
```ts
const [notifications, setNotifications] = useState<Notification[]>([])
const [dailyLimit, setDailyLimit] = useState<DailyLimit | null>(null)
const [isLoading, setIsLoading] = useState(true)
const [error, setError] = useState<string | null>(null)
```

**Derived values (useMemo):**
```ts
const freeSendsRemaining = dailyLimit ? (FREE_SENDS_PER_DAY - dailyLimit.free_sends_used) : 0
const bonusSendsRemaining = dailyLimit ? (dailyLimit.bonus_sends_available - dailyLimit.bonus_sends_used) : 0
const remainingSends = Math.max(0, freeSendsRemaining) + Math.max(0, bonusSendsRemaining)
const canSend = remainingSends > 0
```

Define `FREE_SENDS_PER_DAY = 2` as a module-level constant, not a magic number inline.

**`loadData()` (called in useEffect on mount):**
- Fetch notifications: `supabase.from('notifications').select('*').or('sender_id.eq.${userId},receiver_id.eq.${userId}').order('created_at', { ascending: false }).limit(50)`
- Fetch today's limit: `supabase.from('daily_send_limits').select('*').eq('user_id', userId).eq('date', today).single()`
- If no limit row exists for today (PGRST116 error code), `dailyLimit` is treated as all-zeroes
- Run both fetches in parallel with `Promise.all`
- Set `isLoading` to `false` in the `finally` block

**`refreshLimits()` (exposed in return value):**
- Re-fetches only the `daily_send_limits` row for today
- Updates `dailyLimit` state
- Does not reset `notifications` — only limits need refreshing after a send

**`sendNotification(title, body, emoji?)`:**
1. Guard: if `!canSend`, set `error` and return early — do not throw
2. Determine `sendType: 'free' | 'bonus'` — use `'free'` if `freeSendsRemaining > 0`, otherwise `'bonus'`
3. Get `receiver_id` from `partner.id` via `useAuth()`
4. Insert into `notifications` table: `{ sender_id: user.id, receiver_id: partner.id, title, body, emoji: emoji ?? null, status: 'pending', send_type: sendType }`
5. Optimistically prepend the new notification to local `notifications` state immediately
6. Call Edge Function: `supabase.functions.invoke('send-push-notification', { body: { notificationId: insertedRow.id } })`
7. Call `refreshLimits()` after Edge Function returns
8. On any error: set `error` with a user-readable message, do not leave the optimistic row in state — remove it or mark it as failed

## Edge Cases

- **Push not supported** (desktop Firefox without service worker, or non-HTTPS) → `isPushSupported()` returns `false` → T404 shows a non-blocking informational state, never crashes
- **Permission denied** → `subscribeToPush()` returns `null` without throwing. The push service never calls `Notification.requestPermission()` more than once per invocation — repeated calls after denial are browser no-ops anyway
- **VAPID key missing** → The `!` assertion on `process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY` will throw at runtime. This is intentional — missing VAPID key is a deployment error, not a user error
- **No daily_send_limits row for today** → Treat as `{ free_sends_used: 0, bonus_sends_used: 0, bonus_sends_available: 0 }`. Do not crash. The row is created by T403 Edge Function on first send of the day
- **partner is null** → `sendNotification` must guard against this. If `partner` is null (partner_id not set), reject with `"Partner not connected"` error before any Supabase call
- **Edge Function returns an error** → The notification row already exists with `status: 'pending'`. The hook sets `error` state but does not delete the row — T403 handles retry logic server-side
- **Rapid double-tap on send** → Debounce or disable the send action during the in-flight period. Use a `isSending` ref (not state) to avoid re-renders: `const isSending = useRef(false)`
- **Network offline** → `supabase.from('notifications').insert()` will fail. Catch the error, remove the optimistic row, set `error` to a friendly offline message
- **useNotifications called without AuthProvider** → `useAuth()` will throw its own descriptive error. Do not swallow it

## What This Connects To

- **T401 (DB schema)** — this hook reads and writes `notifications`, `daily_send_limits`, and `push_subscriptions`. Must be complete before T402 can be tested end-to-end
- **T202 (Supabase client)** — imports `getSupabaseBrowserClient()` for all data operations
- **T204 (AuthProvider / useAuth)** — reads `user`, `profile`, and `partner` for sender/receiver IDs
- **T403 (Edge Function)** — `sendNotification` invokes this to deliver the push. T402 does not know or care about VAPID signing or Web Push protocol — that is T403's domain
- **T404 (Permission Prompt)** — calls `isPushSupported()`, `getPushPermission()`, and `subscribeToPush()` from the push service
- **T405 (Notification Builder)** — calls `sendNotification()`, reads `canSend`, `remainingSends`, `isLoading`, `error`
- **T406 (Limit Indicator)** — reads `remainingSends`, `dailyLimit`, `canSend`

## Do NOT

- Do not call `Notification.requestPermission()` from the hook — that is the push service's responsibility, triggered only from T404
- Do not hardcode `2` as the free sends limit inline — use the `FREE_SENDS_PER_DAY` constant
- Do not expose the raw Supabase query objects from the hook — only expose the typed return shape
- Do not delete the `notifications` row if the Edge Function fails — T403 owns status updates
- Do not make the hook responsible for scheduling or retrying failed notifications — that is Edge Function territory
- Do not subscribe to Supabase Realtime in this hook for the initial implementation — polling via `refreshLimits` is sufficient; Realtime can be added in a later task
- Do not use `left`/`right` CSS in any future UI built on top of this hook — use logical properties (`ps`, `pe`)
- Do not call `getSupabaseServerClient()` from this hook — it is a client component and must use `getSupabaseBrowserClient()`
