## Why This Exists

Coupons are a love language. Yahya creates a "breakfast in bed" coupon for Yara. Yara sees it in her wallet, decides to redeem it on a slow Sunday morning, and taps Redeem. Yahya gets a notification, smiles, accepts it, and they share a moment. The entire lifecycle — creation, discovery, redemption, approval — flows through this hook.

`useCoupons` is the single client-side contract for all coupon state and actions. Every component that touches coupons — the wallet grid, the detail sheet, the creation form, the approval queue — gets its data and callbacks from here. The hook owns the fetching, the mutations, the realtime subscription, and the notification triggers. Components stay thin. The hook does the work.

---

## User Story

As Yahya or Yara, I want my coupon wallet to always reflect the latest state of all coupons I have created and received. When I redeem a coupon, my partner is instantly notified. When they approve or reject it, my wallet updates in real time without a page refresh. Surprise coupons stay hidden until my partner chooses to reveal them, at which point I get a notification and the coupon appears in my wallet.

## Full Data Flow

1. **Hook mounts** → calls `fetchCoupons()` which runs two parallel Supabase queries:
   - `myCoupons`: `SELECT * FROM coupons WHERE creator_id = me ORDER BY created_at DESC`
   - `receivedCoupons`: `SELECT * FROM coupons WHERE recipient_id = me ORDER BY created_at DESC` — RLS policy automatically excludes unrevealed surprises (`is_surprise = true AND surprise_revealed = false`)
2. **pendingApprovals** is derived client-side: `myCoupons.filter(c => c.status === 'pending_approval')`
3. **Realtime subscription** is opened on `coupons` table filtered to rows where `creator_id = me OR recipient_id = me` — on any INSERT or UPDATE, re-run `fetchCoupons()`
4. **createCoupon(data)** → inserts new row with `creator_id = me`, `recipient_id = partner_id`, `status = 'active'`. If `is_surprise = true`, `surprise_revealed = false`. Returns inserted Coupon.
5. **redeemCoupon(couponId)** → recipient action: updates `status = 'pending_approval'`. Sends push notification to creator: "Yara wants to redeem [title]".
6. **approveCoupon(couponId)** → creator action: updates `status = 'redeemed'`, sets `approved_at = now()`. Sends push notification to recipient: "Your [title] coupon was approved!".
7. **rejectCoupon(couponId, reason?)** → creator action: updates `status = 'rejected'`, sets `rejected_at = now()`, sets `rejection_reason`. Sends push notification to recipient: "Your [title] coupon was not approved".
8. **revealSurprise(couponId)** → creator action: updates `surprise_revealed = true`. Sends push notification to recipient: "You have a surprise coupon waiting!". Inserts a row in `coupon_history` with event `'revealed'`.
9. **Hook unmounts** → unsubscribes from realtime channel to prevent memory leaks

## Hook API

```ts
// File: src/lib/hooks/use-coupons.ts
export function useCoupons(): UseCouponsReturn
```

No parameters. Reads current user from `useAuth()`. Derives partner ID from the two-user system (the user that is not `me`).

## Implementation Notes

- `"use client"` is not needed — this is a plain TypeScript hook, not a component. No directive required.
- Import `createClient` from `@/lib/supabase/client` (T202) — the browser-side Supabase client.
- Import `useAuth` from `@/lib/providers/AuthProvider` (T204) — provides `user.id`.
- Import `useNotifications` from `@/lib/hooks/use-notifications` (T402) — call `sendNotification(recipientId, title, body)` after each mutation.
- Partner ID: query `profiles` table for the user where `id != me`. Cache this in a `useRef` to avoid repeated queries. Alternatively, derive from the app's two-user constraint by selecting the one row in `profiles` where `id <> user.id LIMIT 1`.
- Use `useState` for `myCoupons`, `receivedCoupons`, `isLoading`, and `error`.
- Derive `pendingApprovals` with `useMemo`: `myCoupons.filter(c => c.status === 'pending_approval')`.
- Use `useEffect` for initial fetch + realtime subscription setup. Return cleanup function that calls `supabase.removeChannel(channel)`.
- Realtime channel filter: `.on('postgres_changes', { event: '*', schema: 'public', table: 'coupons' }, fetchCoupons)`. Do not try to filter by creator_id/recipient_id in the channel config — filter after the event fires or just re-fetch both queries.
- All action functions should be wrapped in `useCallback` to prevent unnecessary re-renders in downstream components.
- On any Supabase error, set `error` state to the error message string. Do not throw — let components read the `error` field.
- After each successful mutation, call `refreshCoupons()` to sync local state rather than manually patching arrays.

## Edge Cases

- **User not authenticated** → `useAuth()` returns null user. Return early with empty arrays and `isLoading: false`. Do not attempt any Supabase queries.
- **Partner not found** → if the profiles query returns zero rows, set `error` and return without crashing. This can happen in dev with a single-user setup.
- **Redeeming an already-redeemed coupon** → Supabase will succeed but the status update is a no-op if you add a DB constraint. Guard in the hook: if `coupon.status !== 'active'`, throw with message `"Coupon is not redeemable"`.
- **Approving/rejecting a non-pending coupon** → guard: if `coupon.status !== 'pending_approval'`, throw with message `"Coupon is not awaiting approval"`.
- **Expired coupons** → `expires_at` is a future timestamp. The hook does not auto-expire. Expiry is handled by a Supabase cron job (separate task). The hook simply displays the `expired` status as returned.
- **Realtime fires before initial fetch completes** → the `fetchCoupons` function is idempotent. Calling it multiple times in quick succession is safe — just results in slightly redundant queries.
- **Component unmounts before fetch resolves** → use an `isMounted` ref pattern or check in the setState callback to avoid setting state on unmounted components.
- **Reveal surprise on already-revealed coupon** → guard: if `coupon.surprise_revealed === true`, return early without error.

## Do NOT

- Don't import the Supabase server client — always use the browser client from `@/lib/supabase/client`
- Don't manually patch `myCoupons` or `receivedCoupons` arrays after mutations — always call `refreshCoupons()`
- Don't hardcode user IDs or partner IDs — always derive from auth context and profiles query
- Don't skip the realtime cleanup — unsubscribe in the `useEffect` return
- Don't throw errors to components — set the `error` string state instead
- Don't filter surprise coupons client-side — trust RLS to exclude them from `receivedCoupons`
- Don't add pagination yet — load all coupons for now (coupon count will remain low per user)
- Don't call `sendPushNotification` if partner ID is null — guard before every notification call
