## Why This Exists

Most apps ask for push notification permission the moment you open them. It's aggressive. It feels transactional. Hayah is different — it's a shared intimate space for two people. The first time Yara opens the app after logging in, she shouldn't be immediately hit with a browser dialog asking for system permission. She should have a second to breathe.

This prompt exists to earn that permission. It shows up 3 seconds after login — warm, calm, and honest. It explains what notifications are actually for: knowing when your partner sends you something that matters. If the user says "maybe later," the app respects that completely and doesn't ask again that session. No dark patterns. No guilt. Just a gentle offer.

The component also handles the graceful failure case: if the browser shows its own block/allow dialog and the user blocks it, Hayah should respond with warmth, not frustration.

---

## User Story

As Yahya or Yara, after I log in, I see a soft modal appear a few seconds later asking if I'd like to enable notifications. It explains what I'll get — a warm nudge when my partner sends me something special. I can tap "Enable Notifications" to allow it, or tap "Maybe Later" to skip it without any pressure. Either way, the modal disappears cleanly and doesn't come back this session.

## Full UX Flow

1. **User completes login** → the page they land on mounts this component
2. **3 seconds pass** (setTimeout, not blocking) → modal fades and scales into view
3. **Check condition first**: only show if `Notification.permission === 'default'`
   - If already `'granted'` → don't show, call `onComplete()` immediately
   - If already `'denied'` → don't show, call `onComplete()` immediately
   - If `sessionStorage.getItem('push-prompt-dismissed')` is set → don't show, call `onComplete()` immediately
4. **Modal is now visible** — user sees the bell icon, title, body copy, two buttons
5. **User taps "Enable Notifications":**
   - Calls `subscribeToPush()` from `@/lib/push-service` (T402)
   - If browser shows its native allow/block dialog and user allows → permission granted, modal closes, calls `onComplete()`
   - If browser dialog is blocked/denied → show gentle fallback message inside the modal: "You can enable notifications later in your browser settings." with a single "Got it" dismiss button
   - On any outcome, set `sessionStorage.setItem('push-prompt-dismissed', '1')`
6. **User taps "Maybe Later":**
   - Set `sessionStorage.setItem('push-prompt-dismissed', '1')`
   - Animate modal out (scale down, fade)
   - Call `onComplete()`
7. **User taps backdrop** → same behavior as "Maybe Later"

## Implementation Notes

- `"use client"` directive required — uses `useEffect`, `useState`, browser APIs
- Check `typeof window !== 'undefined'` and `'Notification' in window` before reading `Notification.permission` — SSR safety
- Use `useEffect` with a `setTimeout` (3000ms) to trigger the delayed appearance; clear the timeout on unmount
- Push service import: `import { subscribeToPush } from '@/lib/push-service'` (from T402)
- The fallback "blocked" state should replace the button row — don't stack a new modal on top, just swap the content inside the same modal
- Use `useState` for: `isVisible`, `isLoading` (while subscribeToPush is running), `isBlocked` (browser denied after prompt)
- Keep the component file under 150 lines — split logic into small named handlers: `handleEnable`, `handleDismiss`, `handleBackdropClick`

## Edge Cases

- **Notification API not available** (some Android WebViews, older browsers) → check `'Notification' in window` — if absent, skip silently and call `onComplete()`
- **Permission already granted** → skip prompt entirely, call `onComplete()` without any delay
- **Permission already denied** → skip prompt entirely, call `onComplete()` — do not re-ask, don't show the modal
- **Session storage unavailable** (private browsing edge cases) → wrap `sessionStorage` access in try/catch; if it throws, treat as if not dismissed (show the prompt)
- **subscribeToPush throws** → catch the error, show a gentle fallback message (same as browser-blocked state), do not crash the component
- **User dismisses during the 3-second delay** (e.g., navigates away) → `useEffect` cleanup clears the timeout so it never fires on an unmounted component
- **Modal opens on a very small screen (320px)** → `max-w-sm` with `mx-4` padding ensures the modal never touches the screen edges; text must wrap gracefully

## Do NOT

- Don't show the modal before the 3-second delay — the delay is intentional UX breathing room
- Don't re-show the prompt after dismissal — sessionStorage check is the gate; respect it
- Don't use `alert()` or native `confirm()` anywhere in this component
- Don't hardcode color hex values — use Tailwind theme tokens only
- Don't navigate the user or refresh the page after permission is granted — stay in place
- Don't add a close (X) button — "Maybe Later" is the escape hatch, keep the UI clean
- Don't show this on non-authenticated routes — the parent (T409) is responsible for gating, but this component should still be safe to conditionally mount
