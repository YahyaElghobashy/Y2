## Why This Exists

Challenges between Yahya and Yara have real stakes ŌĆö CoYYns on the line. When one person believes they've won, there needs to be a moment of declaration, a moment of confirmation, and a moment of transfer. That sequence is what makes the challenge system feel real rather than decorative. Without this flow, challenges are just shared to-do items with a timer. With it, they're genuine contests with consequence.

This component orchestrates the entire resolution lifecycle: the winner taps "Claim Win", the partner is notified and chooses to confirm or dispute, and if confirmed, CoYYns move hands. The flow mirrors the coupon approval pattern (T604) but with financial stakes and adversarial logic ŌĆö both sides have agency, and either side can steer the outcome.

---

## User Story

As Yahya or Yara, when I believe I've completed a challenge and won, I want to formally claim my win so my partner can verify it. I want the process to feel weighty ŌĆö a real declaration, not a tap I might do by accident. If my partner agrees, I want to see the CoYYns land in my account immediately with a satisfying confirmation. If they dispute it, I want the challenge to reactivate so we can finish what we started.

As the partner receiving a win claim, I want to clearly see what is being claimed, have a deliberate choice to confirm or dispute, and understand exactly what CoYYns will transfer before I act.

## Full UX Flow

### Claimant Side (the person claiming the win)

1. **ChallengeCard (T705) shows "Claim Win" button** when challenge status is `active` and deadline has not passed
2. **User taps "Claim Win"** ŌåÆ `ClaimWinDialog` opens
3. **Dialog content:**
   - Title: "Claim Your Win"
   - Body: "You're claiming you won '[challenge title]'. [Partner] will need to confirm this."
   - Displays stakes amount: "+[stakes] CoYYns if confirmed"
   - Two buttons: "Yes, I Won" (copper filled) and "Cancel" (ghost)
4. **User taps "Yes, I Won":**
   - Optimistic UI: button becomes a spinner
   - Supabase update: `challenge.claimed_by = currentUserId`, `challenge.status = 'pending_resolution'`
   - Push notification fired to partner (T402): "­¤Åå [User] claims they won: [title]! Do you agree?"
   - Dialog transitions to waiting state: "Waiting for [Partner] to confirm..."
   - Waiting state shows a pulsing avatar/icon and the message, with a "Cancel Claim" link in case of mistake
5. **User taps "Cancel Claim"** (optional rollback):
   - Supabase update: `challenge.claimed_by = null`, `challenge.status = 'active'`
   - Dialog closes

### Partner Side (the person confirming or disputing)

1. **ChallengeCard shows "[User] claims they won"** banner when `challenge.status = 'pending_resolution'` and `challenge.claimed_by !== currentUserId`
2. **Partner taps the card** ŌåÆ `ConfirmResultDialog` opens
3. **Dialog content:**
   - Title: "[User] claims they won"
   - Body: Shows the challenge title and description
   - Stakes line: "This will transfer [stakes] CoYYns from you to [User]"
   - Current CoYYn balance shown if it's below the stakes amount (warn: partial transfer)
   - Two buttons: "Confirm Win" (success green #7CB67C, filled) and "Dispute" (error red #C27070, outline)
4. **Partner taps "Confirm Win":**
   - Supabase update: `challenge.status = 'completed'`, `challenge.winner_id = challenge.claimed_by`, `challenge.completed_at = now()`
   - CoYYns transfer (see Transfer Logic section below)
   - Push notification to winner (T402): "­¤Äē [Partner] confirmed your win! +[stakes] CoYYns"
   - Dialog closes; ChallengeCard updates to completed state
5. **Partner taps "Dispute":**
   - Supabase update: `challenge.status = 'active'`, `challenge.claimed_by = null`
   - Push notification to claimant (T402): "­¤żö [Partner] disputed your claim. Challenge continues!"
   - Dialog closes; ChallengeCard returns to active state

## CoYYns Transfer Logic

The transfer happens inside the `handleConfirmWin` function. Use `useCoyyns` (T302) for both operations.

```ts
async function handleConfirmWin(challenge: Challenge, currentUserId: string) {
  const winnerId = challenge.claimed_by!
  const loserId = currentUserId
  const stakes = challenge.stakes

  // Determine actual transfer amount (partial if loser can't afford full stakes)
  const loserBalance = await getCoyynsBalance(loserId)
  const transferAmount = Math.min(stakes, loserBalance)
  const isPartial = transferAmount < stakes

  // Update challenge record first
  await supabase
    .from('challenges')
    .update({
      status: 'completed',
      winner_id: winnerId,
      completed_at: new Date().toISOString(),
      actual_transfer: transferAmount,
    })
    .eq('id', challenge.id)

  // Transfer CoYYns
  await addCoyyns(
    winnerId,
    transferAmount,
    `Won challenge: ${challenge.title}`,
    'challenge'
  )
  await spendCoyyns(
    loserId,
    transferAmount,
    `Lost challenge: ${challenge.title}`,
    'challenge'
  )

  // Send notification ŌĆö note partial transfer if applicable
  const notifBody = isPartial
    ? `­¤Äē [Partner] confirmed your win! +${transferAmount} CoYYns (partial ŌĆö they only had ${transferAmount})`
    : `­¤Äē [Partner] confirmed your win! +${transferAmount} CoYYns`

  await sendPushNotification(winnerId, notifBody)
}
```

If the loser's balance is zero, `transferAmount = 0`, no CoYYns move, but the challenge still resolves as completed. The shortfall is noted in `challenge.actual_transfer` vs `challenge.stakes` for any future audit.

## Realtime Updates

Both sides should react to status changes without requiring a page refresh. Use Supabase realtime on the `challenges` table filtered by challenge ID. This mirrors the coupon approval pattern.

```ts
useEffect(() => {
  const channel = supabase
    .channel(`challenge-${challenge.id}`)
    .on(
      'postgres_changes',
      {
        event: 'UPDATE',
        schema: 'public',
        table: 'challenges',
        filter: `id=eq.${challenge.id}`,
      },
      (payload) => {
        onChallengeUpdate(payload.new as Challenge)
      }
    )
    .subscribe()

  return () => { supabase.removeChannel(channel) }
}, [challenge.id])
```

The claimant's waiting dialog should automatically transition to "Confirmed!" when the realtime update fires with `status = 'completed'`.

## Implementation Notes

- `"use client"` directive required ŌĆö uses hooks, animations, and realtime
- Use shadcn/ui `Dialog` as the base for both dialogs
- All loading states use a Framer Motion spinner or the LoadingSkeleton (T104) pulse pattern ŌĆö no native browser spinners
- The waiting state transition (from confirm button to pulsing avatar) should use `AnimatePresence` with a crossfade: exit old content at opacity 0, enter new content at opacity 1, 200ms
- Use `cn()` from `@/lib/utils` for all conditional classes
- Never hardcode `#7CB67C` or any hex ŌĆö use `bg-success`, `text-error`, `text-accent-primary` etc. from the design token system
- The `handleConfirmWin` function must be async and wrapped in a `try/catch` ŌĆö surface errors with a toast, not a full page error
- Use `useCurrentUser()` or equivalent to determine `currentUserId` and `partnerId`
- `addCoyyns` and `spendCoyyns` come from `useCoyyns` (T302) ŌĆö do not write raw Supabase inserts for the CoYYns ledger
- All user-facing strings (button labels, dialog titles, notification bodies) go in `lib/i18n/` following existing patterns

## Edge Cases

- **Loser has fewer CoYYns than stakes:** Transfer whatever is available (`Math.min(stakes, balance)`), mark `challenge.actual_transfer` in the DB, note partial in the winner's push notification
- **Loser has zero CoYYns:** Challenge still completes, no transfer occurs, no error shown to user ŌĆö just resolves cleanly
- **Claimant closes dialog while waiting:** Dialog closes but challenge stays in `pending_resolution` ŌĆö the ChallengeCard will continue showing the waiting state on re-open
- **Both users open ConfirmResultDialog at same time:** Not possible ŌĆö only the non-claimant sees the confirm/dispute UI. Guard with `challenge.claimed_by !== currentUserId`
- **Challenge expires while pending_resolution:** Treat the same as expired (no transfer). A pg_cron job or a client-side deadline check should override `pending_resolution` ŌåÆ `expired` if deadline passes
- **Push notification delivery fails:** The CoYYns transfer still completes ŌĆö notification failure must not block the DB transaction. Use fire-and-forget for push calls
- **Network error during confirm:** Show a toast error, re-enable the confirm button, do not partially apply the transfer ŌĆö the Supabase update and CoYYns calls should be sequenced so failure at any step leaves state consistent

## Do NOT

- Do not write raw CoYYns ledger inserts ŌĆö always use `addCoyyns` / `spendCoyyns` from `useCoyyns` (T302)
- Do not hardcode color hex values ŌĆö use Tailwind design tokens only
- Do not block the transfer on push notification success ŌĆö notifications are fire-and-forget
- Do not allow the claimant to see the confirm/dispute dialog ŌĆö guard with `claimed_by !== currentUserId`
- Do not skip the confirmation step on either side ŌĆö both dialogs are required; no single-tap resolution
- Do not use `router.push()` or page navigation inside these dialogs ŌĆö they are overlays, not routes
- Do not show a full-page error on transfer failure ŌĆö use a toast and re-enable the button
- Do not write a custom pg_cron handler here ŌĆö expired challenge handling is a separate concern (note it as future work)
