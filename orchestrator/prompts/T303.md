## Why This Exists

CoYYns are the love language of the app — a micro-economy Yahya and Yara have built together. Earning them feels like receiving a gift. Spending them feels like choosing something meaningful. The wallet is the moment they see the score: not a score of competition, but of investment in each other.

This component is the primary CoYYns UI. It's the first thing they see on the /us page and on the home dashboard. It needs to feel like opening a wallet — warm, tactile, satisfying. When a balance changes, it shouldn't just update — it should count, visibly, so the change feels earned. This is not a number. It's a story of how much effort has gone into this relationship.

---

## User Story

As Yahya or Yara, when I open the app I want to see my CoYYns balance at a glance in a warm, inviting card. I want to know how much I've earned and spent over time. I want to see my partner's balance so we feel connected. And I want clear, easy buttons to add or spend CoYYns without digging through menus.

---

## Full UX Flow

1. **Component mounts** → calls `useCoyyns()` internally to fetch wallet data
2. **Loading state** → the balance area shows a subtle shimmer placeholder (not a spinner), stats show dashes
3. **Data loads** → balance number animates in using a count-up from 0 to the actual value on first render
4. **Balance changes** (e.g. after adding or spending CoYYns):
   - Number animates from old value to new value — count up if increase, count down if decrease
   - The animation uses Framer Motion's `animate` on a `motion.span` — smooth, 600ms, ease-out, no bounce
5. **Stats row** shows Lifetime Earned and Lifetime Spent as formatted numbers with comma separators
6. **Partner row** shows the partner's name and balance — fetched from `useCoyyns()` which has access to both users' data; viewer's own name is inferred from `useAuth()` (T204)
7. **Tap "Add"** → calls `onAdd()` prop, which the parent uses to open `AddCoyynsForm`
8. **Tap "Spend"** → calls `onSpend()` prop, which the parent uses to open `SpendCoyynsForm`
9. **Error state** → if `useCoyyns()` returns an error, show a soft error message inside the card: "Couldn't load wallet" with a retry option (call refetch from the hook)

## Component API

```tsx
// File: src/components/relationship/CoyynsWallet.tsx

interface CoyynsWalletProps {
  onAdd?: () => void      // opens AddCoyynsForm (provided by parent)
  onSpend?: () => void    // opens SpendCoyynsForm (provided by parent)
  className?: string
}

export function CoyynsWallet({ onAdd, onSpend, className }: CoyynsWalletProps) { ... }
```

The component is self-contained for data. It calls `useCoyyns()` internally and does not accept balance data as props. This keeps the parent clean and makes the wallet reusable anywhere in the app without the parent needing to know how CoYYns data works.

## Implementation Notes

- `"use client"` directive required — uses Framer Motion, hooks, and interactivity
- Import `useCoyyns` from `@/hooks/use-coyyns` (T302 output)
- Import `useAuth` from `@/hooks/use-auth` (T204) to get current user's id — determines which balance is "mine" and which is "partner's"
- **Animated balance counter:** use `useMotionValue` and `useTransform` from Framer Motion, or a simpler approach: store the displayed number in a `useState`, then use `animate()` from `"framer-motion"` (the imperative API) to tween from old to new value on change:
  ```tsx
  const count = useMotionValue(0)
  const rounded = useTransform(count, (v) => Math.round(v).toLocaleString())
  useEffect(() => {
    animate(count, balance, { duration: 0.6, ease: "easeOut" })
  }, [balance])
  // render: <motion.span>{rounded}</motion.span>
  ```
- **Number formatting:** always use `.toLocaleString()` for comma-separated display (1250 → "1,250")
- **Divider lines:** use a `<hr>` or `<div>` with `border-t border-[rgba(44,40,37,0.08)]` — do not use shadcn Separator here, a plain div is sufficient
- **Loading state:** render three lines of `bg-accent-soft/30 rounded animate-pulse` placeholders instead of the number, stats, and partner rows — do not show the icon or CoYYns label until loaded
- **Partner name:** derive from `useCoyyns()` hook data or `useAuth()` — the partner is whichever of {Yahya, Yara} the current user is not. Render as `"{PartnerName} has {X} CoYYns"`
- **Button behavior:** if `onAdd` is undefined, the Add button is still rendered but does nothing on press (no disabled state — it's always available visually). Same for `onSpend`.
- Use `cn()` from `@/lib/utils` for all conditional class logic
- Wrap the entire card in `motion.div` with `initial={{ opacity: 0, y: 8 }}` and `animate={{ opacity: 1, y: 0 }}` — `duration: 0.35`, `ease: "easeOut"` — no bounce, no spring

## Edge Cases

- **Balance is 0** → show "0" not blank. The coin icon still renders. Buttons still work.
- **Very large balance** (e.g. 100,000+) → the mono font handles wide numbers, but cap the font size at 40px regardless. Do not shrink dynamically — if numbers are very wide, the card naturally expands vertically.
- **Balance animates on unmount/remount** → on first mount always animate from 0 to current value so it feels like the wallet is "opening". On subsequent changes, animate from previous value.
- **Partner has 0 CoYYns** → render "Yara has 0 CoYYns" — never hide this row.
- **useCoyyns returns loading: true** → show shimmer placeholders, do not flash 0 before data arrives.
- **useCoyyns returns error** → render a single centered line inside the card: `"Couldn't load wallet"` in `text-secondary` at 13px, with a small `"Retry"` text button below it in `accent-primary`.
- **Lifetime Earned and Spent values are equal** → no special treatment, just render the numbers as-is.
- **Component is used in a narrow context** (e.g. 320px screen) → the two buttons must use `flex-1` so they share available width equally and never overflow.

## Do NOT

- Don't accept balance, earned, or spent as props — data always comes from `useCoyyns()` internally
- Don't hardcode color hex values — use Tailwind design tokens from `lib/theme.ts`
- Don't use a CSS counter or keyframe animation for the balance count-up — use Framer Motion's imperative `animate()` API only
- Don't show a full-page spinner — shimmer placeholders only, scoped to the card content
- Don't add confetti or celebration effects here — those belong in the transaction confirmation flow, not the wallet display
- Don't make the partner row clickable or navigable — it's informational only
- Don't use `left` / `right` directional Tailwind classes — use logical properties (`ps`, `pe`, `ms`, `me`) for RTL readiness
- Don't add any CoYYns history list or transaction log to this component — that belongs in a separate component
