## Why This Exists

Yara takes hormonal contraceptive pills on a 21-active + 7-break schedule. That creates a fully predictable 28-day cycle. Knowing which day she's on — and what phase is coming — is genuinely useful information. Yahya needs to know when PMS is approaching so he can show up better. He needs to know when her period is likely so plans can be made with care. This hook is the engine that turns a single start date into all of that actionable insight.

This is one of the most intimate features in the app. It belongs only to Yahya. Yara's view of the health module is completely separate. The data must be guarded at two levels: the database (RLS from T501) and right here in this hook. If either layer is wrong, private health information is exposed. Both layers must be correct.

---

## User Story

As Yahya, when I open the health section of the app, I see accurate, real-time information about the current cycle day, phase, and what's coming next — all derived automatically from the pill start date set in config. I never need to manually track or count days. The app knows, and it tells me in a way I can act on.

## Full Calculation Flow

1. **Hook initializes** → calls `useAuth()` to get the current user's profile
2. **Yahya-only guard runs immediately** → if the current user is not the cycle owner, return all nulls and empty arrays — no data is fetched, no calculations are run
3. **Config is fetched** from `cycle_config` table via Supabase client (T202)
4. **Secondary guard** → if `profile?.id !== config?.owner_id`, return nulls — this handles the edge case where RLS somehow passes but ownership doesn't match
5. **Calculations run** using `pill_start_date` and today's date:
   - `daysSinceStart = differenceInDays(today, parseISO(pill_start_date))`
   - `totalCycleDays = active_days + break_days` (typically 28)
   - `currentDay = (daysSinceStart % totalCycleDays) + 1` (1-indexed, range 1–28)
   - `phase = currentDay <= active_days ? 'active' : 'break'`
   - `daysUntilBreak = phase === 'active' ? (active_days - currentDay + 1) : null`
   - `daysUntilActive = phase === 'break' ? (totalCycleDays - currentDay + 1) : null`
   - `isPMSWindow = phase === 'active' && daysUntilBreak <= pms_warning_days`
   - `isPeriodLikely = phase === 'break'`
   - `nextPeriodDate = addDays(parseISO(pill_start_date), daysSinceStart + (daysUntilBreak ?? 0))`
6. **Logs are fetched** from `cycle_logs` table — only entries where `date >= cycleStartDate` for the current 28-day window
7. **Return object** is assembled and returned to consumers

## Implementation Notes

- `"use client"` is not a directive on hooks — but all consumers of this hook must be client components
- Use `useAuth()` from T204 to get `profile` — this is the source of user identity
- Use the Supabase browser client from T202 (`createBrowserClient`) for all DB calls
- Use `date-fns` functions (`differenceInDays`, `addDays`, `parseISO`, `startOfDay`) — already in `package.json`, do not install anything new
- Wrap all Supabase calls in try/catch; surface errors via the `error` field in the return object
- Use `useState` for `config`, `cycleLogs`, `isLoading`, `error`
- Use `useEffect` to trigger initial fetch — depend on `profile?.id` so it re-fetches if user changes
- `refreshCycle` is a stable `useCallback` that re-runs the fetch sequence
- `updateConfig` calls `supabase.from('cycle_config').upsert(...)`, then calls `refreshCycle()`
- `addLog` calls `supabase.from('cycle_logs').insert(...)`, then calls `refreshCycle()`
- For `cycleLogs`, fetch only the current cycle window: entries where `date >= currentCycleStartDate`
- `currentCycleStartDate` is derived by: `addDays(parseISO(pill_start_date), Math.floor(daysSinceStart / totalCycleDays) * totalCycleDays)`

## CRITICAL: Yahya-Only Guards

This section is the most important part of this task. Read it twice.

The cycle data is private health information. It must never be readable by the wrong user. There are two independent guard layers:

**Layer 1 — Database RLS (from T501):** The `cycle_config` and `cycle_logs` tables have Row Level Security policies that only allow `SELECT` when `owner_id = auth.uid()`. This means Yara's Supabase session will simply receive zero rows when she queries these tables. The DB will not leak data regardless of what the client does.

**Layer 2 — Hook-level identity guard (this task):** At the very top of the hook's logic — before any fetch is initiated — check the current user's identity. If `profile` is null, or if after fetching config the `profile.id !== config.owner_id`, the hook must return the null/empty state immediately.

The implementation order for the guard:
```ts
// 1. Get current user from auth context
const { profile } = useAuth()

// 2. In the fetch function, first check if profile exists
if (!profile) {
  setIsLoading(false)
  return  // no fetch, return null state
}

// 3. After fetching config, verify ownership matches
if (config && config.owner_id !== profile.id) {
  // RLS should have prevented this, but check anyway
  setConfig(null)
  setCycleLogs([])
  setIsLoading(false)
  return
}
```

The null return shape when the guard fires:
```ts
{
  config: null,
  currentDay: null,
  phase: null,
  daysUntilBreak: null,
  daysUntilActive: null,
  isPMSWindow: false,
  isPeriodLikely: false,
  nextPeriodDate: null,
  cycleLogs: [],
  isLoading: false,
  error: null,
  updateConfig: async () => {},  // no-op
  addLog: async () => {},         // no-op
  refreshCycle: async () => {},   // no-op
}
```

The no-op actions are important — consumers (T503, T504) may call `addLog` or `updateConfig` without knowing the guard fired. The no-ops ensure those calls silently do nothing rather than throwing.

## Edge Cases

- **`pill_start_date` is in the future** → `daysSinceStart` will be negative. Guard against this: if `daysSinceStart < 0`, set `currentDay = null`, `phase = null`, return null state with an appropriate `error` message
- **`active_days` or `break_days` is zero** → guard against division by zero in the modulo calculation; log a console error and return null state
- **No config row exists yet** → `config` will be null, all calculated fields remain null, `isLoading` becomes false — this is valid state (user hasn't set up cycle yet), not an error
- **Multiple config rows** → take the first row only; the DB schema from T501 should enforce one row per owner, but be defensive with `.single()` or `.limit(1)`
- **Log fetch fails but config fetch succeeds** → surface the error via the `error` field, but still return the calculated fields from config — partial data is better than nothing
- **User logs out mid-session** → `profile` becomes null, `useEffect` re-runs, guard fires, all state resets to null/empty

## Do NOT

- Do not fetch data if the Yahya-only guard fires — skip the Supabase calls entirely
- Do not throw errors from `updateConfig` or `addLog` when the guard has fired — use no-ops
- Do not use `left`/`right` CSS properties anywhere — this is a utility hook with no UI, but keep it in mind for any future additions
- Do not install `date-fns` — it is already in `package.json`
- Do not use `new Date()` directly for "today" — use `startOfDay(new Date())` for consistent day-boundary comparisons
- Do not store calculated values in state — derive them during render from `config` and today's date so they are always fresh
- Do not expose raw Supabase error objects — map them to plain `string` messages in the `error` field
- Do not make the hook responsible for knowing which user is "Yahya" by name or hardcoded ID — compare `profile.id` against `config.owner_id` from the DB row
