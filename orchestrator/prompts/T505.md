## Why This Exists

Cycle tracking only becomes meaningful when patterns are visible. A list of logged days tells you nothing at a glance — a calendar tells you everything. Yahya needs to open the health screen and immediately see the shape of the month: which days are active phase, which days are PMS window, when the next period break lands. Planning ahead — a trip, a dinner, an important conversation — all of it becomes easier when the calendar is right there.

This is a YAHYA-ONLY feature. The cycle calendar is deeply personal and is never visible to Yara's session. It renders only when a valid cycle config exists.

---

## User Story

As Yahya, when I open the health screen, I see a full month calendar where every day is color-coded by cycle phase. I can navigate backwards to see past months and confirm the pattern is tracking correctly, or forward to anticipate what's coming. Tapping any day shows me the cycle day number, the phase label, and any log note I've added. If I haven't logged a note for that day, I can add one right there.

---

## Full UX Flow

1. **Calendar mounts** → shows the current calendar month by default
2. **Each day cell** is color-coded based on the calculated cycle phase for that date:
   - Active phase days: soft accent background (`accent-soft` #E8D5C0)
   - PMS window days: soft warning background (#FFF8E8)
   - Break/period days: soft error background (#FDE8E8)
   - Days outside any defined phase: plain white background
3. **Today's cell** has a `accent-primary` (#C4956A) ring/border to anchor the user in the present
4. **Days with a log entry** have a small filled dot (4px, `accent-primary`) centered below the day number
5. **Future days** render at reduced opacity (60%) to signal they are projected, not confirmed
6. **Tap a day cell** → a detail section expands below the calendar (not a modal, not a popover — inline expansion for mobile comfort):
   - Shows "Day 14 of 28" in JetBrains Mono
   - Shows phase label ("Active Phase", "PMS Window", "Period / Break") in DM Sans
   - If a log entry exists: shows mood, symptoms, and notes for that date
   - If no log entry: shows an "Add note" button that triggers the log entry flow
7. **Month navigation** — left chevron and right chevron flank the month/year label in the header:
   - Tapping arrows animates the calendar grid out (slide + fade) and the new month in
   - Month label updates: "March 2026", "April 2026", etc.
8. **No cycle config** (useCycle returns null) → component returns null entirely, renders nothing

## Implementation Notes

- `"use client"` directive required (uses hooks, Framer Motion, local state)
- Import `useCycle` from `@/hooks/use-cycle` (T502) — if it returns null config, return null immediately
- Use `date-fns` for all calendar math: `startOfMonth`, `endOfMonth`, `eachDayOfMonth`, `getDay`, `isSameDay`, `isToday`, `isFuture`, `format`
- Do not install date-fns if it is already in package.json — check first
- Local state: `viewMonth: Date` (default to current month), `selectedDay: Date | null`
- Build an array of day objects for the grid: `{ date: Date, phase: CyclePhase | null, hasLog: boolean }`
- Phase is derived by calling the phase calculation logic from useCycle for each day in the month — the hook should expose a helper like `getPhaseForDate(date: Date): CyclePhase | null`
- Calendar grid offset: the first day of the month may not be Sunday — pad the start of the grid with empty cells equal to `getDay(startOfMonth(viewMonth))`
- Month transition animation: `AnimatePresence` with `mode="wait"`, outgoing grid slides left and fades, incoming slides from right and fades in — 200ms, ease-out
- Selected day detail section: `AnimatePresence` with `mode="wait"`, animate in with `y: 8 → 0` and `opacity: 0 → 1`, 180ms ease-out
- Use `motion.div` with `whileTap={{ scale: 0.96 }}` on each day cell for tactile feedback
- Use `cn()` from `@/lib/utils` for all conditional class merging
- The legend sits below the grid, above the detail section, as a horizontal row of three labeled dots

## Edge Cases

- **Month with 28 days (February non-leap)** → grid must not leave extra blank rows; render only the rows needed
- **Month starts on Saturday** → first row has 6 empty cells before day 1; grid must handle this cleanly
- **Cycle config not yet set** (useCycle returns null config) → return null — no skeleton, no placeholder, nothing
- **No log entries for any day** → dots simply don't render; calendar remains useful from phase colors alone
- **Future month navigation** → future days render at 60% opacity; phase projections are shown as normal because they are calculated from the fixed cycle config
- **Cycle day overflow** → if the cycle is 28 days and the user navigates 3 months forward, the phase calculation wraps correctly using modulo arithmetic in the useCycle hook
- **Selected day is in a padded empty cell** → empty cells are not tappable; clicking one does nothing
- **Rapid month navigation** → AnimatePresence `mode="wait"` prevents animation stacking; only one transition runs at a time

## Do NOT

- Don't use a third-party calendar library — build the grid from date-fns primitives
- Don't open a modal or drawer for the day detail — expand it inline below the grid
- Don't hardcode any hex color values — use Tailwind theme tokens and CSS variables
- Don't show the calendar to Yara — the null-config guard must be the first line of logic
- Don't paginate or lazy-load months — generate the full grid for the viewed month synchronously
- Don't use `left`/`right` CSS properties — use logical properties (`inset-inline-start`, `ms-`, `me-`) for RTL readiness
- Don't skip the legend — it is required for the component to be self-explanatory
