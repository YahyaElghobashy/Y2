## Why This Exists

CoYYns is the shared reward currency between Yahya and Yara. It needs a reliable, reactive data layer that any component in the CoYYns feature can consume without duplicating Supabase calls, without managing loading state individually, and without risking stale balances after a mutation. This hook is that single source of truth.

Every component in the CoYYns feature — the wallet display, the badge, the add form, the spend form, the history list — will call `useCoyyns()`. None of them should touch Supabase directly. This hook owns all reads, all writes, and the live subscription that keeps balances in sync when the partner makes a transaction on their device simultaneously.

---

## User Story

As Yahya or Yara, when I open the CoYYns section of the app, my current balance loads immediately, my partner's balance is visible, and my recent transactions appear below. When I earn or spend CoYYns, the balance updates on screen the moment the transaction completes. If my partner earns CoYYns on their device while I have the app open, my view of their balance updates automatically without me refreshing. If I try to spend more CoYYns than I have, I'm told immediately — nothing is written to the database.

---

## Full UX Flow

1. **Component mounts** → `useCoyyns()` is called. `isLoading` is immediately `true`.
2. **Three parallel fetches fire:**
   - Fetch current user's `coyyns_wallets` row by `user_id`
   - Fetch partner's `coyyns_wallets` row by partner's `user_id` (from `useAuth().partner`)
   - Fetch the last 50 `coyyns_transactions` rows for the current user, ordered by `created_at DESC`
3. **All three resolve** → `isLoading` becomes `false`, `wallet`, `partnerWallet`, and `transactions` are populated
4. **Realtime subscription opens** on the `coyyns_wallets` table. When either wallet row changes, update the corresponding state value in place.
5. **User taps "Add CoYYns"** → calls `addCoyyns(amount, description, category?)`:
   - Inserts a row into `coyyns_transactions` with `type: 'earn'`, positive `amount`
   - On success: calls `refreshWallet()` to re-fetch the wallet row
6. **User taps "Spend CoYYns"** → calls `spendCoyyns(amount, description, category?)`:
   - Checks `wallet.balance >= amount` before touching the database
   - If insufficient: sets `error` to a human-readable message, returns early, nothing is written
   - If sufficient: inserts a row into `coyyns_transactions` with `type: 'spend'`, negative `amount`
   - On success: calls `refreshWallet()`
7. **Any mutation error** → `error` is set to the Supabase error message string. Caller components surface this.
8. **Component unmounts** → Supabase realtime subscription is removed via cleanup in `useEffect` return

## Hook API

```ts
// File: src/lib/hooks/use-coyyns.ts

type UseCoyynsReturn = {
  wallet: CoyynsWallet | null          // current user's wallet
  partnerWallet: CoyynsWallet | null   // partner's wallet (read-only display)
  transactions: CoyynsTransaction[]    // last 50 for current user, desc order
  isLoading: boolean
  error: string | null
  addCoyyns: (amount: number, description: string, category?: string) => Promise<void>
  spendCoyyns: (amount: number, description: string, category?: string) => Promise<void>
  refreshWallet: () => Promise<void>   // re-fetches wallet + transactions
}

export function useCoyyns(): UseCoyynsReturn
```

## Implementation Notes

- `"use client"` is NOT on the hook file itself — hooks are plain TypeScript. Components that call this hook must be client components.
- Import the Supabase browser client from `@/lib/supabase/client` (established in T202).
- Import `useAuth` from `@/lib/providers/AuthProvider` (established in T204) to get `user` and `partner`.
- Return early with no-ops if `user` is null — the hook must be safe to call before auth resolves.
- The three initial fetches should run in parallel using `Promise.all` inside the initial `useEffect`.
- `refreshWallet` re-fetches the user's wallet row and the last 50 transactions. It does not re-fetch the partner wallet (realtime handles partner updates).
- Realtime subscription: use `supabase.channel('coyyns_wallets_changes').on('postgres_changes', ...)`. Filter on `user_id=eq.${user.id}` OR `user_id=eq.${partner.id}` — update the correct state entry when a change arrives.
- The `addCoyyns` and `spendCoyyns` functions must clear `error` to `null` at the start of each call before setting it on failure.
- Use a `category` fallback of `'manual'` if the caller passes no category.
- `amount` passed to `addCoyyns` must be a positive integer. Validate this at the start of the function and set `error` if not.
- `amount` passed to `spendCoyyns` must also be positive. The hook inserts it as negative: `amount: -Math.abs(amount)`.

---

## Edge Cases

- **Auth not yet resolved** (`user` is null) → return the zero/null state immediately, do not fire Supabase calls. When `user` becomes available, the `useEffect` dependency array will re-run.
- **Partner is null** (solo session, edge case) → `partnerWallet` stays `null`. No error is set — this is a valid state.
- **Wallet row does not exist for user** → this should not happen if T301's `on_new_user` trigger ran correctly, but handle gracefully: `wallet` stays `null`, `error` is not set, components must handle null wallet.
- **Insufficient balance** → `spendCoyyns` sets `error` to `"Insufficient CoYYns balance"` and returns. No Supabase write occurs.
- **Supabase write fails mid-transaction** → set `error` to the Supabase error message. The wallet state is NOT optimistically updated, so no rollback is needed.
- **Realtime disconnect** → Supabase handles reconnection internally. The hook does not need custom reconnect logic, but `refreshWallet` can be called manually by the user via a pull-to-refresh gesture (wired up in T303).
- **Rapid back-to-back mutations** → `isLoading` is not used as a mutation guard here (it reflects initial load only). Components rendering the add/spend forms should disable their submit button during in-flight requests using local state.

## What This Connects To

- **T301 (CoYYns DB)** — provides the `coyyns_wallets` and `coyyns_transactions` tables this hook reads and writes
- **T202 (Supabase client)** — provides the browser-side Supabase client instance
- **T204 (AuthProvider)** — provides `user`, `profile`, and `partner` via `useAuth()`
- **T303 (CoyynsWallet display component)** — primary consumer, shows balance and lifetime stats
- **T304 (CoyynsBadge)** — reads `wallet.balance` for compact display in headers or cards
- **T305 (AddCoyynsForm)** — calls `addCoyyns()` on submit
- **T306 (SpendCoyynsForm)** — calls `spendCoyyns()` on submit, surfaces `error` for insufficient balance
- **T307 (CoyynsHistory)** — reads `transactions` array for the scrollable history list

## Do NOT

- Don't call Supabase directly from any CoYYns UI component — this hook is the only data access layer
- Don't use optimistic updates — balance must only update after a confirmed server response
- Don't store `amount` as a string — always a `number`
- Don't hardcode user IDs, table names, or column names outside this hook file
- Don't set `isLoading: true` during mutations — reserve it for the initial page load fetch only
- Don't subscribe to the entire `coyyns_wallets` table without row-level filters — subscribe only to the two known user IDs
- Don't forget to unsubscribe from realtime in the `useEffect` cleanup — memory leaks will cause stale updates
