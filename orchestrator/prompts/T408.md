## Why This Exists

Every notification Yahya sends to Yara, and every one Yara sends to Yahya, should feel like a love letter preserved in amber. The NotificationHistory isn't a log file — it's a record of moments. A trail of "thinking of you" nudges, reminders, and sweet interruptions that make up the quiet texture of their shared life. When Yara wants to revisit that emoji Yahya sent at 2pm on a Thursday, she should find it here. When Yahya wants to know if his "miss you" message actually landed, this is where he looks.

This component surfaces that history in a warm, readable, uncluttered list — split logically between what was sent and what was received, so neither person ever has to wonder what the other has seen.

---

## User Story

As Yahya or Yara, when I open the notifications history, I see a chronological list of all notifications — split into what I sent and what I received. Each row shows the emoji, title, body, a subtle status indicator, and how long ago it was sent. If nothing has been sent yet, I see a gentle empty state that invites me to send my first notification.

---

## Full UX Flow

1. **Component mounts** → calls `useNotifications()` to fetch the notification list
2. **Loading state** → use LoadingSkeleton (T104) with `variant="list"` while data loads
3. **Data resolves** → notifications are split by `sender_id` compared to `currentUser.id` from `useAuth()`
4. **Filter prop** determines what is shown:
   - `'all'` → shows both sent and received sections
   - `'sent'` → shows only notifications where `sender_id === currentUser.id`
   - `'received'` → shows only notifications where `recipient_id === currentUser.id`
5. **Each section** (Sent / Received) renders a section heading, then a `StaggerList` of notification rows
6. **Each notification row** shows: emoji + title, body text, status dot + label + relative time
7. **Tapping a row** (future scope) — no action currently. Rows are display-only.
8. **Empty state** → if the filtered list is empty, render `EmptyState` with Bell icon and "No notifications yet"
9. **`limit` prop** caps the number of rows per section (default 50)

## Implementation Notes

- `"use client"` directive required (uses hooks)
- Import `useNotifications` from `@/hooks/use-notifications` (T402)
- Import `useAuth` from `@/hooks/use-auth` (T204)
- Import `StaggerList` from `@/components/shared/StaggerList` (T103)
- Import `EmptyState` from `@/components/shared/EmptyState` (T105)
- Import `LoadingSkeleton` from `@/components/shared/LoadingSkeleton` (T104)
- Use `formatDistanceToNow` from `date-fns` for relative timestamps — pass `{ addSuffix: true }`
- Derive `sentNotifications` and `receivedNotifications` by comparing each notification's `sender_id` to `currentUser?.id`
- Apply `limit` with `.slice(0, limit)` after filtering, before rendering
- Partner name: use `currentUser.id === notification.sender_id ? partnerName : 'You'` — derive partner display name from auth context or fall back to "Partner"
- Status dot color: map `notification.status` → `'delivered' | 'sent' | 'failed'` to color classes using a small helper or lookup object
- Do not inline hex colors — map to Tailwind classes or CSS variables
- `StaggerList` wraps the array of row cards; each child animates in with staggered fade-up
- When `filter === 'sent'`, render only the Sent section (no "Received" heading)
- When `filter === 'received'`, render only the Received section (no "Sent" heading)
- When `filter === 'all'` and one section is empty, omit that section entirely (no empty heading)
- If both sections are empty (or the single filtered section is empty), render `EmptyState`

## Edge Cases

- **No notifications at all** → render `EmptyState` (Bell icon, "No notifications yet", no CTA needed)
- **Only sent, none received (or vice versa)** with `filter='all'` → show only the populated section, silently omit the empty one
- **`limit` is very small (e.g., 3)** → rows are simply sliced; no "show more" needed at this stage
- **Very long body text** → clamp to 2 lines with `line-clamp-2` to keep rows consistent height
- **`sender_id` equals `currentUser.id`** but filter is `'received'` → that row should NOT appear (strictly filter by direction)
- **`currentUser` is null** (auth not yet resolved) → show LoadingSkeleton, do not attempt to classify notifications
- **Unknown status value** → fall back to yellow "sent" dot rather than crashing

## Do NOT

- Don't hardcode color hex values — use Tailwind tokens or CSS variables
- Don't use `left` / `right` directional properties — use logical equivalents (`ps`, `pe`)
- Don't fetch notifications directly inside this component — delegate entirely to `useNotifications`
- Don't add tap/click handlers on rows at this stage — rows are display-only
- Don't render empty section headings — omit sections with zero items
- Don't add pagination or infinite scroll — `limit` prop is the ceiling for now
- Don't use `new Date()` directly for display — always route through `formatDistanceToNow`
