## Why This Exists

Every feature in Hayah that reads or writes data — coupons, CoYYns, notifications, the period tracker — traces back to one foundational truth: there are exactly two users, and they are connected to each other. This migration creates that truth in the database. Without it, there is no auth, no partner linking, no privacy guarantees, no RLS.

The `profiles` table is not just a convenience wrapper around `auth.users`. It is the identity backbone of the entire system. When Yahya opens the app, his profile row is what tells the app who he is, who his partner is, and what data he can see. When a coupon is created, it references a profile. When a CoYYn is sent, it flows from one profile to another. Every piece of Hayah data traces back to a row in this table.

This migration must be run exactly once, on a fresh Supabase project, before any other work begins. It creates the schema, locks it down with RLS, sets up the auto-profile trigger, and seeds the two real users with their partner link.

---

## User Story

As the system (not a human user), when a new user signs up via Supabase Auth, a corresponding `profiles` row is automatically created from their auth metadata. The two users (Yahya and Yara) are pre-created and linked as partners. Their profiles are private — each user can read their own profile and their partner's profile, but cannot modify the other's data and cannot delete any profile.

---

## Full DB Flow

1. **Supabase Auth creates a user** via `auth.users` (email + password signup)
2. **Trigger fires** on `INSERT` to `auth.users` → calls `handle_new_user()`
3. **Function reads** `NEW.raw_user_meta_data->>'display_name'` and `NEW.email`
4. **Inserts a row** into `public.profiles` with matching `id`, `display_name`, and `email`
5. **`partner_id` starts as NULL** — it is set manually during seed or via a future linking flow
6. **RLS policies activate** — the new profile is immediately protected

For the seed specifically:
1. Two auth users are pre-created (Yahya + Yara) with known UUIDs
2. Their `profiles` rows are inserted directly (bypassing trigger via seed, or trigger handles it)
3. A second `UPDATE` sets `partner_id` on each row pointing to the other user's UUID

## SQL Schema

**File to create:** `supabase/migrations/001_auth_profiles.sql`

```sql
-- ============================================================
-- T201: Auth Profiles Migration
-- Creates the profiles table, RLS policies, trigger, and seed
-- Run once on a fresh Supabase project before anything else
-- ============================================================


-- ── 1. PROFILES TABLE ────────────────────────────────────────

create table public.profiles (
  id            uuid        primary key references auth.users (id) on delete cascade,
  display_name  text        not null,
  email         text        not null,
  avatar_url    text,
  partner_id    uuid        references public.profiles (id) on delete set null,
  role          text        not null default 'user',
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

-- Index on partner_id for fast partner lookups
create index profiles_partner_id_idx on public.profiles (partner_id);

comment on table public.profiles is
  'One row per auth user. Extends auth.users with display name, avatar, and partner link.';
comment on column public.profiles.partner_id is
  'UUID of the other user. Self-referential. Exactly one pair exists in this app.';


-- ── 2. UPDATED_AT TRIGGER ────────────────────────────────────

create or replace function public.set_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

create trigger profiles_set_updated_at
  before update on public.profiles
  for each row
  execute function public.set_updated_at();


-- ── 3. AUTO-CREATE PROFILE ON AUTH SIGNUP ────────────────────

create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer set search_path = public
as $$
begin
  insert into public.profiles (id, display_name, email)
  values (
    new.id,
    coalesce(new.raw_user_meta_data ->> 'display_name', split_part(new.email, '@', 1)),
    new.email
  );
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row
  execute function public.handle_new_user();


-- ── 4. ROW LEVEL SECURITY ────────────────────────────────────

alter table public.profiles enable row level security;

-- Policy: users can read their own profile
create policy "profiles: own read"
  on public.profiles
  for select
  using (auth.uid() = id);

-- Policy: users can read their partner's profile
create policy "profiles: partner read"
  on public.profiles
  for select
  using (
    id = (
      select partner_id
      from public.profiles
      where id = auth.uid()
    )
  );

-- Policy: users can update only their own profile
create policy "profiles: own update"
  on public.profiles
  for update
  using (auth.uid() = id)
  with check (auth.uid() = id);

-- No INSERT policy from client — inserts are handled by trigger only
-- No DELETE policy — profiles are permanent

-- ── 5. SEED DATA ─────────────────────────────────────────────
-- Replace the UUIDs below with the real UUIDs from your Supabase
-- Auth dashboard after creating the two users manually.
--
-- Steps:
--   1. Go to Supabase Dashboard → Authentication → Users
--   2. "Invite user" for yahya@example.com (set password after)
--   3. "Invite user" for yara@example.com  (set password after)
--   4. Copy each user's UUID into the variables below
--   5. Run this seed block

do $$
declare
  yahya_id uuid := '00000000-0000-0000-0000-000000000001'; -- REPLACE with real UUID
  yara_id  uuid := '00000000-0000-0000-0000-000000000002'; -- REPLACE with real UUID
begin
  -- Update display names (trigger already created the rows)
  update public.profiles
  set display_name = 'Yahya'
  where id = yahya_id;

  update public.profiles
  set display_name = 'Yara'
  where id = yara_id;

  -- Link partners (bidirectional)
  update public.profiles set partner_id = yara_id  where id = yahya_id;
  update public.profiles set partner_id = yahya_id where id = yara_id;
end;
$$;
```

## Implementation Notes

- **File location:** `supabase/migrations/001_auth_profiles.sql` — Supabase CLI expects migrations in this folder, numbered sequentially. This is migration 001 and has no predecessors.
- **`security definer`** on `handle_new_user()` is required — the trigger runs as the function owner (service role), not as the inserting user. Without this, the trigger cannot write to `public.profiles` during signup.
- **`set search_path = public`** on the trigger function is a Supabase-recommended security hardening measure to prevent schema injection attacks.
- **`on delete cascade`** on `profiles.id → auth.users.id` means if a Supabase auth user is deleted, their profile is deleted too. This is intentional — orphan profiles should never exist.
- **`on delete set null`** on `partner_id` means if the partner's auth record is deleted (shouldn't happen, but defensive coding), `partner_id` becomes NULL rather than erroring. The remaining user's profile stays intact.
- **No `insert` RLS policy** is intentional. The only way to create a profile is via the `on_auth_user_created` trigger, which runs as service role and bypasses RLS. This prevents any client from directly inserting profile rows.
- **The seed UUIDs are placeholders.** The agent running this migration must replace them with the real UUIDs from the Supabase Auth dashboard. Document the real UUIDs in a local `.env.local` note or in the project's private notes — not in the codebase.
- **`coalesce` fallback** in `handle_new_user()` uses the email prefix as display name if no `display_name` metadata is passed. For the seed users, display names are set explicitly via the UPDATE block.
- This migration does not create any Storage buckets or Edge Functions. Those come in later tasks.
- Run via Supabase CLI: `supabase db push` or `supabase migration up` (check which CLI version the project uses).

## Edge Cases

- **Trigger fires during seed:** If you create the auth users via the Supabase dashboard before running the seed block, the trigger will have already created profile rows with placeholder display names. The seed UPDATE statements handle this by overwriting display_name — this is idempotent and safe.
- **Seed block run twice:** The UPDATE statements are idempotent (they set the same values). Running the seed twice does not break anything.
- **`partner_id` circular reference:** The profiles table has a self-referential FK (`partner_id → profiles.id`). Postgres allows this. The seed first creates both rows (via trigger), then sets `partner_id` in a separate UPDATE — this ordering avoids FK violation because both rows exist before either link is set.
- **RLS partner read policy — recursive concern:** The `partner read` policy does a subquery on `profiles` itself. Postgres handles this correctly because the subquery is scoped to `auth.uid() = id` (own row), which is covered by the first policy. There is no infinite recursion.
- **User tries to change their own `partner_id`:** The `own update` policy technically allows this (users can update their own row). If this is a concern, add a BEFORE UPDATE trigger that raises an exception if `partner_id` is changed. For V1 with exactly 2 known users, this is acceptable risk — the client app does not expose a UI for editing `partner_id`.
- **User tries to change their own `id` or `role`:** Same as above — the app client never exposes these fields. Defense in depth via server-side validation is planned for T202.
- **Migration run on a non-empty database:** If re-running after partial setup, the `create table` statement will error. Always run migrations on a fresh project or use `if not exists` guards when re-running manually for development resets.

## What This Connects To

- **T202 (Supabase Client)** — imports and uses the `profiles` table shape. The TypeScript type `UserProfile` will be derived from this schema.
- **T203 (Login Page)** — submits to `supabase.auth.signInWithPassword()`. After login, the session resolves a `profiles` row for the authenticated user.
- **T204 (AuthProvider)** — fetches the current user's profile row on mount. Uses `partner_id` to load the partner profile. Both are stored in Zustand.
- **Every future data table** (CoYYns, coupons, notifications, period tracker) will have a `user_id` column that references `profiles.id`. This is the anchor point.
- **The period tracker RLS (T501)** will add an owner-only policy stricter than the partner-read pattern established here — Yara should never see those rows.

## Test Requirements

This is a SQL migration — there are no React components to test with Vitest. Instead, write a manual verification checklist that the auditor runs against a local Supabase instance. Create the file at:

`supabase/tests/001_auth_profiles_checks.sql`

```sql
-- Run after migration to verify correctness

-- 1. Table exists and has all columns
select column_name, data_type, is_nullable
from information_schema.columns
where table_schema = 'public' and table_name = 'profiles'
order by ordinal_position;
-- Expected: id, display_name, email, avatar_url, partner_id, role, created_at, updated_at

-- 2. RLS is enabled
select relrowsecurity
from pg_class
where relname = 'profiles' and relnamespace = 'public'::regnamespace;
-- Expected: true

-- 3. Policies exist
select policyname, cmd
from pg_policies
where tablename = 'profiles'
order by policyname;
-- Expected: 3 rows — own read (SELECT), partner read (SELECT), own update (UPDATE)

-- 4. Trigger exists
select trigger_name, event_manipulation, event_object_table
from information_schema.triggers
where trigger_name = 'on_auth_user_created';
-- Expected: 1 row, INSERT, auth.users

-- 5. Partner link is bidirectional (after seed)
select
  a.display_name as user_name,
  b.display_name as partner_name
from public.profiles a
join public.profiles b on a.partner_id = b.id;
-- Expected: 2 rows — (Yahya, Yara) and (Yara, Yahya)

-- 6. No profiles without a display_name
select count(*) from public.profiles where display_name is null or display_name = '';
-- Expected: 0
```

Additionally, the auditor should manually verify in the Supabase dashboard:
- Log in as Yahya → `select * from profiles` returns exactly 2 rows (own + partner)
- Log in as Yara → same result
- Neither user can delete their profile row (should receive policy violation error)
- Neither user can directly insert into profiles (should receive policy violation error)

## Do NOT

- Do not use `uuid_generate_v4()` for the placeholder seed UUIDs — use real UUIDs from the Auth dashboard
- Do not grant the anon role any access to `profiles` — RLS handles access for authenticated users only
- Do not add a public `INSERT` policy — profile creation is trigger-only
- Do not create a `DELETE` policy — profiles must never be deleted via client code
- Do not store passwords, tokens, or any secrets in this migration file
- Do not add `is_active` or `deleted_at` soft-delete columns — Supabase Auth handles user lifecycle
- Do not run `supabase db reset` in production — it wipes all data
- Do not skip the `set search_path = public` hardening on the trigger function
- Do not create a second migration file for the seed — keep seed in this file using the `do $$ ... $$` block, so the migration is self-contained
