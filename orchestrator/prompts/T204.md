## Why This Exists

Every feature in Hayah depends on knowing who is using the app and who their partner is. The shopping list needs to know Yahya's profile to attribute items. The love coupons need Yara's partner_id to send them correctly. Prayer tracking needs the user's timezone and display name. Without a reliable, always-available auth context, every component would need to independently fetch the session — creating waterfall loading, inconsistent state, and duplicated Supabase calls.

AuthProvider solves this once, at the root of the app. It subscribes to Supabase's real-time auth state stream, fetches both the user profile and their partner's profile, and makes that data available to every component instantly via a single hook. When the session changes — sign in, sign out, token refresh — the entire app reacts consistently from one source of truth.

This is the foundation everything else is built on.

---

## User Story

As Yahya or Yara, when I open the app, it knows who I am immediately. I never see a flash of the wrong name, a broken avatar, or a screen that loads and then re-loads when auth resolves. When I sign out, the app clears my session and returns me to the login page cleanly. My partner's information is also available to any feature that needs it, without that feature needing to fetch it separately.

## Full UX Flow

1. **App mounts** → AuthProvider initializes, `isLoading` is `true`
2. **Supabase `onAuthStateChange` fires** → checks for an existing session
3. **No session found:**
   - `user`, `profile`, `partner` are set to `null`
   - `isLoading` set to `false`
   - Any protected route middleware (T205) redirects to `/login`
4. **Session found:**
   - `user` is set to the Supabase `User` object
   - Provider fetches `profiles` row where `id = user.id`
   - If `profile.partner_id` exists, fetch the partner's `profiles` row
   - `profile` and `partner` are set in context
   - `isLoading` set to `false`
   - App renders normally with full identity context
5. **Session token refreshes silently** (Supabase handles this automatically) → `onAuthStateChange` fires again with the refreshed session → profile data is re-fetched to stay current
6. **User taps Sign Out:**
   - `signOut()` is called from `useAuth()`
   - Supabase session is cleared via `supabase.auth.signOut()`
   - `user`, `profile`, `partner` are set to `null`
   - `router.push('/login')` navigates away
   - No stale data remains in context

### Context Shape

```ts
// src/lib/types/user.types.ts

export type Profile = {
  id: string
  display_name: string
  email: string
  avatar_url: string | null
  partner_id: string | null
  role: string
  created_at: string
  updated_at: string
}

export type AuthContextType = {
  user: User | null        // Supabase auth User object from @supabase/supabase-js
  profile: Profile | null  // Row from the profiles table matching user.id
  partner: Profile | null  // Row from profiles table matching profile.partner_id
  isLoading: boolean       // True until the initial onAuthStateChange has resolved
  signOut: () => Promise<void>
}
```

## Implementation Notes

- `"use client"` directive is required on `AuthProvider.tsx` — it uses `useState`, `useEffect`, and `useRouter`
- Create the React context with `createContext<AuthContextType | null>(null)` — the null default forces `useAuth()` to throw if used outside the provider
- Subscribe to auth changes with `supabase.auth.onAuthStateChange((_event, session) => { ... })` inside a `useEffect`. Return the `subscription.unsubscribe` as the cleanup function
- When `session` arrives from `onAuthStateChange`, set `user` from `session.user`, then run the profile fetch. Do not expose the Supabase session object directly — only expose the typed `User` and `Profile`
- Fetch profile: `supabase.from('profiles').select('*').eq('id', session.user.id).single()`
- Fetch partner: if `profile.partner_id` is non-null, `supabase.from('profiles').select('*').eq('id', profile.partner_id).single()`. Otherwise set `partner` to `null`
- Batch the two fetches with `Promise.all` if `partner_id` is known up front — avoids a serial waterfall
- `isLoading` must remain `true` until `onAuthStateChange` fires at least once. Supabase fires this immediately on mount even for no-session cases, so the loading window is very short
- `signOut` implementation: call `await supabase.auth.signOut()`, then `router.push('/login')`. State cleanup is handled automatically by the `onAuthStateChange` listener which will fire with `null` session
- Use `useRouter` from `next/navigation` inside the provider for the redirect
- The `useAuth` hook must read from the context and throw a descriptive error if `null`: `if (!context) throw new Error('useAuth must be used within an AuthProvider')`
- Import the Supabase browser client from `@/lib/supabase/client` (created in T202)

## Edge Cases

- **Token expired on mount** → Supabase auto-refreshes the token before firing `onAuthStateChange`. Handle the case where a profile fetch fails after a technically valid session — set `profile` to `null` and treat as unauthenticated to avoid a broken half-state
- **Profile row missing** → If `profiles` returns no row for the authenticated user (e.g. sign-up flow not completed), set `profile` to `null` and `isLoading` to `false`. T205 middleware or T206 (profile setup) handles routing the user to complete onboarding
- **Partner profile deleted or partner_id dangling** → Partner fetch may return an error. Catch it silently and set `partner` to `null`. Do not throw or block the rest of the auth flow
- **Component calls `useAuth()` before provider mounts** → The thrown error must be clear and actionable: `"useAuth must be used within an AuthProvider"` — not a generic null ref
- **Rapid sign out / sign in cycle** → Each `onAuthStateChange` event must reset all state synchronously before setting new values. Avoid stale closure issues by always reading fresh state from the event callback, not from React state variables
- **Network offline on mount** → Supabase may not resolve immediately. `isLoading` stays `true`. App should show a global loading state (handled by AppShell or layout, not by this provider) while waiting

## What This Connects To

- **`src/app/layout.tsx`** wraps the entire app with `<AuthProvider>`. It is the outermost client wrapper
- **T205 (Middleware)** reads the Supabase session server-side for route protection. AuthProvider is the client-side counterpart — both must agree on auth state
- **T206 (Profile Setup page)** reads `profile` from `useAuth()` to check if onboarding is complete
- **T207 (Settings Update)** calls Supabase to update the profile row, then the change is reflected next time `onAuthStateChange` fires and triggers a re-fetch
- **Every feature module** (shopping, coupons, health, etc.) uses `useAuth()` to access `profile.id`, `profile.display_name`, and `partner.id` without making their own Supabase calls

## Do NOT

- Don't expose the raw Supabase `Session` object through context — only expose typed `User` and `Profile` shapes
- Don't fetch the profile inside individual feature components — that is AuthProvider's job
- Don't use Zustand for auth state — React Context is the right tool for provider-pattern auth
- Don't redirect inside the provider based on route logic — that is T205 middleware's job
- Don't hardcode the `/login` path as a string literal in more than one place — define it as a constant
- Don't skip the `unsubscribe` cleanup in `useEffect` — it will cause memory leaks and stale listener bugs
- Don't call `router.refresh()` after sign out — the redirect handles it
- Don't fetch partner eagerly if `partner_id` is null — skip the query entirely
