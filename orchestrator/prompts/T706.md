## Why This Exists

Challenges are the competitive heartbeat of the CoYYns economy. When Yahya wants to bet Yara that he can read a book before she finishes her TV show, or when Yara wants to challenge Yahya to wake up early for a week вҖ” this form is how that bet begins. It's not a form. It's a handshake, a gauntlet thrown, a game started.

The form must feel light and fun, not bureaucratic. Picking an emoji makes it personal. Setting stakes makes it real. The moment you tap "Create Challenge", your partner gets a push notification вҖ” the game is on. CoYYns don't leave your wallet here; they're pledged. The actual transfer only happens when the challenge resolves. This distinction matters for trust: creating a challenge is a commitment, not a deduction.

---

## User Story

As Yahya or Yara, when I want to challenge my partner, I tap "Create Challenge" in the Marketplace Challenges tab. A bottom sheet slides up. I pick an emoji that matches the vibe, write a title, optionally describe the rules, set how many CoYYns are on the line, and optionally set a deadline. I tap "Create Challenge" and my partner immediately gets a notification. The sheet closes and I see the new challenge appear in the list.

---

## Full UX Flow

1. **Trigger** вҖ” User taps the "Create Challenge" button in the Challenges tab of the Marketplace (T701)
2. **Bottom sheet slides up** вҖ” `AnimatePresence` + `motion.div` with `y` from `"100%"` to `0`, 300ms ease-out cubic. A dark overlay fades in behind it
3. **Form is visible:**
   - Title row: "Create a Challenge" in Playfair Display, "Challenge [Partner Name]" subtitle in DM Sans text-secondary
   - Emoji selector row: a horizontal scrollable strip of quick-pick emojis; tapping one selects it (highlighted with `bg-accent-primary/10` ring)
   - Title input: single-line text input, placeholder "Challenge title"
   - Description textarea: optional, placeholder "Description (optional)", 3 rows
   - Stakes row: numeric input labeled with "CoYYns" to the right; default value 50
   - Deadline row: a date picker input (native `<input type="date">` styled to match the design system), labeled "Deadline (optional)", prefixed with a calendar icon
4. **User taps "Create Challenge":**
   - RHF validates via Zod schema вҖ” inline errors appear below each field on failure
   - On valid: button enters loading state (spinner, text "CreatingвҖҰ", disabled)
   - Calls `supabase.from('challenges').insert(...)` with the form data plus `creator_id` from `useAuth`
   - Calls the push notification utility (T402) to notify the partner: `"вҡЎ [CreatorName] challenged you: [title] for [stakes] CoYYns!"`
   - On success: closes the sheet (`onClose()`), fires a success toast ("Challenge created!"), calls `onSuccess?.()`
   - On error: loading clears, inline error message appears at the bottom of the form
5. **User taps the overlay or drags the handle down** вҶ’ sheet dismisses, no data saved, no side effects

## Database: challenges Table

This task requires the `challenges` table. Run the following migration in your Supabase project before implementing the component:

```sql
-- Migration: create challenges table
CREATE TABLE IF NOT EXISTS public.challenges (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  creator_id    uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  title         text NOT NULL,
  description   text,
  emoji         text,
  stakes        integer NOT NULL CHECK (stakes > 0),
  deadline      timestamptz,
  status        text NOT NULL DEFAULT 'active'
                  CHECK (status IN ('active', 'claimed', 'resolved', 'cancelled')),
  winner_id     uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  claimed_by    uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now()
);

-- Keep updated_at current automatically
CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;

CREATE TRIGGER challenges_updated_at
  BEFORE UPDATE ON public.challenges
  FOR EACH ROW EXECUTE FUNCTION public.set_updated_at();

-- RLS
ALTER TABLE public.challenges ENABLE ROW LEVEL SECURITY;

-- Both users in the relationship can read all challenges
CREATE POLICY "challenges_select" ON public.challenges
  FOR SELECT USING (
    auth.uid() = creator_id OR
    auth.uid() IN (SELECT id FROM public.profiles)
  );

-- Only the creator can insert
CREATE POLICY "challenges_insert" ON public.challenges
  FOR INSERT WITH CHECK (auth.uid() = creator_id);

-- Both users can update (for claiming, resolving, cancelling)
CREATE POLICY "challenges_update" ON public.challenges
  FOR UPDATE USING (
    auth.uid() = creator_id OR
    auth.uid() IN (SELECT id FROM public.profiles)
  );
```

## Zod Schema

```ts
import { z } from "zod"

const isFutureDate = (val: string) => new Date(val) > new Date()

export const createChallengeSchema = z.object({
  title: z
    .string()
    .min(1, "Name your challenge")
    .max(100, "Title must be 100 characters or less"),
  description: z
    .string()
    .max(500, "Description must be 500 characters or less")
    .optional(),
  emoji: z.string().optional(),
  stakes: z
    .number({ invalid_type_error: "Enter a number" })
    .int("Must be a whole number")
    .min(1, "Must wager at least 1 CoYYn")
    .max(1000, "Maximum 1,000 CoYYns"),
  deadline: z
    .string()
    .optional()
    .refine(
      (val) => !val || isFutureDate(val),
      { message: "Deadline must be in the future" }
    ),
})

export type CreateChallengeFormValues = z.infer<typeof createChallengeSchema>
```

## Implementation Notes

- `"use client"` directive required вҖ” uses React Hook Form, Framer Motion, Supabase client, useAuth
- Use `AnimatePresence` to mount/unmount the overlay and sheet; when `open` is false, nothing renders in the DOM
- Sheet animation: `motion.div` with `initial={{ y: "100%" }}` вҶ’ `animate={{ y: 0 }}` вҶ’ `exit={{ y: "100%" }}`, `transition={{ duration: 0.3, ease: [0.25, 0.46, 0.45, 0.94] }}`
- Overlay animation: `motion.div` with `initial={{ opacity: 0 }}` вҶ’ `animate={{ opacity: 1 }}` вҶ’ `exit={{ opacity: 0 }}`, 200ms
- Use `useForm<CreateChallengeFormValues>` with `zodResolver(createChallengeSchema)` and `defaultValues: { stakes: 50, emoji: "вҡЎ" }`
- Emoji options constant (defined at module level, not inside the component):
  ```ts
  const EMOJI_OPTIONS = ["вҡЎ", "рҹҸҶ", "рҹ“ҡ", "рҹҸғ", "рҹҺ®", "рҹҚі", "рҹ’Ә", "рҹҺҜ", "рҹҺЁ", "рҹҢҷ"]
  ```
- Emoji selection: a `Controller` field from RHF managing `emoji`; clicking an emoji calls `field.onChange(emoji)`
- Stakes field: use `<input type="number">` with `valueAsNumber: true` in the `register` call so RHF passes a `number` not a `string` to Zod
- Deadline field: use `<input type="date">` вҖ” native date picker; the value is an ISO date string (`YYYY-MM-DD`); store as-is, Supabase will accept it for a `timestamptz` column
- Read current user from `useAuth` (T204) вҖ” use `user.id` as `creator_id` and `partnerProfile.display_name` for the "Challenge [Partner Name]" subtitle
- Insert into Supabase: `supabase.from('challenges').insert({ creator_id, title, description, emoji, stakes, deadline: deadline ? new Date(deadline).toISOString() : null })`
- Push notification (T402): call after successful insert, passing the partner's user ID as the recipient and the formatted string `вҡЎ [creatorName] challenged you: [title] for [stakes] CoYYns!`
- Reset the form on successful submission via `reset()` from `useForm`
- Do not close the sheet if the Supabase insert fails вҖ” display the error inline and keep the form data intact for retry
- Tapping the overlay calls `onClose()` вҖ” no form submission, no data saved
- Use `cn()` from `@/lib/utils` for all conditional class merging
- Use design tokens only вҖ” no hardcoded hex values anywhere in the file

## Edge Cases

- **Partner name unavailable** вҶ’ subtitle falls back to "Challenge your partner" if `partnerProfile` is null or loading
- **Supabase insert fails** вҶ’ show inline error at the bottom of the form: "Failed to create challenge вҖ” please try again"; keep form open and populated
- **Push notification fails** вҶ’ do not block the success flow; log the error silently. The challenge was created; a missed notification is recoverable, a broken UX is not
- **Deadline is today** вҶ’ the Zod refinement uses `new Date(val) > new Date()` вҖ” today's date evaluates to approximately midnight which may be in the past; show "Deadline must be in the future" and block submission
- **Stakes input left blank** вҶ’ `valueAsNumber` returns `NaN`; the Zod schema catches this with `invalid_type_error: "Enter a number"` вҖ” display this inline below the stakes input
- **Very long title (> 100 chars)** вҶ’ Zod blocks submission; inline error appears below title input
- **Rapid double-tap on Create** вҶ’ loading state disables the button after first tap
- **Sheet open on a slow connection** вҶ’ form is immediately interactive; the Supabase call only fires on submit, not on open
- **No emoji selected** вҶ’ `emoji` field is optional; submitting without one is valid; store as null in the database

- Don't hardcode color hex values вҖ” use Tailwind design tokens only
- Don't use `router.push()` or navigate away on success вҖ” only close the sheet

- Don't install a custom date picker library вҖ” use native `<input type="date">` styled to match the design system
